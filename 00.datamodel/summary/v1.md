### 2.1 Person (후보자)

|----------|----------|-----------|------|
| `site_user_mapping.id` | `person_id` | 100% (7,780,115) | 전역 고유 ID |
| `profile.name` → REDACTED | `name` | PII 마스킹됨 | `<REDACTED>` 처리 |
| `resume.id` (main_flag=1) | `resume_id` | 96.23% (7,715,508) | **main_flag=1 필터 필수** |
| career 기반 LLM 추출 | `total_experience_years` | 계산 필요 | days_worked 100% 제로 → started_on/ended_on 기반 계산 |

- 유저당 이력서: 98.38%가 1개, main_flag=1 필터로 대표이력서 사용
- `global_user_ref` 100% 빈값 — 외부 시스템 연결 불가

### 2.3 Chapter (경험 단위)

**소스**: `resume.career`

| 소스 필드 | 노드 속성 | Fill Rate | 비고 |
|----------|----------|-----------|------|
| career.id | `chapter_id` | 100% | |

| 텍스트 발췌 | `evidence_chunk` | 조합 생성 | Vector Index용 |

**품질 이슈**:

**정규화 필요**: duration_months 계산 로직 (started_on/ended_on → 월수 변환)

### 2.5 Skill (기술)

**소스**: `resume.skill`

| 소스 필드 | 노드 속성 | Fill Rate | 비고 |
|----------|----------|-----------|------|
| skill.code (codehub 매핑) | `skill_id` | **2.4%** (2,398/101,925) | 표준 코드 한정 |
| skill.name | `name` | 100% (20.8M) | 사용자 직접 입력 포함 |
| skill.type | `category` (변환) | 100% | HARD_SKILL 50.2% / SOFT_SKILL 47.5% |
| 정규화 후 생성 | `aliases` | 정규화 필요 | 엑셀/Excel/EXCEL 등 |

**규모**: 20,810,452건, 이력서 커버리지 38.3% (3,074,732 이력서), 이력서당 평균 6.77개

**품질 이슈** (가장 시급):
- **97.6% 비표준 코드**: 101,925개 고유 코드 중 codehub 매핑 2,398개(2.4%)
- 동일 스킬 다중 표기로 실제 보유자 과소 추정
- 스킬 입력 상한(cap) 20개 존재 추정 — 스킬 다양성 과소 추정 가능
- SOFT_SKILL이 TOP 10의 60% 차지 — 성실성(25.2%), 긍정적(17.3%) 등
- 2022년 이후 이력서당 스킬 수 164% 급증, AI/GenAI 스킬 폭발적 성장

**정규화 필요** (Critical):
1. NLP 기반 유사 스킬 클러스터링
2. codehub 표준 코드 매핑
3. 신규 스킬 코드 등록
4. category 체계 재정의 (language/framework/database/infra/tool → codehub HARD_SKILL/SOFT_SKILL과 매핑)

## 3. 엣지 구현 데이터 품질

### 3.1 FK 무결성

14개 테이블 간 **15개 FK 관계에서 고아 레코드 0건** — 데이터 무결성 완전.

### 3.2 JOIN 전략

| JOIN 유형 | 관계 | 근거 |
|----------|------|------|
| **INNER JOIN** | resume ↔ profile (site_user_mapping_id) | 양방향 완전 매핑 |
| **INNER JOIN** | resume ↔ workcondition (resume_id) | 완전 1:1 |
| **LEFT JOIN** | resume → career, education, certificate, skill 등 | 선택적 데이터 (6.3%~95.6%) |

**주의**: 실제 JOIN 키는 `site_user_mapping_id` (명세의 `site_user_ref`가 아님)

### 3.3 엣지별 데이터 가용성

| 엣지 | 데이터 소스 | 가용성 | 비고 |
|------|-----------|--------|------|
| `Person-[:HAS_CHAPTER]->Chapter` | resume → career | 68.9% (5.52M 이력서) | LEFT JOIN |
| `Chapter-[:NEXT_CHAPTER]->Chapter` | career.started_on 정렬 | 68.9% | 동일 resume 내 시간순 |
| `Chapter-[:PERFORMED_ROLE]->Role` | career.job_classification_codes | ~100% (경력 보유자) | codehub 100% 매핑 |
| `Chapter-[:USED_SKILL]->Skill` | resume.skill | 38.3% | 경력-스킬 직접 연결 불가 (resume 단위) |
| `Chapter-[:OCCURRED_AT]->Organization` | career.company_name | 99.96% | 정규화 후 매핑 |
| `Chapter-[:PRODUCED_OUTCOME]->Outcome` | career_description + self_intro | 16.9%~64.1% | LLM 추출 |
| `Chapter-[:HAS_SIGNAL]->SituationalSignal` | 텍스트 소스 | ~65-70% | LLM 추출 |
| `Organization-[:HAS_VACANCY]->Vacancy` | job-hub | 별도 분석 필요 | |
| `Vacancy-[:REQUIRES_ROLE]->Role` | JD LLM 추출 | 별도 분석 필요 | |
| `Vacancy-[:REQUIRES_SKILL]->Skill` | job-hub.skill | 별도 분석 필요 | |
| `Vacancy-[:NEEDS_SIGNAL]->SituationalSignal` | JD LLM 추출 | 별도 분석 필요 | |
| `Organization-[:IN_INDUSTRY]->Industry` | NICE 업종 코드 | 외부 소스 | 마스터 데이터 |
| `Vacancy-[:MAPPED_TO]->Person` | 매핑 결과 | 런타임 생성 | |

**핵심 제약**: `career_description`은 `career_id` FK가 없어 career 항목별 매핑 불가, resume 단위 귀속. Outcome/Signal 추출 시 경력 항목별 매칭은 LLM이 텍스트 내 컨텍스트로 판단해야 함.

| company.vacancy.seniority | JD에서 LLM 추출 | job-hub 별도 분석 필요 |
| candidate.role_evolution.pattern | career 궤적에서 LLM 추론 | position_grade(39%) + title(29%) 저입력이 정확도 제약 |
| candidate.total_experience_years | started_on/ended_on 기반 계산 | days_worked 100% 제로 → 직접 계산 필요 |
| 최근 경험의 scope_type | position_grade/title + LLM | 미입력 60-70% → LLM 의존 |

### 피처별 v1 활성화 전망

| 피처 | 예상 ACTIVE 비율 | 주요 병목 |
|------|-----------------|----------|
| F1 stage_match | 중간 (~50-60%) | 회사명→Organization 정규화, NICE 매핑 |
| F2 vacancy_fit | 중간 (~50-65%) | career_description 16.9% 보유율 |
| F3 domain_fit | 높음 (~70%+) | industry_codes 66% 빈배열이나 NICE 보완 가능 |
| F4 culture_fit | **매우 낮음 (<10%)** | work_style_signals 데이터 부재 |
| F5 role_fit | 중간 (~50-60%) | position_grade/title 저입력, days_worked 미계산 |

### 5.1 스킬 코드 정규화 (Critical, 난이도 높음)

|------|-----|

| 항목 | 값 |
|------|-----|
| 고유 회사명 수 | 4,479,983개 |
| BRN 입력률 | 62% |
| 다중 표기 사례 | "쿠팡" 관련 표기 3종 = 합산 42,085건 |

**접근**: BRN 기반 1차 클러스터링 (62%) → 회사명 유사도 매칭 2차 보완 (38%)

### 5.3 days_worked 계산 (Critical, 난이도 낮음)

| 항목 | 값 |
|------|-----|
| 대상 | career 18,709,830건 |
| 현재 상태 | 100% 제로값 |
| 계산 소스 | career.started_on, career.ended_on |

**접근**: `duration_months = (ended_on - started_on)` 단순 계산. ended_on이 null이면 현재 날짜 사용.

### 5.4 certificate type 매핑 (Critical, 난이도 낮음)

| `LANGUAGE_TEST` | `LANGUAGE_EXAM` |
