## 3. SIE 모델 — 경력기술서 구조 추출

### 3.1 목적

경력기술서 내 **경력-프로젝트-성과 구조 추출**. 기존 표면적 NER → SIE(Structure Information Extraction) 중심 구조로 전환하여 파이프라인 단순화.

### 3.2 모델 선정

| 구분 | GLiNER2 (주력) | NuExtract 1.5 (보조) |
|------|--------------|---------------------|
| 기반 아키텍처 | BERT 계열 (microsoft/deberta-v3-base) | 생성형 SLM (Phi-3.5-mini, 3.51B) |
| 추적성 | **매우 우수** — Start/End Span, Confidence 기본 제공 | 보통 — 별도 string-matching 필요 |
| 문맥 길이 | 2048 토큰 제한 (Chunking 필수) | 무제한에 가까운 Sliding Window |
| Hallucination | **없음** — Encoder 특성상 원문에서 정확한 값 추출 | 존재 — 생성형 특성 |
| VRAM 요구 | 낮음 (경량 인코더) | 약 15GB (RTX 3090/4090 구동 가능) |
| 다국어 | 학습 데이터 의존 | 기본적으로 다국어 성능 우수, 혼합 문서 강함 |

### 3.3 운영 전략

**GLiNER2 주력**: Span 기반 추출로 Hallucination 차단, field별 confidence/threshold 개별 설정 가능, 대규모 일괄 처리 효율
**NuExtract 1.5 보조**: 초기 데이터 레이블링 → 수동 검수 → LoRA 파인튜닝 → Full SFT

**하이브리드 라우팅**:
- 표준 길이 이력서 → GLiNER2 (추적성 + 정확도)
- 장문 경력기술서 / 혼합 언어 → NuExtract 1.5 (문맥 길이 무제한 + 다국어)

### 3.4 온톨로지 연결

SIE 추출 결과는 다음 노드/엣지에 직접 매핑:

| SIE 추출 필드 | 온톨로지 매핑 | 비고 |
|-------------|------------|------|
| experience.company | Organization.name | 정규화 후 매핑 |
| experience.title | Role (PERFORMED_ROLE) | 직무분류 코드 참조 |
| experience.start/end_date | Chapter.period_start/end | duration_months 계산 |
| experience.description | Chapter.scope_summary | evidence_chunk 소스 |
| experience.achievements | Outcome 노드 | LLM 추출 보조 |
| project.tech_stack | Skill (USED_SKILL) | 스킬 정규화 필요 |

**CareerDescription(16.9%)에 대한 SIE 적용**:
- 현재 career_id FK 부재로 resume 단위 귀속 → SIE 모델이 Span 위치 정보로 career별 귀속 가능성 제공
- GLiNER2의 Start/End Span을 활용하여 원문 내 경력별 구간 자동 분리 기대

### 11.5 Graceful Degradation 패턴

```python
def compute_feature(company_ctx, candidate_ctx):
    if any(required_field is None for required_field in feature.required_inputs):
        return FeatureResult(score=None, status="INACTIVE")
```

필수 데이터가 없으면 해당 피처를 null/INACTIVE로 처리. 부분 완성을 허용하여 가용 데이터만으로 매핑 수행.

| T1 JD (내부) | 0.80 | job-hub 공고 데이터 |
| T2 NICE | 0.70 | 외부 기업정보 |
| T3 Crawl (사이트) | 0.60 | 홈페이지 크롤링 |
| T4 Crawl (뉴스) | 0.55 | 뉴스 (펀딩 0.65, 실적 0.60) |

### 11.7 structural_tensions Taxonomy (8개)

| Tension | 설명 |
|---------|------|
| tech_debt_vs_features | 기술 부채 vs 기능 개발 |
| speed_vs_reliability | 속도 vs 안정성 |
| founder_vs_professional_mgmt | 창업자 경영 vs 전문 경영 |
| efficiency_vs_growth | 효율성 vs 성장 |
| scaling_leadership | 리더십 확장 |
| integration_tension | 통합 갈등 |
| build_vs_buy | 자체 구축 vs 외부 도입 |
| portfolio_restructuring | 포트폴리오 재구성 |

- 배타성 규칙 + related_tensions 구조
- Vacancy의 hiring_context와 연계하여 기업이 겪는 구조적 긴장을 표현

### 11.8 Operating Model Facets 구조화 시그널

JD 텍스트 LLM 추출 이전에 job-hub 구조화 필드에서 추출 가능한 기초 시그널:

| Facet | job-hub 소스 | 시그널 |
|-------|-------------|--------|
| speed | `overview.always_hire` = true | 상시채용 |
| speed | `overview.close_on_hire` = true | 채용시마감 (긴급 충원) |
| autonomy | `work_condition.work_schedule_option_types` ∋ FLEXIBLE_WORK | 유연근무 |
| autonomy | `work_condition.work_schedule_option_types` ∋ WORK_HOURS_NEGOTIABLE | 근무시간 협의 |
| autonomy | `overview.recruitment_option_types` ∋ REMOTE_WORK_AVAILABLE | 재택근무 |

> F4 전체 ACTIVE 비율은 <10%이나, 위 구조화 시그널은 LLM 없이 즉시 추출 가능

### 11.9 Domain Depth 구조화 추출

LLM 추출 이전에 구조화된 직무 코드로 `primary_domain`을 추정:

1. 전체 경력의 `Career[].jobClassificationCodes` 수집
2. 희망 근무조건의 `WorkCondition.workJobField.industryCodes` 보조 수집
3. 가장 빈번한 코드 = primary_domain (code-hub JOB_CLASSIFICATION_SUBCATEGORY 라벨 조회)
4. Confidence: `min(0.80, 0.50 + 반복횟수 × 0.10)` — 구조화 데이터이므로 높은 신뢰도
