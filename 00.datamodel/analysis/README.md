### 5대 핵심 발견

1. **데이터 무결성 우수, 구조적 공백 존재**: 14개 테이블 간 15개 FK 관계에서 고아 레코드 0건으로 데이터 무결성이 완전하다. 다만 7개 컬럼이 100% 빈값(약 5,200만 레코드)이며, 특히 `global_user_ref`(통합회원키) 미연결과 `days_worked`(근무일수) 미계산이 즉각적인 조치를 요한다.

2. **이력서 노후화 심각**: `user_updated_at`(회원 수정일시) 기준 중앙값 2.8년 경과, 31.6%가 5년 이상 방치 상태이다. 반감기는 31.5개월이며 90일 이내 활성 이력서는 13.92%에 불과하다(`updated_at` 기준은 11.56%). ACTIVE 구직자의 90일 활성률은 47.8%로 전체 대비 3.85배 높다.

3. **스킬 생태계 급변**: 2022년 이후 이력서당 평균 스킬 수가 164% 증가했다. AI/GenAI 스킬(Gemini 7.1배, ChatGPT 5.0배, AI Agent 4.9배)이 폭발적으로 성장하는 반면, 레거시 소프트스킬과 Java 기반 기술은 감소 추세이다.

4. **품질 양극화**: 이력서 품질 점수 평균 52.4점(100점 만점, 10개 섹션 휴리스틱 — §9.8 참조)이다. 별도의 6개 섹션 가중 모델(Report 11)에서는 경력자 74.6점 vs 신입 38.2점으로 Cohen's d=2.80의 극도로 큰 효과크기를 보인다. 두 모델의 가중치가 다르므로 직접 비교 시 주의가 필요하다. 경력기술서 보유율 16.9%, 어학 등록률 6.3%로 품질 개선 여지가 크다.

5. **정규화 필수**: 회사명 448만 고유값, 전공명 4.7만 고유값에서 동일 개체의 다중 표기가 다수 존재한다. 스킬 코드 101,925개 중 codehub 매핑은 2.4%(2,398개)에 불과하여, 스킬 정규화 파이프라인이 가장 시급하다.

### 2.3 PII 보호 구조

- **ALIAS 마스킹**: `birthday → '1900-01-01'` (생년월일은 age로 대체), `salary → '<REDACTED>'`, `name → '<REDACTED>'`
- **센티넬 값**: `deleted_at`, `user_updated_at`에서 `1900-01-01` 사용
- **PII 마스킹 함수** (스키마에 정의):
  - `mask_telephone()`: 대한민국 전화번호 패턴(01X, 02, 0XX) -> `<REDACTED:telephone>`
  - `mask_email()`: 주요 도메인 이메일(gmail, naver, hanmail 등) -> `<REDACTED:email>`
  - `mask_pii()`: 전화번호+이메일 통합 마스킹 (텍스트 필드에 적용)
- **참조키**: `site_user_ref` = magma 시스템의 mem_sys_no 매핑

### 2.4 조인 무결성

14개 테이블 간 15개 FK 관계를 전수 검증한 결과, 고아 레코드 0건으로 데이터 무결성이 완전하다.

| JOIN 유형 | 관계 | 근거 |
|----------|------|------|
| **INNER JOIN** | resume <-> profile (via `site_user_mapping_id`) | 양방향 완전 매핑 |
| **INNER JOIN** | resume <-> workcondition (via `resume_id`) | 완전 1:1 |
| **LEFT JOIN** | resume -> career, education, certificate, skill 등 | 선택적 데이터 (커버리지 6.3%~95.6%) |

주요 특성:
- 유저당 이력서: 1개 98.38%, 2개 1.04%, 3개 이상 0.58%
- 대표이력서(`main_flag=1`): 7,715,508건 (96.23%)
- 실제 JOIN 키는 `site_user_mapping_id` (명세의 `site_user_ref`가 아님)
- `career_description`은 `career_id` FK가 없어 career 항목별 매핑 불가, resume 단위 귀속

> 대표이력서 필터: `WHERE main_flag = 1` 항상 권장.

### 3.4 학력-최종학력 정합성 이슈

`final_education_level`(이력서 헤더)과 실제 `education` 테이블의 최고 학력 간 **35.6% 불일치** (95% CI: [35.52%, 35.59%], n=7,665,296)가 존재한다. 불일치의 96%는 final_education_level이 실제보다 높은 경우(이력서 미완성 효과)이다:

- BACHELOR 선언 + 고교 학력만 입력: 1,438,819건 (18.8%) -- 재학 또는 학력 미입력 추정

- GRADUATE 선언 + 고교 학력만 입력: 153,250건 (2.0%)

> 학력 필터 적용 시 `final_education_level` 단독 사용은 35.6% 오분류를 유발하므로, `education.school_type` 교차 검증 또는 미완성 이력서(BASIC_INFO_COMPLETED) 제외를 권고한다. (AI 추정)

## 7. 데이터 신선도

### 7.1 날짜 컬럼 정의 및 활용 기준

| 컬럼 | 스키마 COMMENT | 데이터 검증 결과 | 신선도 분석 활용 |
|------|---------------|----------------|---------------|
| `updated_at` | '수정일(유저)' | 이력서 내용 수정 시점. 90일 활성 11.56% | 이력서 콘텐츠 갱신의 좁은 기준 |
| `user_updated_at` | '회원 수정일시' | 회원 활동(로그인/프로필/이력서) 시점. 90일 활성 13.92% | **신선도 기본 기준 (권장)** |
| `indexed_at` | '데이터 변경일' | ClickHouse 임포트 시점. 100%가 90일 이내 | 분석 부적합 -- 임포트 타임스탬프 |
| `created_at` | (COMMENT 없음) | ClickHouse 적재 시점 (2026-01-14~02-25) | 분석 부적합 -- 원본 생성일 아님 |

> `indexed_at`은 스키마 COMMENT가 '데이터 변경일'이나, 실제 데이터는 100% 2026년 1-2월에 집중되어 ClickHouse 임포트/인덱싱 시점임이 확인되었다. 신선도 분석에 사용하면 안 된다.

| 비교 항목 | 값 |
|----------|-----|
| 동일 날짜 비율 | 70.9% |
| `user_updated_at`이 더 최신인 비율 | 54.8% |
| `updated_at`이 더 최신인 비율 | 28.9% |
| 평균 차이 | 72.5일 (`user_updated_at`이 더 최신) |

`user_updated_at`은 이력서 수정 외에 회원 로그인/프로필 변경도 반영하므로 더 최신이다. TalentAgent의 "연락 가능성" 판단에 가장 적합한 기준이다.

> 이하 모든 신선도 분석은 `user_updated_at`(회원 수정일시) 기준이다.
