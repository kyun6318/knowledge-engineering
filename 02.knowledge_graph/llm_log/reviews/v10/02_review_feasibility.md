## 1. 일정 실현 가능성

### 1.1 Phase 0 (1주) — 매우 타이트

1주 안에 수행해야 할 작업:
- GCP 프로젝트 초기화 + 9개 서비스 설정
- 3개 서비스 계정 + IAM
- 3개 DB 프로파일링
- LLM PoC 20건 (Haiku vs Flash vs Sonnet)
- 임베딩 모델 검증 20쌍
- Batch API 타이밍 테스트
- 50건 Gold Set 수동 작성
- Go/No-Go 판단

**판정: 비현실적 (1주 → 2주 필요)**

특히 50건 Gold Set 수동 작성은 도메인 전문가가 파트타임이면 1주 이상 소요. LLM PoC 20건도 프롬프트 반복 설계를 고려하면 3일은 필요.

**GraphRAG Core v2도 1주로 동일하게 설정했으나, Pre-Phase 0에서 샘플 100건 확보를 전제**로 하고 있어, 사전 준비가 더 구체적.

### 1.2 Phase 1 (5주) — 도전적이나 가능

- Week 2-3: DB 커넥터 + 3-Tier 비교 모듈 (2주)
- Week 4-5: CandidateContext 파이프라인 + 1,000건 파일럿 (2주)
- Week 5-6: Graph 적재 + API MVP (2주, 4-5와 일부 병렬)

DE+MLE 2명이면 가능하나, 3-Tier 비교 모듈(Tier 2 synonym + embedding 하이브리드)이 2주 내 완성되려면 code-hub synonym 사전이 이미 존재해야 함.

### 1.3 Phase 2 (8주) — 핵심 병목

전체 배치 처리 600K(DB 500K + 파일 100K)를 8주 안에 80%+ 완료해야 함.

**기준 시나리오**: 12시간/라운드 × 60 라운드 = 30일 ≈ 6주
→ 파서 구현(3주) + 배치 처리(6주)는 8주에 맞지 않음. 파서와 배치 병렬 실행 전제.

**리스크**:
- Batch API 응답 시간이 24시간 초과 시 비관 시나리오(60일) 돌입
- 파일 파서 HWP 품질 문제 시 Gemini Multimodal 폴백 추가 개발
- Neo4j Professional 전환 시 다운타임

**판정: 가능하나 버퍼 소진 가능성 높음**

### 1.4 Phase 3 (7주) — 가장 복잡

Organization ER(2주) + MappingFeatures(2주) + GraphRAG vs Vector 실험(2주)이 핵심.

**우려**:
- Organization ER 한국어 특수 처리는 2주로도 부족할 수 있음 ("토스" = "비바리퍼블리카", "카카오" ≠ "카카오뱅크" 등)
- 1,000건+ 전수 검수는 도메인 전문가 파트타임으로 1주 이상
- GraphRAG vs Vector 실험에 5명 평가자 확보 + 교육 + 250건(50 JD × 5명) 평가 수행

**판정: 7주 타이트, 8주 권장**

### 1.5 전체 27주 — 조건부 가능

**필수 전제 조건 충족 시**: 27주 내 완료 가능
**전제 조건 1개라도 미충족 시**: 지연 불가피

## 2. 전제 조건 리스크 분석

### 2.1 차단 의존성 (Blocking Dependencies)

| 전제 조건 | 미충족 시 영향 | 충족 확률 | 비고 |
|----------|-------------|----------|------|
| resume-hub read replica 접근 | **전체 프로젝트 차단** | 70% | 조직 간 협의 필요 |
| job-hub read replica 접근 | Phase 3 차단 | 70% | 동일 |
| code-hub read replica 접근 | 3-Tier 비교 불가 | 70% | 동일 |
| NICE DB 접근 계약 | stage_estimate 불가 | 50% | 외부 계약 |
| Anthropic Batch API 할당량 | 처리 속도 제한 | 80% | Tier 확인 필요 |
| PII 법률 검토 | 마스킹 전략 확정 | 60% | 법무팀 일정 |

**3개 DB read replica 접근이 가장 큰 리스크**. 70% 확률이면 3개 모두 확보할 확률은 ~34%.

### 2.2 가정 검증 필요

| 가정 | 현재 값 | 실패 시 영향 |
|------|---------|------------|
| A2': 파일 이력서 ~100K | 20% | 200K면 비용 2배 |
| A4: 이력서당 Career 평균 3 | 3 | 5면 노드 수 1.67배 |

## 4. 인력 실현 가능성

### 4.1 1 DE + 1 MLE + 도메인 전문가 (파트타임)

**긍정적**:
- DE/MLE 역할 분담이 명확 (GCP 인프라 vs LLM/프롬프트)
- 병렬 작업 가능한 구간이 많음

**우려**:
- 도메인 전문가 "파트타임"이 구체적으로 몇 시간/주인지 불명
- Gold Set 50건/200건 수동 작성, Organization ER 1,000건 검수 등에 도메인 전문가 풀타임 필요한 구간 존재
- DE/MLE 1명이 병가/휴가 시 단일 장애점 (Single Point of Failure)

**판정: 도메인 전문가 투입 시간을 Phase별로 명시 필요**

### 4.2 1인 팀 시 1.5-2x 소요

문서에서 "1인 팀 시 1.5-2× 소요"로 40-54주. 이는 비현실적인 기간이므로, 최소 2인 체제가 전제.

## 5. 기술적 실현 가능성

### 5.1 Neo4j AuraDB 8M 노드 + 25M 엣지

- AuraDB Free: 200K 노드 한도 → Phase 1만 가능
- AuraDB Professional: 800K+ 노드라고 하지만, **8M 노드 + 25M 엣지 수용 가능한지 미검증**
- 8M 노드 수준이면 AuraDB Enterprise 또는 자체 호스팅이 필요할 수 있음

**판정: Phase 0에서 AuraDB Professional 노드 한도 정확히 확인 필수**

### 5.2 Anthropic Batch API 600 청크 처리

- 600 청크 × 1,000건/청크 = 600K 이력서
- 10 동시 배치 → 60 라운드
- 기준 12시간/라운드 → 30일

**Anthropic Batch API의 동시 활성 배치 수가 10 미만이면 일정 초과**.
GraphRAG Core v2에서 Gemini Flash 병행 전략을 제시한 것이 현명한 대비.

### 5.3 HWP 파서

pyhwp/hwp5는 성숙도가 낮은 라이브러리. HWP 비율이 높으면(한국 이력서 특성상 20-30%) 파싱 실패율이 상당할 수 있다.

**판정: Gemini Multimodal 폴백이 현실적이나, 비용 증가 ($0.001 → $0.003/파일)**
