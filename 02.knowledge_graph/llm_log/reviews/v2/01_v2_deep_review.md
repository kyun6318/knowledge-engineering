## 1. 실현 가능성 분석

### 1.1 핵심 전제 조건의 검증 가능성

v2 계획의 실현 가능성은 **3가지 핵심 전제**에 달려 있다.

| # | 전제 조건 | 현재 검증 상태 | 검증 실패 시 영향 |
|---|---|---|---|
| P1 | LLM(Haiku)이 한국어 이력서에서 scope_type/outcomes/signals를 70%+ 정확도로 추출 | **미검증** | 시스템 자체 무의미 — Critical |
| P2 | PII 마스킹 후 외부 API 전송이 법적으로 허용 | **미검증** | 비용 14배 증가, 아키텍처 전면 변경 — Critical |
| P3 | 이력서 파싱(PDF/HWP)이 경력 블록 단위 분리를 70%+ 성공률로 수행 | **미검증** | LLM 입력 품질 저하 → 추출 정확도 연쇄 하락 — High |

**리뷰 판단**: Phase 0 PoC가 이 3가지를 모두 검증하도록 설계되어 있어 구조적으로는 건전하다. 그러나 **P1과 P3이 동시에 실패하는 시나리오**에 대한 대응 계획이 없다. 파싱 품질이 낮으면 LLM 입력 품질도 낮아지므로, 이 두 가지는 독립적이 아닌 **상관 리스크**다.

### 1.2 타임라인 현실성

| Phase | 계획 일정 | 현실적 추정 | 차이 원인 |
|---|---|---|---|
| Phase 0 | 3~4주 | 4~5주 | 법무 확인(PII)이 1주 내 나오지 않을 수 있음 |
| Phase 1 | 6~7주 | 8~10주 | HWP 파싱(2주), CandidateContext 통합테스트(3주+), 프롬프트 반복 최적화 |
| Phase 2 | 3~4주 | 4~5주 | 500K 배치 처리 + 에러 핸들링 + 품질 평가 동시 진행의 부하 |
| **총계** | **12~15주** | **14~18주** | 약 20% 초과 |

**핵심 병목**:
1. **프롬프트 엔지니어링 반복**: v2는 프롬프트를 1회 설계로 가정하지만, 실제로는 5~10회 반복 최적화가 필요. 이력서 유형(자유서술/표/2단)별로 프롬프트 분기가 필요할 수 있음
2. **HWP 파싱**: python-hwp 라이브러리의 성숙도가 낮아 LibreOffice headless 변환이 필요하고, 이 과정에서 레이아웃 손실이 발생
3. **법무 확인 지연**: PII 전략이 Phase 0 의사결정에 blocking이지만, 법무 응답이 늦어지면 전체 일정이 밀림

### 1.3 비용 모델의 신뢰도

v2의 비용 모델은 **합리적**이다. Gold Label 인건비를 건당 20,000원($14.60)으로 산정하여 현실적이며, 시나리오별 총비용 비교도 유용하다.

다만 **2가지 누락 비용**이 있다.

| 항목 | v2 추정 | 보정 추정 | 차이 원인 |
|---|---|---|---|
| 프롬프트 최적화 LLM 비용 | 미계상 | ~$200 | 프롬프트 반복 테스트 시 Sonnet 사용 불가피 |
| 오케스트레이션 인프라 | 미계상 | ~$600/년 | Cloud Workflows 또는 Prefect 호스팅 비용 |

이 누락 비용은 시나리오 A 총비용($8,605)의 ~10% 수준으로, 전체 판단에 큰 영향을 미치지 않는다. v2의 비용 모델은 전반적으로 **신뢰할 수 있는 수준**이다.

## 3. 부족 설계 상세 분석

### 3.1 [Critical] 에러 핸들링 전략 부재

**현재 상태**: 02 문서에 에러 핸들링 언급 없음. 04 문서에도 "에러율 모니터링" 1줄만 존재.
**필요한 내용**:

| 에러 유형 | 발생 빈도 (예상) | 필요한 정책 |
|---|---|---|
| LLM API 호출 실패 (5xx) | 1-2% | 재시도 3회 (exponential backoff) |
| LLM 응답 파싱 실패 (JSON 형식 오류) | 2-5% | 재시도 2회 (temperature 조정) → skip + 로그 |
| LLM 응답 스키마 불일치 (필수 필드 누락) | 5-10% | 부분 수용 (있는 필드만 사용) |
| NICE DB 타임아웃 | 0.5% | 재시도 3회 → past_company_context = null |
| 이력서 파싱 실패 | 3-5% | skip + dead-letter 큐 |
| Graph 적재 실패 (Neo4j 제약 조건 위반) | 0.5-1% | 재시도 (트랜잭션 롤백 후) |

**dead-letter 큐 설계**: 처리 실패 건을 별도 저장하여 이후 수동/자동 재처리
**에러율 가정**: 전체 처리 중 2-5% 실패 예상 → 500K의 10,000~25,000건

### 3.2 [Critical] 인력 배치 미명시

**현재 상태**: 04 문서에 "12주"라고만 하고, DE/MLE/도메인 전문가 배치가 없음
**필요한 내용**:

| 역할 | 인원 | 투입 시기 | 주요 업무 |
|---|---|---|---|
| DE (Data Engineer) | 1명 | Phase 0~2 전체 | 파이프라인 구축, 인프라, 배치 처리 |
| MLE (ML Engineer) | 1명 | Phase 0~2 전체 | 프롬프트 설계, 품질 평가, ML Distillation |
| 도메인 전문가 (HR/채용) | 1명 (파트타임) | Phase 0, 2 | Gold Label 검수, taxonomy 검증 |

**타임라인 기준**: DE 1명 + MLE 1명 풀타임. 1인 작업 시 1.5~2배로 늘어남.

### 3.3 [High] 증분 처리 (Incremental Pipeline) 미설계

**현재 상태**: 04 문서 Phase 3에 "Active Learning"만 언급. 최초 처리 후 신규/갱신 데이터의 증분 처리 전략이 없음.
**필요한 내용**:
- 변경 감지: 파일 hash 비교 또는 DB updated_at
- 신규 건: 기존 파이프라인 동일 처리
- 갱신 건: 변경된 경력 블록만 재추출 (비용 절감)
- 삭제 건: Graph에서 관련 노드/엣지 soft-delete
- 처리 주기: 일 1회 배치 권장

### 3.4 [High] Graph 업데이트/삭제 전략 미설계

**현재 상태**: Graph "적재"만 설계. 프롬프트 변경 시 기존 데이터의 업데이트/삭제 방법이 없음.
**필요한 내용**:
- 필드 수준 재추출: CandidateContext JSON에서 해당 필드만 업데이트 → Graph 노드/엣지 갱신
- 전체 재처리: 새 Graph 인스턴스에 적재 (blue-green 배포)
- Context JSON 버전 관리: GCS/S3에 버전별 보관
- Graph 노드에 `extracted_at`, `prompt_version` 메타데이터 부착

### 3.5 [High] 데이터 품질 검증 파이프라인 부재

**현재 상태**: Phase 2에 "품질 평가"가 있지만, 파이프라인 내 실시간 품질 체크가 없음.
**필요한 내용**:
- **추출 후 즉시 검증**: JSON 스키마 유효성, 필수 필드 존재 여부, confidence 분포 이상 감지
- **배치 수준 통계**: situational_signals 분포가 이전 배치와 크게 다르면 알림 (LLM 동작 변경 감지)
- **자동 리젝트 기준**: confidence < 0.2인 필드가 전체의 50%+ → 해당 이력서 수동 검토 큐

### 3.6 [High] 배치 처리 / 병렬 설계 상세

**현재 상태**: 02 문서에 "Batch API"만 언급. 500K 이력서의 구체적 처리 아키텍처가 없음.
**필요한 내용**:

- 동시 배치 수: 5~10개 (API quota에 따라)
- 예상 처리 시간: 2~3일 (500K ÷ 50K/일)
- 처리 오케스트레이션: Cloud Workflows 또는 Prefect 권장

### 3.7 [Medium] 테스트 전략 부재

**현재 상태**: "통합 테스트 (이력서 200건)"이라고만 언급. 테스트 계층과 기준이 없음.
**필요한 내용**:

| 테스트 레벨 | 대상 | 기준 | 시점 |
|---|---|---|---|
| Unit | Rule 추출 모듈 | 정규식 커버리지 > 90% | Phase 1 |
| Integration | 단일 이력서 E2E | 파싱→추출→Graph 적재 성공 | Phase 1 |
| Batch | 1,000건 배치 | 에러율 < 5%, 처리시간 < 2시간 | Phase 1 말 |

| Scale | 500K 전체 처리 | 에러율 < 3%, 배치 완료 < 3일 | Phase 2 |

### 3.8 [Medium] 모니터링/로깅 전략 미명시

**현재 상태**: Phase 2에 "모니터링 대시보드"만 언급. 구체적 도구/메트릭 없음.

## 4. 문서 간 구조적 문제

### 4.1 비용 기준 불일치

02 문서는 일반 API 가격, 03 문서는 Batch 할인 가격을 사용. 동일 지표에 대해 2배 차이가 나므로, **02 문서에 Batch 가격도 병기**하거나, 모든 비용을 Batch 기준으로 통일해야 한다.

### 4.2 ML Distillation 3중 중복

- 02 문서 §9: ML Distillation 적용 범위 상세
- 04 문서 §2-3: ML Knowledge Distillation 실행 계획
- 05 문서 §3: v1 계승 목록에서 ML Distillation 재언급

**권장**: 02에서 "범위 제한적" 결론만, 04에서 실행 계획, 05에서 리스크만 다루도록 역할 분리.

### 4.3 운영 전략의 분산

증분 처리, 롤백, 모니터링, TTL 등 운영 관련 내용이 04와 05 문서에 산재. **운영 전략을 04 문서에 집중** 배치하고, 05에서는 리스크만 다루는 것이 바람직하다.

## 5. 기술적 우려 사항

### 5.1 한국어 이력서의 비정형성

v2의 Rule 추출 정규식은 "㈜카카오 / 백엔드 개발자 / 2020.03 ~ 2023.06" 같은 정형 패턴을 가정한다. 그러나 실제 한국어 이력서는:
- 표 형태(셀 경계가 텍스트로 변환 시 깨짐)
- 자유서술형(구조 없이 산문으로 기술)
- 포트폴리오 스타일(이미지 위주, 텍스트 최소)

**가정 A7(Rule 추출 70%)은 정형 이력서에만 해당**하며, 자유서술형에서는 30~40%로 떨어질 수 있다. 이력서 유형 분류 → 유형별 처리 전략 분기가 필요할 수 있다.

### 5.2 LLM Hallucination 리스크

v2는 LLM 추출 시 "근거 없이 추론하지 마세요"라는 프롬프트 지침에 의존한다. 그러나:
- situational_signals 14개 taxonomy 중 LLM이 **존재하지 않는 signal을 생성**할 리스크
- outcomes에서 **정량 수치를 날조**할 리스크
- scope_type을 **직급명만으로 과잉 추론**할 리스크 (예: "시니어 개발자" → HEAD로 오분류)

**완화 필요**: evidence_span(원문 인용) 필수화, confidence threshold 기반 필터링, 2-pass 검증(추출 → 검증 프롬프트)

### 5.3 NICE 데이터 시점 불일치

v2는 NICE 현재 데이터로 past_company_context를 생성하면서 `confidence = max(0.20, 0.60 - years_gap * 0.08)`로 감쇠한다. 이 감쇠 함수는:
- 합리적이나 **검증되지 않은 임의 계수**(0.08/년)
- 5년 전 = 0.20(하한)이면 실질적으로 무의미한 데이터를 유지하는 것

**권장**: Phase 2 품질 평가에서 이 감쇠 함수의 적절성을 검증하는 항목 추가.

### 5.4 MappingFeatures의 실용성 — 검증 계획 존재하나 세부 부족

v2는 Phase 2-2에서 "Human eval 상관관계 | MappingFeatures (50건) | r > 0.4 | r > 0.6" 기준을 이미 설정했다. 이는 적절하다.

다만 다음이 추가로 필요하다:
- vacancy_fit의 VACANCY_SIGNAL_ALIGNMENT 테이블이 **수동 설계**임을 명시하고, Phase 2 검증 후 조정 계획
- domain_fit의 embedding 모델별 한국어 도메인 유사도 성능 차이 확인 (PoC에서)
- r < 0.4(최소 기준 미달) 시의 **대응 계획** (weight 재조정? 피처 재설계?)

## 6. 긍정적 평가 (유지할 사항)

### 6.1 핵심 설계 판단이 건전

1. **Rule-first, LLM-for-reasoning**: v4 추출 난이도에 맞는 현실적 비율(Rule 25%, LLM 65%)
2. **Pipeline A~E 분리**: 데이터 소스별 독립 파이프라인으로 관심사 분리
3. **Batch API 활용**: 대량 처리에서 50% 비용 절감
4. **NICE Lookup 분리**: API 비용 0인 데이터를 LLM에 보내지 않는 효율적 설계
5. **Phase 0 PoC 게이트**: 본격 투자 전 핵심 전제 검증
6. **Graceful Degradation**: null 허용 필드 명시로 운영 안정성 확보

### 6.2 비용 모델의 투명성

- 가정을 ID로 체계화하고 "잘못되면" 시나리오까지 명시
- 4가지 시나리오별 총비용 비교가 의사결정에 유용
- v1 대비 비용 변화를 "효율적이라서가 아니라 방법이 다르기 때문"이라고 정직하게 설명

## 8. 최종 판정

> v2 계획은 v1 대비 **근본적으로 올바른 방향 전환**을 달성했다. v4 온톨로지 정합성, LLM 의존도의 현실적 인정, 비용 모델의 투명성 모두 높은 수준이다.
>
> 다만, **"설계"에서 "실행"으로 전환하기 위한 운영 관점이 부족**하다. 에러 핸들링, 증분 처리, 배치 아키텍처, 테스트 전략을 보강하면 실행 가능한 계획이 된다.
>
> 타임라인은 12~15주 → 14~18주로 현실 보정이 필요하며, 비용은 시나리오 A 기준 약 $8,605(~1,179만 원)이 현실적이다. 누락 비용(프롬프트 최적화+오케스트레이션 ~$800)을 포함해도 $9,400 이내로, 전체 판단에 큰 영향은 없다.
