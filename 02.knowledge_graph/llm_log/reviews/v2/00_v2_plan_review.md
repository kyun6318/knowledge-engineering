### 1.5 `05_assumptions_and_risks.md` — ✅ 우수

**강점**:
- 15개 가정을 ID로 체계화하고 "잘못되면" 시나리오까지 명시
- PII 리스크를 Critical로 정확히 분류하고 4가지 옵션을 제시
- v1에서 유효한 부분을 명시적으로 계승 목록화 — 기존 작업 존중

**수정 필요 사항**:

| 위치 | 이슈 | 수정 제안 |
|---|---|---|
| §2.1 PII 옵션 | "Azure/AWS Private Endpoint" 옵션 | 이 옵션이 현실적으로 가장 **균형 잡힌 대안**인데 비용 추정이 "API+α"로 모호. Anthropic/OpenAI의 enterprise private 배포 비용을 **구체적으로 조사**하거나 최소 범위라도 제시해야 |
| §2.5 Confidence | "동일 입력 3회 추출 → 일치율" | 비용 3배라고 했지만, **Batch API 할인(50%) 적용 시 1.5배**로 수정해야 일관성 유지. 또한 3회가 아닌 2회(majority vote 불가)도 검토 가능 |
| §2.8 활성화 비율 | stage_match 40-60% | NICE 매칭률(가정 A5)이 60%일 때 stage_match도 60%에 가까워야 하나, 40%로 시작하는 이유 불명확. **NICE 매칭 성공 but stage 판정 실패 케이스**를 분리해서 설명해야 |

**부족 사항 (추가 필요)**:

| 위치 | 이슈 | 추가 제안 |
|---|---|---|
| 전체 | **이력서 갱신/증분 처리 전략** 미언급 | 최초 500K 처리 후 신규 이력서가 유입될 때의 incremental pipeline 설계가 없음. 이는 운영 단계의 핵심 |
| 전체 | **Graph 데이터 TTL/보존 정책** 미언급 | 오래된 이력서의 Graph 데이터를 언제까지 유지할지 |

## 3. 전체 수준 평가

### ✅ 타당한 설계 판단 (유지)

1. **Rule-first, LLM-for-reasoning** 원칙 — v4 추출 난이도에 맞는 현실적 비율 조정
2. **Pipeline A~E 분리** — 데이터 소스별 독립 파이프라인으로 관심사 분리 적절
3. **Batch API 활용** — 대량 처리에서 50% 비용 절감은 필수
4. **NICE Lookup 분리** — API 비용 0인 데이터를 LLM에 보내지 않는 설계 우수
5. **Phase 0 PoC 게이트** — 본격 투자 전 검증 단계가 리스크를 줄여줌
6. **Graceful Degradation** — null 허용 필드 명시로 운영 안정성 확보

### ⚠️ 과잉 설계 (축소 권장)

1. **SLM 5개 모델 비교** — PII 불가 시에만 해당, 분량 대비 활용도 낮음
2. **ML 분류기 4개 모델 비교** — Phase 2 선택적 항목에 과도한 지면
3. **role_expectations 별도 프롬프트** — vacancy와 통합이 명백한데 별도로 설명

## 4. 최종 결론

> [!IMPORTANT]
> v2 계획은 **v1 대비 근본적으로 개선**되었다. v4 온톨로지와의 정합성, LLM 의존도의 현실적 인정, 비용 모델의 구체성 모두 높은 수준이다.

**핵심 보완 3가지**:
