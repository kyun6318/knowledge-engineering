v10에서 7건이던 부족한 설계가 v11에서 **5건으로 감소**. 프롬프트 설계, PII 마스킹, 검증 체크포인트 추가로 핵심 누락은 해소되었으나, **실제 구현 단계에서 마주할 현실적 문제**에 대한 설계가 일부 미비.

## U1. Career별 vs 전체 이력 LLM 호출 전략 미확정 (High)

**위치**: 03_prompt_design.md §2.2~2.3

**문제**: CandidateContext 추출에서 두 가지 레벨의 추출이 필요:
1. **Career별**: scope_type, outcomes, situational_signals
2. **전체 이력**: role_evolution, domain_depth, work_style_signals

이 두 레벨을 **하나의 LLM 호출로 처리하는지, 2-pass로 처리하는지 미확정**.

**영향**:
- 1-pass: Career 수가 많은 이력서(5+)에서 토큰 초과 + 품질 저하
- 2-pass: 비용 2배 + 레이턴시 2배
- Career별 개별 호출: 비용 3배+ (이력서당 3회 호출)

**권고**: 명시적으로 호출 전략을 정의. 추천:
- Career 1~3개: 1-pass (전체 이력서를 한 번에)
- Career 4+개: Career별 개별 호출 + 마지막에 전체 이력 1회 (role_evolution/domain_depth)
- max_tokens 2,048에서 Career 3개 출력이 한계이므로, 이 분기가 비용에 직접 영향

## U2. 파일 이력서 섹션 분리 로직 부재 (High)

**위치**: 01_extraction_pipeline.md §4, 04_pii_and_validation.md

**문제**: Pipeline B'(파일 폴백)의 핵심인 **"비정형 PDF/DOCX/HWP에서 Career 블록을 분리하는 로직"**이 전혀 기술되지 않음.

현재 §4.1에는 "섹션 분리 → Career 블록 추출"이라고 한 줄로 언급될 뿐, 구체적 방법 없음.

**왜 중요한가**:
- 한국 이력서는 형식이 극도로 다양 (자유형식, 잡코리아 양식, 사람인 양식, 회사 자체 양식 등)
- "경력사항" 섹션 시작점 탐지 자체가 난제
- 하나의 Career 종료와 다음 Career 시작의 구분 (회사명/날짜 패턴 의존)

**권고**: 섹션 분리 전략을 최소한 다음 수준으로 기술:
- 섹션 헤더 패턴 (정규식 기반: "경력", "경력사항", "EXPERIENCE" 등)
- Career 구분 기준 (회사명 + 기간 패턴)
- 분리 실패 시 처리 (전체 텍스트를 단일 Career로 취급 → LLM에 위임)
- 또는 "파일 이력서는 LLM에 전체 텍스트를 전달하고 Career 분리도 LLM이 수행" 전략 (비용 증가)

## U3. resume-hub DB 스키마와의 실제 매핑 미검증 (Medium)

**위치**: 01_extraction_pipeline.md §2.1, §3.1

**문제**: v11은 resume-hub.Career.companyName, resume-hub.Career.position 등의 필드를 참조하지만, **실제 DB 스키마가 이와 일치하는지 검증되지 않음**.

모든 매핑이 가정에 기반:
- `resume-hub.Career.companyName` → 실제 필드명이 `company_name`이거나 `corp_name`일 수 있음
- `resume-hub.Career.workDetails` → JSONB인지, TEXT인지, null 비율은?
- `resume-hub.Skill` → 테이블이 별도인지, Career 내 JSONB인지?

**영향**: Phase 0에서 DB 프로파일링 시 불일치 발견되면 매핑 재설계 필요 (1~2일 추가)

**권고**: v11에 "본 문서의 DB 필드명은 가정이며, Phase 0 DB 프로파일링 결과에 따라 조정" 면책 조항 추가. 또는 Phase 0 이전에 실제 스키마를 확인하여 반영.

## U4. PII 매핑 테이블 저장소 설계 미비 (Medium)

**위치**: 04_pii_and_validation.md §1.3

**문제**: PII 마스킹 시 생성되는 매핑 테이블(person_id → 원본 이름/전화번호)의 **저장소가 "별도 보안 저장소"로만 언급**되고 구체적 설계 없음.

고려 필요 사항:
- 500K+ 건의 PII 매핑은 Secret Manager에 부적합 (소량 키-값 전용)
- BigQuery에 저장하면 접근 통제가 IAM 수준 → 세밀한 통제 어려움
- 별도 DB(CloudSQL + 암호화)가 필요할 수 있으나 비용/관리 부담

**권고**: 최소한 다음 중 하나 선택:
1. **BigQuery 별도 데이터셋** (접근 IAM 분리) + Column-level Security
2. **CloudSQL (PostgreSQL)** + 디스크 암호화 + 별도 서비스 계정
3. **GCS 암호화 파일** (CMEK) + 접근 로그 (가장 단순)

## U5. 한국어 전화번호 변형 패턴 부족 (Low)

**위치**: 04_pii_and_validation.md §1.4

**문제**: 전화번호 정규식이 `010-\d{4}-\d{4}` 하나뿐. 한국 전화번호의 변형:

| 형식 | 예시 | 현재 탐지 |
|------|------|----------|
| 표준 | 010-1234-5678 | O |
| 공백 | 010 1234 5678 | X |
| 연속 | 01012345678 | X |
| 국제 | +82-10-1234-5678 | X |
| 괄호 | (010)1234-5678 | X |
| 점 | 010.1234.5678 | X |
| 일반 전화 | 02-1234-5678 | X |
| 구형 | 011-123-4567, 016-123-4567 | X |

**영향**: 정규식 1개로 탐지율 95%+ 주장은 과대 추정. 실제로는 70~80%일 수 있음.

**권고**: 종합 정규식 패턴 추가:
```
(?:\+82[-\s]?)?0?1[016789][-.\s]?\d{3,4}[-.\s]?\d{4}
```

## U6. 이메일 정규식 정확도 (Low)

**위치**: 04_pii_and_validation.md §2.3

**문제**: `[\w.-]+@[\w.-]+\.\w+` 정규식이 과도하게 광범위하여 **기술 문서의 라이브러리/패키지 참조**를 잡을 수 있음 (예: `user@latest`, `git@github.com`).

**영향**: 마스킹 오탐(false positive)이 LLM 입력 품질 저하

**권고**: 이력서/JD 맥락에서는 크게 문제되지 않으므로 Low. 다만 tech_stack 필드에서 `git@github.com`이 마스킹되지 않도록 **마스킹 적용 범위를 텍스트 필드로 한정** 권고.
