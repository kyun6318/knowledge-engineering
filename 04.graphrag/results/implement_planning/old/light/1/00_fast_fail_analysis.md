# Fast Fail 전략이 GraphRAG KG 프로젝트에 부적합한 이유

> **문서 목적**: standard.1 계획(26~33주)을 8주 Fast Fail로 압축하는 전략의 구조적 한계를 분석하고,
> 데이터/KG 프로젝트에서 Fast Fail이 왜 위험한지 근거를 정리한다.
>
> **결론**: Fast Fail 대신 검증 사이클을 보존하면서 일정을 압축하는 "Staged Fast Fail"(12~14주)을 채택한다.
>
> 작성일: 2026-03-08

---

## 1. Fast Fail 전략 개요

Fast Fail(빠른 실패) 전략은 불확실성이 높은 프로젝트에서 **최소 비용으로 실행 가능성을 검증**하는 접근법이다.
핵심 원칙은 "동작한다/안 한다"를 빠르게 판별하여 자원 낭비를 줄이는 것이다.

### 소프트웨어 프로젝트에서 유효한 경우

- 새로운 API/프레임워크의 기술적 feasibility 검증
- 사용자 반응을 확인하는 MVP 프로토타입
- 아키텍처 의사결정을 위한 spike
- **공통 특징**: 결과가 이진적(성공/실패)이고, 실패 시 코드만 폐기하면 됨

### 데이터/KG 프로젝트에서 부적합한 이유

아래 5개 근거로 설명한다.

## 2. 근거 1: 품질 신호가 이진적이지 않다

### 문제

Fast Fail은 "동작한다/안 한다"가 명확할 때 유효하다.
그러나 이 프로젝트의 핵심 질문은 연속적 품질 메트릭이다:

| 질문 | 판단 방식 | 필요 조건 |
|------|----------|----------|
| scope_type 분류 정확도가 70%인가 80%인가? | Gold Set 200건 이상 평가 | 프롬프트 3~5회 튜닝 반복 |
| CandidateContext 추출 F1이 55%인가 70%인가? | 전문가 라벨링 + 정량 비교 | 50건 PoC → 200건 평가 |
| Embedding 한국어 분별력이 충분한가? | Mann-Whitney U 통계 검정 | 20쌍 이상 비교 데이터 |
| stage_estimate 정확도가 75%인가 85%인가? | NICE 데이터 대조 | NICE DB 접근 + 매칭 로직 |

### 8주 Fast Fail에서 발생하는 문제

8주 안에 rush하면 "F1이 45%"라는 결과를 얻는다.
이때 두 가지 해석이 가능하다:

```
해석 A: 접근법 자체가 틀렸다 → 폐기해야 한다
해석 B: 프롬프트를 덜 다듬었을 뿐이다 → 2주만 더 튜닝하면 70% 가능
```

**둘 중 어느 것인지 구분할 수 없다.**
이것이 "Fail 신호 자체가 노이즈"가 되는 핵심 문제다.

### 정상적인 검증 사이클

```
[PoC 50건] → 품질 측정 → 프롬프트 튜닝 (2~3회) → 재측정
    → F1 개선 곡선 확인 → 수렴 판단 → 모델/전략 확정
```

이 사이클을 생략하면 "실패했다"는 결론의 신뢰도가 없다.

## 3. 근거 2: 핵심 파이프라인의 직렬 의존성

### 파이프라인 체인

```
파싱 → 섹션분할 → 경력블록분리 → PII마스킹 → LLM추출 → Context생성 → Graph적재 → Embedding → Mapping
```

이 체인의 특성:

1. **오류 전파(Error Propagation)**: 앞 단계의 품질이 뒤 단계의 상한을 결정한다
2. **디버깅 불가**: 최종 결과가 나쁠 때 "어느 단계에서 열화되었는가"를 추적하려면 각 단계별 검증 데이터가 필요하다
3. **재처리 비용**: 앞 단계를 수정하면 뒤의 모든 단계를 다시 실행해야 한다

### 구체적 시나리오

```
파싱 정확도 85% × 섹션분할 90% × LLM추출 80% = 최종 정확도 61.2%

vs.

파싱 정확도 95% × 섹션분할 95% × LLM추출 85% = 최종 정확도 76.7%
```

Fast Fail에서 최종 61.2%를 보고 "접근법 실패"로 판단하면,
실제로는 파싱만 개선하면 76.7%까지 올라갈 수 있는 상황을 놓친다.

### 8주 일정에서의 현실

| 단계 | 최소 구현 | 검증+튜닝 | Fast Fail 할당 | 부족분 |
|------|---------|----------|-------------|-------|
| 전처리 (파싱+분할+PII) | 1.5주 | 0.5주 | 1주 | -1주 |
| CompanyContext | 1주 | 0.5주 | 0.5주 | -1주 |
| CandidateContext | 2주 | 1주 | 1.5주 | -1.5주 |
| Graph 적재 | 1주 | 0.5주 | 0.5주 | -1주 |
| Embedding+Mapping | 1주 | 0.5주 | 0.5주 | -1주 |
| Phase 0 검증 | 2주 | - | 1주 | -1주 |
| **합계** | **8.5주** | **3주** | **5주** | **-6.5주** |

구현 최소치(8.5주)만으로도 8주를 초과하며, 검증 시간이 전혀 없다.

## 4. 근거 3: 규모 전환 시 다른 종류의 문제가 출현한다

### 규모별 문제 스펙트럼

| 규모 | 드러나는 문제 | Fast Fail에서의 위험 |
|------|------------|-------------------|
| **50건 (PoC)** | 프롬프트 품질, 스키마 적합성, LLM 응답 포맷 | PoC 없이 대량 처리 시 프롬프트 결함이 전량에 영향 |
| **1,000건 (MVP)** | Batch API 연동, 파싱 실패 3-tier, dead-letter 처리, 엣지케이스 | MVP 검증 없이 규모 확장 시 실패율 예측 불가 |
| **450K건 (전체)** | chunk 관리, 비용 폭증, Neo4j Professional 전환, rate limit, 메모리 | 소규모 테스트 없이 전량 처리 시 비용 폭발 |

### 구체 사례: Batch API 비용

```
Phase 1 MVP (1,000건): $1.15 → 프롬프트 문제 발견 → 수정 후 재처리 $1.15
Phase 2 전체 (450K건): $1,350 → 프롬프트 문제 발견 → 재처리 $1,350
  (한국어 토큰 보정 적용: 450K × $0.00300/건 [standard.21])

Fast Fail에서 MVP 단계 생략 시:
  → 첫 번째 전체 처리 $1,350 + 수정 후 재처리 $1,350 = $2,700 (2배 비용)
```

### 구체 사례: Neo4j 스키마 변경

```
MVP (1,000건): 스키마 수정 → 재적재 5분
전체 (450K건): 스키마 수정 → 재적재 8~12시간 + Vector Index 재생성

Fast Fail에서 MVP 없이 전체 적재 후 스키마 변경 발견:
  → 12시간 재적재 + 중간 서비스 중단
```

## 5. 근거 4: "Fail"의 비용이 소프트웨어와 근본적으로 다르다

### 소프트웨어 Fast Fail vs 데이터 Fast Fail

| 차원 | 소프트웨어 프로토타입 | 데이터/KG 프로젝트 |
|------|-------------------|-----------------|
| 실패 시 폐기 대상 | 코드 (git reset) | 코드 + 처리된 데이터 + API 비용 + 인프라 상태 |
| 폐기 비용 | ~$0 | $1,350+ (Batch API) + Cloud Run 비용 + 시간 |
| 실패 원인 진단 | 코드 리뷰로 가능 | 파이프라인 각 단계별 검증 데이터 필요 |
| 재시작 비용 | 코드만 재작성 | 전체 파이프라인 재실행 (물리적 시간) |
| 되돌리기 | git revert | Neo4j 노드 삭제 + GCS 정리 + BigQuery 정리 |

### 비용 비교 시나리오

| 시나리오 | Fast Fail 8주 | standard.1 단계적 검증 |
|----------|------------|-------------|
| LLM 모델 선택 오류 발견 | $1,350 Batch API 재처리 | $1.15 (1,000건) 재처리 |
| Embedding 차원 변경 필요 | 450K Vector Index 재생성 (6시간) | 1,000건 재생성 (5분) |
| Graph 스키마 변경 | 450K 노드 재적재 (12시간) | 1,000건 재적재 (5분) |
| 프롬프트 구조 변경 | 전량 재추출 ($1,350+) | PoC 50건 재추출 ($1) |
| **최악 시나리오 누적 비용** | **~$3,000 + 2주 재처리** | **~$10 + 1일 재처리** |

## 6. 근거 5: 의사결정 포인트가 순차 의존적이다

### standard.1의 의사결정과 의존 관계

```
Phase 0 의사결정 (Week 4-5)
    │
    ├─ [D1] Embedding 모델 확정
    │   └─ → Vector Index 차원 결정 → Neo4j 스키마 확정
    │
    ├─ [D2] 텍스트 추출 방법 확정
    │   └─ → 전처리 파이프라인 구현 방식 결정
    │
    ├─ [D3] LLM 모델 선택
    │   └─ → Batch API 비용 구조 확정 → 전체 처리 비용 예측
    │
    ├─ [D4] PII 전략 확정
    │   └─ → API vs On-premise → 인프라 아키텍처 결정
    │
    ├─ [D5] 오케스트레이션 도구
    │   └─ → Cloud Workflows vs Prefect → 구현 스택 결정
    │
    ├─ [D6] LLM 호출 전략
    │   └─ → 단건 vs 묶음 → CandidateContext 구현 방식
    │
    ├─ [D7] 섹션 분할 전략
    │   └─ → Rule-based vs LLM → 전처리 모듈 설계
    │
    └─ [D8] Graph DB 플랜
        └─ → Free vs Professional → 비용 구조 + 노드 제한
```

### Fast Fail에서의 의사결정 생략 결과

```
"일단 Haiku로 가자" → Phase 2에서 Sonnet이 필요했음을 발견
  → 450K 전량 재처리 ($1,350 → $8,100, Sonnet 6배 비용 증가)

"일단 768차원으로 가자" → gemini-embedding-001이 3072차원이 더 좋았음
  → Vector Index 전량 재생성 + Neo4j 스키마 변경 + 재적재

"PII는 나중에" → 법무 검토에서 마스킹 필수 판정
  → 전체 파이프라인 재설계 + 재처리
```

의사결정을 건너뛰는 것은 "빠른 실패"가 아니라 **"비싼 수정"을 미루는 것**이다.

## 7. 결론 및 대안

### Fast Fail이 부적합한 핵심 요약

| # | 근거 | 핵심 |
|---|------|------|
| 1 | 품질 신호가 이진적이지 않다 | Fail인지 튜닝 부족인지 구분 불가 |
| 2 | 파이프라인이 직렬 의존적이다 | 어디서 실패했는지 진단 불가 |
| 3 | 규모 전환 시 다른 문제가 출현한다 | 소규모 검증 없이 대규모 진입은 도박 |
| 4 | Fail의 비용이 코드 폐기와 다르다 | API 비용 + 처리 시간이 매몰 비용 |
| 5 | 의사결정이 순차 의존적이다 | 결정 생략 = 비싼 수정의 이연 |

### 대안: Staged Fast Fail (13~15주)

standard.1(26~33주)를 8주가 아닌 **13~15주로 압축**하되, 다음을 보존한다:

1. **Phase 0 의사결정** — 모델/Embedding/PII 확정 (2.5주)
2. **Phase 1 MVP 1,000건** — 소규모 E2E 검증 + 테스트 인프라 (6.5~7주)
3. **Phase 2 코어 KG** — 크롤링 제외, 전체 데이터 처리 + 품질 평가 (4~5주)

이 전략은 "빨리 실패한다"가 아니라 **"빨리 검증한다"**에 초점을 맞춘다.
