# V1 계획 비교 리뷰 + 타당성 분석

> **대상**: `c_00~c_03` (c계열, 4개 문서) vs `g_01` (g계열, 1개 문서)
> **목적**: 두 계획의 차이를 구조적으로 비교하고, 각 리뷰 포인트의 타당성을 분석하여 v2 계획의 기반을 마련한다.

---

## Part 1. 구조 비교

| 차원 | g계열 (`g_01`) | c계열 (`c_00~c_03`) |
|------|-------------|-----------------|
| **문서 수** | 1개 (97줄) | 4개 (672줄 합산) |
| **태스크 추적성** | 없음 (Phase 단위 요약만) | **v5의 73개 태스크를 1:1로 S/G/공동 분류** |
| **Data Contract** | 필수 필드 목록만 (3줄) | **JSON 스키마 3종 + 필수 조건 + 전달 경로** |
| **타임라인** | Phase별 기간만 (2.5/3/3.5주) | **주차별 병렬 간트 + 인터페이스 포인트 5개** |
| **비용** | 없음 | **팀별 Phase별 비용 분배표 ($2,115 vs $3,435~6,785)** |
| **리스크** | 2개 (싱크 불일치, 필터링 역전) | **4개 + 완화 방안** |
| **Go/No-Go** | 없음 | **Phase별 통과 조건 명시** |
| **S&F 팀 정의** | 역할 3줄 서술 | **범위 6분류 + 산출물 5종 명세 + 타임라인 + 비용** |
| **GraphRAG 팀 정의** | 역할 3줄 서술 | **독립 실행 계획 G-0~G-4 + 대기 구간 선행 작업** |

## Part 2. 리뷰 포인트 (15건)

### ✅ c계열이 g계열보다 우수한 점 (10건)

| # | 포인트 | 상세 | 영향도 |
|---|--------|------|--------|
| C1 | **태스크 추적성** | g계열은 v5의 개별 태스크가 어느 팀으로 갔는지 추적 불가. c계열은 73개 전수 분류. | **CRITICAL** |
| C2 | **Data Contract 구체성** | g계열은 "person_id, chapter_id, skills" 필드 이름만 나열. c계열은 JSON 샘플 + 시간순 정렬 조건 + GCS 경로까지 명시. | **HIGH** |
| C3 | **대기 구간 활용** | g계열은 "S&F 기다린다"만 표시. c계열은 대기 중 선행 작업(API 골격, PII 미들웨어 등)을 명시. | **HIGH** |
| C4 | **비용 분배** | g계열은 비용 언급 없음. c계열은 S&F $2,115~2,265 / GraphRAG $3,435~6,785로 정량 분배. | **HIGH** |
| C5 | **Go/No-Go 기준** | g계열은 없음. c계열은 Phase별 통과 조건(적재량, 쿼리 p95, 매칭 Top-10 등) 명시. | **HIGH** |
| C6 | **S&F 산출물 명세** | g계열은 "JSON 전달"만 기술. c계열은 PoC→1K→480K→JD→보강 5단계로 시점+형식+경로 정의. | **HIGH** |
| C7 | **리스크 4건** | g계열은 2건(싱크, 역전). c계열은 +2건(S&F 지연 블로킹, 경계 모호) 추가. | MEDIUM |
| C8 | **Phase 4 포함** | g계열은 Phase 1~3만 다룸. c계열은 Phase 0/4까지 전 Phase 포괄. | **HIGH** |
| C9 | **인력 명시** | g계열은 인력 미언급. c계열은 GraphRAG DE 1 + MLE 1 명시. | MEDIUM |
| C10 | **의사결정 주체 재배치** | g계열은 없음. c계열은 v5의 11개 의사결정 포인트를 S&F/GraphRAG/공동으로 재배치. | MEDIUM |

### ⚠️ g계열이 c계열보다 우수하거나 c계열에 부족한 점 (5건)

| # | 포인트 | 상세 | 영향도 |
|---|--------|------|--------|
| G1 | **이벤트 기반 연동(Kafka/PubSub)** | g계열 Phase 1-1에서 "Kafka 또는 GCS Event/PubSub 이벤트 기반 적재" 명시. c계열은 산출물 ③에서 PubSub 언급하지만 GraphRAG 측 수신 아키텍처 미기술. | MEDIUM |
| G2 | **Vector DB 외부화 대안** | g계열은 "Milvus/Pinecone 등 Vector DB" 또는 "Elasticsearch" 대안을 명시적으로 언급. c계열은 c_03에서 "분리 시 Vector DB(Milvus 등) 또는 Neo4j 내 유지"라고만 기술하고 선택 기준 부재. | MEDIUM |
| G3 | **마이크로서비스 강조** | g계열은 "Microservice/Polyglot 아키텍처"를 핵심 효과로 강조. c계열은 분리 배경에서 문제점 3개만 나열하고 아키텍처적 비전이 약함. | LOW |
| G4 | **GraphRAG 팀 기간 더 공격적** | g계열은 Phase 1~3을 약 9주로 추정. c계열은 12.5주. 차이 원인: c계열은 대기 구간(W2~5, W7~9)을 기간에 포함하고 g계열은 순수 작업 기간만 표기. | **HIGH** — 기간 산정 기준 차이 |
| G5 | **네트워크 홉(Hop) 리스크** | g계열 §3에서 S&F→GraphRAG API 호출 시 "네트워크 통신 추가" 리스크를 간접적으로 시사. c계열은 API 체인 다이어그램만 있고 레이턴시 SLA 미정의. | MEDIUM |

### G계열 우수 포인트 타당성

## Part 4. v2 계획 개선 방향 (15건 종합)

v2는 **c계열 4개 문서를 기반(골격 유지)**으로 하되, g계열 및 리뷰에서 도출된 **5개 개선점을 반영**한다.

| # | 개선 항목 | 반영 위치 | 상세 |
|---|---------|---------|------|
| I1 | GraphRAG 수신 아키텍처 명시 | `00_overview §4` | GCS JSONL 도착 → PubSub 통지 → Cloud Run Job 자동 적재 트리거 |
| I2 | Vector DB 선택 기준 추가 | `00_overview §2.1` | Neo4j Vector Index 유지 기준(Person < 1M) vs 외부 Vector DB 전환 기준 |
| I3 | 아키텍처 비전 1줄 추가 | `00_overview §1` | "Polyglot Persistence 아키텍처로 각 DB의 주특기 활용" |
| I4 | 순수 작업/대기 구간 분리 표기 | `02_graphrag §1` | 12.5주 중 순수 작업 ~8주 + 대기 ~4.5주 (대기 중 선행 작업 포함) |
| I5 | API 체인 레이턴시 SLA 추가 | `00_overview §4.2` | S&F API p95 < 500ms, GraphRAG API p95 < 2s, 전체 체인 p95 < 3s |
