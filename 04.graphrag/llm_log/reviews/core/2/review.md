# GraphRAG Core 확장 계획 v2 — 심층 리뷰

> **리뷰 대상**: `03.ml-platform/plans/graphrag/core/2/` (7개 문서)
> **리뷰 관점**: 타당성, 실현 가능성, 과도한 설계, 부족한 설계
> **리뷰 일자**: 2026-03-09

## 1. 총평

v2는 v1 리뷰의 Must 6건 + Should 8건을 **전부 반영**하여 실현 가능성이 크게 향상되었다. Phase 2 기간 연장(8주), 크롤링 선택적 분리, 에이전트 서빙 API 추가, Organization ER 2주 확대는 v1의 핵심 약점을 정확히 보완한다.

그러나 **심층 분석 결과 8건의 신규 이슈**가 발견되었다. 대부분 LOW~MEDIUM 수준이지만, 일부는 프로덕션 운영 안정성에 직접 영향을 미치므로 반영이 필요하다.

**결론**: v2 계획은 **실행 가능하며 추가 리비전 없이 진행 가능**하되, 아래 지적 사항을 Phase 진행 중 점진적으로 보완할 것을 권장한다.

## 4. 과도한 설계 (Over-Engineering)

### 4.1 Phase 2 기본 시나리오 여유 — 활용 계획 미흡 (LOW)

기본 시나리오(12h/라운드)에서 5주 중 22.5일이면 완료되어 **12.5일 여유**가 생긴다. 이 여유에 대한 명시적 활용 계획이 "2-4와 병행"으로만 기술되어 있다.

**현재 상태**: 여유가 있는 것은 리스크 관리 측면에서 긍정적이며, 8주를 7주로 줄이는 것은 권장하지 않음. 비관 시나리오 대비가 더 중요하다.

**권장**: 여유 발생 시 활용 우선순위 명시 — ① Phase 3 사전 준비(NICE DB 접근, JD 샘플 확보) ② 품질 메트릭 고도화 ③ 프롬프트 최적화

### 4.2 Gold Label 비용 — 검증 필요 (LOW)

Phase 4 Gold Test Set 비용이 **$5,840** (전문가 1명 × 40시간)으로, 전체 인프라+LLM 비용($2,375~3,253)의 **1.8~2.5배**에 달한다.

```
현재 추정:
  전문가 1명 × 40시간 × $146/시간 = $5,840

검토 필요:
  - 200건 매칭 쌍 검수에 40시간이 필요한가?
    (건당 12분 = 200건 × 12분 = 40시간 — 매칭 품질 검수라면 타당)
  - 국내 전문가 대안 ~200만원으로 대체 가능 (04문서에 이미 언급)
  - 200건이 최소 요건인지 (100건으로 축소하면 비용 절반)
```

**권장**: Gold Label 규모를 **100건으로 시작**, 결과에 따라 200건으로 확대하는 2단계 접근 고려. 비용 $2,920 절감 가능.

### 4.3 매칭 가중치 사전 확정 — 과잉 명세 (VERY LOW)

Phase 3-0에서 매칭 가중치(skill=0.35, exp=0.15 등)를 사전 설정했으나, 데이터 없이 추정한 초기값이므로 이 값 자체보다 **튜닝 프로세스**가 더 중요하다. 현재는 50건 수동 검증이 있으나 가중치 재조정까지의 루프가 미설계.

**권장**: Phase 3-4 완료 시 50건 검증 결과 기반 **가중치 재조정 1일 추가** 명시.

## 5. 부족한 설계 (Under-Engineering)

### 5.1 Neo4j AuraDB Free의 Auto-Pause 문제 (HIGH)

**핵심 문제**: AuraDB Free는 **72시간 비활성 시 자동 일시 중지**된다. Phase 1에서 GraphRAG REST API를 Cloud Run Service로 배포하고 AuraDB Free에 연결하는데, API 트래픽이 없는 기간(주말 등)에 Neo4j가 **자동 중지**될 수 있다.

```
영향 시나리오:
  - 금요일 오후 에이전트 팀에 API 연동 안내 (Week 6)
  - 월요일 아침 에이전트가 API 호출 → Neo4j 중지 상태 → 503 에러
  - 재활성화에 수 분~수십 분 소요 (AuraDB Free)

대응 옵션:
  A) Cloud Scheduler로 12시간마다 health check 호출 (비용 $0)
  B) Phase 1부터 Professional 사용 ($65/월 추가)
  C) API에 Neo4j 재활성화 로직 추가 (cold-start 허용)
```

**권장**: **옵션 A(Scheduler health check)를 Phase 1에 추가**. 가장 저비용이며, 운영 복잡도도 낮다. API의 `/health` 엔드포인트를 12시간마다 호출하는 Cloud Scheduler Job 1개만 추가하면 된다.

### 5.2 API 레이어의 PII 노출 위험 (MEDIUM)

GraphRAG API가 후보자 데이터를 반환하는데, **API 응답에 포함되는 PII 범위가 정의되어 있지 않다**.

```
현재 API 설계:
  GET /api/v1/candidates/{candidate_id}
  Response: { "person": {...}, "chapters": [...], "skills": [...] }

질문:
  - "person" 객체에 이름, 연락처 등 PII가 포함되는가?
  - 이력서 원본 텍스트가 응답에 포함되는가?
  - PII 마스킹이 적용된 상태로 반환하는가?
  - 에이전트가 PII를 재조합할 수 있는가?
```

**권장**: Phase 1 API 설계 시 **API 응답 필드 정의서**에 PII 포함 여부를 명시하고, 필요 시 **응답 레벨 PII 필터링** 로직을 추가. 이는 법무 PII 검토(Week 0)와 연동되어야 한다.

### 5.3 MAPPED_TO 관계 규모 추정 부재 (MEDIUM)

매칭 임계값 0.4 이상이면 MAPPED_TO 관계를 생성하는데, **예상 관계 수가 추정되어 있지 않다**.

```
규모 추정:
  - Person: 360K~450K
  - Vacancy: 10K
  - 만약 각 Vacancy에 평균 50명이 매칭된다면: 10K × 50 = 500K 관계
  - 만약 각 Vacancy에 평균 500명이 매칭된다면: 10K × 500 = 5M 관계

영향:
  - Neo4j Professional 인스턴스 크기 결정에 영향
  - 매칭 배치 처리 시간에 영향 (5M 관계 UNWIND 적재 시간)
  - 쿼리 성능에 영향 (MAPPED_TO traversal)
```

**권장**: Phase 3-0(매칭 설계 문서)에서 **예상 관계 수 추정**과 이에 따른 **Neo4j 인스턴스 사이징 검토**를 포함시킬 것. 임계값 0.4가 실제 데이터에서 어느 정도의 관계를 생성하는지 Phase 3-4 초기에 **소규모 테스트(JD 100건 × Person 1,000건)**로 사전 검증.

### 5.4 chapter_id 생성 방식의 비결정성 (MEDIUM)

Graph 적재 코드에서 `chapter_id = f"{candidate_id}_ch{j}"`로 인덱스 j 기반으로 ID를 생성한다. 동일 후보자의 이력서가 재처리될 때 경력 블록 순서가 바뀌면 **다른 chapter_id가 생성**되어 중복 노드가 발생할 수 있다.

```python
# 현재 코드 (문제 가능):
chapter_id = f"{c['candidate_id']}_ch{j}"
# j=0이 "삼성전자 3년" → 재처리 시 j=0이 "네이버 2년"이 될 수 있음

# 개선안 A: 내용 기반 해시
chapter_id = f"{candidate_id}_{hashlib.md5(f'{company}_{role}_{start_date}'.encode()).hexdigest()[:8]}"

# 개선안 B: Phase 4의 DETACH DELETE 재생성 전략에 의존
# → 이미 설계되어 있음 (04 문서 §4-4)
```

**평가**: Phase 4에서 이력서 업데이트 시 "DETACH DELETE 후 재생성" 전략이 이미 설계되어 있으므로, **Phase 2 대규모 최초 적재에서는 문제없다**. 단, 증분 처리에서 동일 candidate의 재적재가 발생할 때 반드시 DETACH DELETE를 선행해야 한다는 점을 **Phase 2 적재 코드에도 명시**할 것.

**심각도**: 증분 처리 전략이 이미 존재하므로 MEDIUM→LOW로 하향 가능하나, Phase 2에서의 명시가 필요.

### 5.5 Gemini Flash 병행 시 파이프라인 분기 설계 부재 (MEDIUM)

v2에서 Gemini Flash 대안을 Week 0에서 검증하도록 했으나, **실제 병행 시 구현 설계가 없다**.

```
부족한 부분:
  1. 두 API의 프롬프트 포맷 차이 (Anthropic XML tags vs Gemini 일반 텍스트)
  2. 응답 JSON 구조 차이 처리
  3. BigQuery batch_tracking에 api_provider 컬럼 추가 필요
  4. 품질 비교 기준 (동일 건을 양쪽에서 처리한 결과 비교)
  5. 병행 비율 조정 로직 (품질 기반 동적 분배?)
```

**권장**: Phase 2-0(리팩토링)에서 **LLM 추출 모듈에 provider 추상화 레이어** 추가를 태스크로 명시. 실제 병행 필요 확률이 낮으므로(동시 5+ 이면 Anthropic 단독 가능) 추상화 인터페이스만 설계하고 Gemini 구현은 필요 시 추가.

### 5.6 Neo4j Professional 인스턴스 사이징 부재 (LOW)

비용 추정에서 Neo4j Professional을 **$65~200/월**로 넓은 범위로 잡았으나, 실제 필요 인스턴스 크기에 대한 분석이 없다.

```
데이터 규모 추정:
  Phase 2 완료 시:
    Person: 360K+ 노드
    Chapter: 1.8M+ 노드 (Person × 5 avg)
    Skill: ~5K 노드
    Role: ~3K 노드
    Organization: ~50K 노드
    관계: ~4M+
    Vector Index: 1.8M × 768d × 4byte = ~5.5GB

  Phase 3 완료 시:
    + Vacancy: 10K 노드
    + MAPPED_TO: 500K~5M 관계 (추정)
    + vacancy_embedding: 10K × 768d

AuraDB Professional 플랜:
  - 8GB: $65/월 → 5.5GB Vector Index만으로 한도 근접
  - 16GB: ~$130/월 → Phase 2 완료 시 적합
  - 32GB: ~$260/월 → Phase 3 완료 시 필요할 수 있음
```

**권장**: Phase 2-2(Neo4j 전환) 시 **1,000건 적재 후 메모리 사용량 외삽**으로 인스턴스 크기를 결정하는 절차를 추가. 비용 추정의 $65~200 범위는 유지하되, **$200 초과 시 대응 방안**(Vector Index 분리, 768d→384d 차원 축소 등)도 고려.

### 5.7 잔여 배치 처리 모니터링 부재 (LOW)

Phase 2에서 80% 완료 후 잔여 20%를 "Phase 3 백그라운드"로 처리한다고 했으나, **완료 시점 추적 방법과 Phase 3 리소스 영향**이 명시되어 있지 않다.

```
질문:
  - 잔여 90K건의 백그라운드 처리가 Phase 3의 DE/MLE 작업과 리소스 충돌하지 않는가?
  - Batch API 동시 활성 batch를 Phase 3 JD/CompanyContext 처리와 공유해야 하는가?
  - 잔여 처리 완료 시점을 어떻게 추적하는가?
```

**권장**: Phase 3 주간 리포트에 **잔여 배치 처리 현황 포함** (BigQuery batch_tracking 쿼리). Batch API 동시 사용 시 **Phase 3 작업에 최소 2 batch 예약**, 잔여 처리에 나머지 할당.

### 5.8 증분 파이프라인 유입량 근거 부재 (LOW)

Phase 4에서 일일 증분 유입량을 **~1,000건/일**로 가정했으나, 이 수치의 근거가 "가정"으로만 기술되어 있다.

```
현재:
  "가정: 월 20,000~30,000건 유입 → 일 ~1,000건 (영업일 기준)"

영향:
  - 일 1,000건 → Batch API $3/일 → 월 $60
  - 일 100건 → 주 1회 증분으로 변경 가능, 월 $3
  - 일 5,000건 → Batch API + 별도 Queue 필요, 월 $300

Phase 0에서 확인하도록 명시되어 있으나(04 문서 §4-4):
  "Phase 0 프로파일링에서 실제 유입량 확인 후 보정"
  → 이 부분이 Phase 0 산출물에는 명시되어 있지 않음
```

**권장**: Phase 0 산출물 체크리스트에 **"일일 이력서 유입량 확인"** 항목 추가.

## 11. 결론

v2는 v1의 **모든 핵심 리스크를 체계적으로 해소**한 성숙한 개선판이다.

**강점:**
- v1 HIGH 리스크 3건 전부 해소 (Phase 2 기간 연장, 크롤링 분리, 에이전트 API)
- Go/No-Go 게이트 + 버퍼 2주로 일정 리스크 관리
- 자동 품질 메트릭 + 통계적 샘플링으로 대규모 처리 품질 보장
- 서비스 계정 분리, 롤백 전략 등 운영 안정성 보강

**신규 발견 이슈:**
- AuraDB Free Auto-Pause (HIGH — 대응 간단)
- API PII 노출 범위 미정의 (MEDIUM)
- MAPPED_TO 관계 규모 미추정 (MEDIUM)
- 비용 문서 간 경미한 불일치 ($20~$55)

**최종 판단**: 전체 27주, $8,215~8,890 계획은 **2명 체제에서 실행 가능한 수준**이다. 신규 이슈 N1~N4의 즉시 반영(추가 비용/일정 없음)을 권장하며, **추가 리비전 없이 실행 단계로 진입 가능**하다.
