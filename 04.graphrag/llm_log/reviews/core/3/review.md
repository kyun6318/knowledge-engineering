## 1. 총평

v3는 v2 리뷰의 N1~N9 전 항목과 extraction logic v12를 **빠짐없이 반영**한 완성도 높은 계획이다. AuraDB Auto-Pause 대응, PII 필드 정의, MAPPED_TO 규모 추정, provider 추상화 등 v2에서 지적된 부족한 설계가 모두 보완되었고, v12의 적응형 호출/Hybrid 섹션 분리/v19 관계명이 각 Phase에 정합적으로 통합되었다.

그러나 v2의 **부족한 설계 보강 과정에서 일부 과도한 설계가 새로 유입**되었고, 600K 볼륨 확대에 따른 **실현 가능성 리스크**가 v2보다 증가했다. 또한 v12 통합 과정에서 **코드 예시의 실제 구현 격차**와 **문서 간 비용 불일치**가 일부 존재한다.

**결론**: v3 계획은 **실행 가능하며 추가 리비전 없이 진행 가능**하다. 다만 아래 7건의 과도한 설계 축소와 5건의 부족한 설계 보강을 Phase 진행 중 점진적으로 반영할 것을 권장한다.

## 3. 과도한 설계 (Over-Engineering)

### O1. PII 매핑 CMEK 인프라 — Phase 0에서의 조기 구축 (MEDIUM)

**현상**: Phase 0(Week 1)에서 Cloud KMS 키링 생성 + CMEK 버킷 생성 + 전용 서비스 계정 생성을 수행한다. 이는 v12 S2의 요구사항이지만, Phase 0는 **PoC + Go/No-Go** 단계이다.

```
Phase 0에서의 CMEK 구축 비용:
  - KMS 키 생성: $0.06/월
  - 전용 버킷: $0
  - 전용 서비스 계정: $0
  - 설정 시간: DE 0.5일

문제:
  - Go/No-Go에서 No-Go가 나오면 불필요한 인프라가 생성됨
  - Phase 0의 1주(5일)에서 DE가 환경 구성에 2.5일(기존 2일 + CMEK 0.5일) 소요
  - 크롤링 분석까지 하면 DE의 Week 1이 매우 타이트
```

**권장**: CMEK 버킷 생성을 **Phase 1-B(전처리 모듈 구현, Week 2-3)로 이동**. Phase 0에서는 일반 GCS 버킷으로 PoC를 진행하고, Go 판정 후 CMEK를 적용하는 것이 합리적이다. Go/No-Go 이전에 PII 매핑 테이블의 암호화 수준을 결정할 필요는 없다.

### O2. LLM Provider 추상화 — 과도한 인터페이스 설계 (LOW)

**현상**: Phase 2-0에서 ABC 추상 클래스 + 팩토리 패턴을 구현하지만, **GeminiProvider는 `pass`(빈 구현)**이다. Anthropic Batch API 한도에 도달할 확률이 낮은 상태에서 추상화 레이어를 사전 구축한다.

```
현재 설계:
  - LLMProvider (ABC): extract, submit_batch, poll_batch 3개 메서드
  - AnthropicProvider: 실제 구현
  - GeminiProvider: pass (빈 구현)
  - get_provider() 팩토리

실제 필요성:
  - Anthropic 단독으로 600K 처리 가능 (동시 10 batch)
  - Gemini 병행은 "보험" 수준
  - 추상화 없이도 나중에 if/else 분기로 충분
```

**평가**: 추상화 자체는 0.5일 작업이고 코드 품질 향상에 기여하므로 **해로운 수준은 아니다**. 다만 v2 리뷰에서 "인터페이스만 설계"라고 권장했는데, v3에서는 팩토리 패턴과 BigQuery 컬럼 추가까지 확대되었다. **현재 수준 유지**를 권장하되, 추가 확장(예: 라우팅 로직, 품질 기반 동적 분배)은 자제할 것.

### O3. 가중치 튜닝 Grid Search — 과도한 자동화 (LOW)

**현상**: Phase 3-4에서 50건 검증 결과 기반 grid search 코드가 포함되어 있다. 5개 가중치를 3×3×3 격자로 탐색하는 27개 조합을 자동 평가한다.

```
문제:
  - 50건으로 Pearson 상관관계를 측정하는 것은 통계적으로 약함
  - 27개 조합 × 50건 = 1,350회 스코어링은 과도하지 않으나, 50건의 표본 크기에서 r 차이 0.01~0.02는 무의미
  - 검증 50건 중 25건이 "적합", 20건이 "부적합", 5건이 "애매"라면, 실질적으로 2-3개 조합만 비교하면 충분
  - v2 리뷰에서는 "가중치 재조정 1일"만 권장했는데, 자동화 코드까지 작성됨
```

**권장**: Grid search를 **수동 비교**로 대체. 50건 규모에서는 전문가가 Top-10 결과를 3~4개 가중치 조합으로 비교하는 것이 더 신뢰도가 높다. 자동화는 Gold Label 200건 이상 확보 후 Phase 5에서 검토.

### O4. operating_model 진정성 체크 키워드 — 하드코딩 과잉 (LOW)

**현상**: v12 C3에서 "LLM 진정성 검증 → 단순 confidence 규칙"으로 단순화하라고 했는데, v3에서는 키워드 하드코딩으로 구현했다.

```python
# 현재 코드:
has_specific_context = any(
    keyword in str(evidence)
    for keyword in ["주 스프린트", "리뷰 빈도", "CI/CD", "OKR", "주도적"]
)

# 문제:
# - 5개 키워드로 operating_model의 speed/autonomy/process를 판별하는 것은 취약
# - "2주 스프린트"는 매칭되지만 "bi-weekly sprint"는 누락
# - "주도적"은 autonomy 관련이지만 speed/process에도 매칭됨
# - JD 텍스트에 "OKR"이 없어도 목표 기반 문화일 수 있음
```

**권장**: v12 C3의 원래 의도대로 **"evidence 텍스트가 N자 이상이면 유효"** 같은 더 단순한 규칙으로 대체하거나, 키워드를 facet별로 분리하고 외부 설정 파일로 관리. 단, 이 필드 자체가 v1 INACTIVE 상태의 culture_fit(10%)에만 영향하므로 **실질 영향은 매우 낮다**.

### O5. 증분 처리의 DETACH DELETE 4단계 — Phase 4에서의 과도한 세분화 (MEDIUM)

**현상**: Phase 4의 증분 처리에서 Chapter 삭제를 4단계로 나눈다: (1) 공유 노드 관계 제거 → (2) 비공유 노드 관계+노드 삭제 → (3) HAS_CHAPTER + Chapter 삭제 → (4) NEXT_CHAPTER 재연결.

```
문제:
  - 4개의 별도 Cypher 쿼리를 순차 실행하는 것은 트랜잭션 실패 위험을 4배 증가
  - v12 §1.4의 핵심은 "공유 노드(Skill, Organization, Role) 삭제 방지"인데,
    이는 단순히 DETACH DELETE 대신 Chapter 노드와 비공유 노드만 삭제하면 됨
  - Neo4j 4.4+에서는 단일 쿼리로 가능:
    MATCH (p:Person)-[:HAS_CHAPTER]->(c:Chapter)
    WHERE p.person_id = $pid
    OPTIONAL MATCH (c)-[r1]->(owned) WHERE owned:Outcome OR owned:SituationalSignal
    OPTIONAL MATCH (c)-[r2]->(shared) WHERE shared:Skill OR shared:Organization
    DELETE r1, r2, owned, c
  - NEXT_CHAPTER 재연결도 동일 트랜잭션에서 처리 가능
```

**권장**: 4단계를 **2단계**(관계+비공유 노드 삭제 → Chapter 삭제 + NEXT_CHAPTER 재연결)로 통합하거나, 가능하다면 **단일 Cypher 쿼리**로 합쳐 트랜잭션 원자성을 확보할 것. v12의 의도(공유 노드 보호)는 유지하되 구현 복잡도를 낮춘다.

## 4. 부족한 설계 (Under-Engineering)

### U1. 600K 볼륨에서 Phase 2 Batch 처리 실현 가능성 — 타이트해진 일정 (HIGH)

**핵심 문제**: v2에서 450K → v3에서 600K로 볼륨이 33% 증가했으나, Phase 2 기간은 동일한 8주다. 적응형 호출(v12 M1)로 인해 총 호출 수가 450K → 1,140K로 **2.5배 증가**했다.

```
v3 시나리오 분석 (chunk 2,000으로 확대 후):
  낙관 (6h/라운드): 28.5일 → 87% + 여유 6.5일 (v2: 100%)
  기본 (12h/라운드): 40일 → 87% (v2: 100%)              ★ 80% 목표는 달성
  비관 (24h/라운드): 80일 → 44% → Phase 3 백그라운드 필수  ★ v2보다 악화

v2 대비 악화 정도:
  - 기본 시나리오: v2 100% → v3 87% (여유 12일 → 0일)
  - 비관 시나리오: v2 78% → v3 44% (심각한 미달)

추가 리스크:
  - N+1 pass 건(20%)은 chunk 크기 1,000 유지 → 혼합 chunk 관리 복잡
  - 적응형 호출 비율이 v12 가정(80/20)과 다를 경우 비용/시간 변동 큼
```

**권장**:
1. **Phase 2를 9주로 1주 연장** 검토 (Phase 3은 7→6주로 축소 가능, 3-1 JD 파서가 단순)
2. 또는 **처리 우선순위 전략 명시**: DB 500K를 먼저 처리(적응형 호출 비율 검증) → 파일 100K는 후순위
3. **비관 시나리오 대응 계획**이 "Phase 3 백그라운드"로만 기술되어 있는데, 44%만 완료되면 "백그라운드"가 아닌 **주요 작업**이 됨. Batch API 할당 계획을 Phase 3 초반에 명시적으로 배치할 것.

### U2. PII 마스킹 코드의 치명적 버그 — 반복 치환 시 오프셋 오류 (HIGH)

**현상**: `mask_pii()` 함수에서 전화번호를 순회하며 치환하는데, **치환 후 텍스트 길이가 변경**되어 이후 매치의 `m.start()/m.end()` 오프셋이 **틀어진다**.

```python
# 현재 코드 (02_phase1 §1-B):
for pattern in PHONE_PATTERNS:
    for i, m in enumerate(re.finditer(pattern, masked)):
        token = f"[PHONE_{i:03d}]"
        mapping[token] = m.group()
        masked = masked[:m.start()] + token + masked[m.end():]
        # ↑ 여기서 masked 길이가 변경됨
        # ↑ 다음 re.finditer 호출의 오프셋이 이미 계산된 상태

# 동일 문제가 EMAIL_PATTERN에서도 발생:
for i, m in enumerate(re.finditer(EMAIL_PATTERN, masked)):
    token = f"[EMAIL_{i:03d}]"
    mapping[token] = m.group()
    masked = masked[:m.start()] + token + masked[m.end():]
```

```
구체적 시나리오:
  입력: "연락처: 010-1234-5678, 보조: 010-9876-5432"
  1차 매칭: 010-1234-5678 (start=5, end=18)
  치환 후: "연락처: [PHONE_000], 보조: 010-9876-5432"
  2차 매칭: re.finditer는 원본 기준 오프셋을 사용 → 잘못된 위치에서 치환 시도

  결과: 마스킹 실패 또는 텍스트 손상
```

**권장**: **역순 치환** (뒤에서부터 치환하여 오프셋 보존) 또는 **re.sub()에 콜백 함수 사용**으로 수정.

```python
# 수정 예시:
def mask_phones(text: str) -> tuple[str, dict]:
    mapping = {}
    counter = [0]
    def replacer(m):
        token = f"[PHONE_{counter[0]:03d}]"
        mapping[token] = m.group()
        counter[0] += 1
        return token
    for pattern in PHONE_PATTERNS:
        text = re.sub(pattern, replacer, text)
    return text, mapping
```

**심각도 근거**: PII 마스킹이 실패하면 LLM에 PII가 전달되어 **법적 리스크**가 발생한다. PoC 20건에서는 문제가 드러나지 않을 수 있으나, 600K 대규모 처리에서 다중 전화번호를 가진 이력서에서 반드시 발생한다.

### U4. Hybrid 섹션 분리의 LLM 폴백 — 비용 제어 부재 (MEDIUM)

**현상**: 파일 이력서 100K건 중 30%(30K건)가 LLM 폴백으로 예상되고, 건당 $0.002로 $60이 추정된다. 그러나 패턴 실패율이 30%보다 높으면(예: HWP 이력서의 비정형 포맷) 비용이 급증한다.

```
시나리오:
  - 패턴 성공 70% (예상대로): LLM 30K건 × $0.002 = $60 ✓
  - 패턴 성공 50%: LLM 50K건 × $0.002 = $100
  - 패턴 성공 30%: LLM 70K건 × $0.002 = $140
  - 패턴 성공 10%: LLM 90K건 × $0.002 = $180

비용 자체는 $60~180 범위로 전체에서 작지만, 문제는:
  1. LLM 폴백은 Batch API가 아닌 개별 호출 → 처리 시간 영향 큼
  2. 30K건의 개별 LLM 호출 시간은 ~15시간 (2초/건) vs Batch면 수 시간
  3. Phase 2의 타이트한 일정(U1)에서 추가 병목이 될 수 있음
```

**권장**:
1. **Phase 2-1 Week 8 종료 시점**에서 패턴 성공률을 1,000건 샘플로 검증. 50% 미만이면 패턴 개선에 1일 투자.
2. LLM 폴백도 **Batch API로 묶어 처리**하는 설계 추가 (현재는 개별 호출만 설계됨).
3. **비용 알림 임계값**: Phase 2 Budget Alert($2,800)에 LLM 폴백 비용이 포함되어 있는지 확인.

### U5. 소프트 삭제 후 쿼리 성능 영향 미고려 (LOW)

**현상**: Phase 4에서 소프트 삭제(`is_active = false`)를 도입하면서, "모든 쿼리에 `is_active: true` 필터 필수"로 기술하고 있다. 그러나 Phase 1~3에서 작성된 **Cypher 쿼리 5종과 매칭 쿼리에는 is_active 필터가 없다**.

```
영향:
  - Phase 4 이전: is_active 필드 자체가 없으므로 문제 없음
  - Phase 4 이후: 삭제된 Person/Chapter가 검색 결과에 포함될 수 있음
  - 운영 6개월 후 10% 삭제 시: 쿼리 결과에 비활성 데이터 혼입

필요 조치:
  - Q1~Q5, 매칭 쿼리 모두에 WHERE p.is_active <> false 추가
  - is_active에 인덱스 추가 (CREATE INDEX ON :Person(is_active))
  - 기존 Person 노드에 is_active = true 일괄 설정 (마이그레이션)
```

**권장**: Phase 4-4(증분 자동화) 구현 시 **Cypher 쿼리 마이그레이션 태스크**를 명시적으로 추가. "모든 쿼리에 필터 필수"라는 서술만으로는 실제 반영이 누락될 수 있다.

## 5. 실현 가능성 (Feasibility)

### 5.1 Phase 2 — 600K Batch 처리: 타이트해짐

| 항목 | v2 | v3 | 평가 |
|------|----|----|------|
| 볼륨 | 450K | **600K** | 33% 증가 |
| 총 호출 수 | 450K | **~1,140K** | **2.5배 증가** (적응형 호출) |
| 기간 | 8주 | **8주** (동일) | ★ 타이트 |
| 기본 시나리오 완료율 | 100% | **87%** | ↓ 악화 |
| 비관 시나리오 완료율 | 78% | **44%** | ↓↓ 심각 악화 |

**평가**: v2에서 "여유 12일"이 있던 기본 시나리오가 v3에서는 **여유 0일**로 변경되었다. chunk 크기 확대(1K→2K)로 부분 상쇄되지만, N+1 pass 건은 chunk 1K 유지이므로 혼합 관리가 필요하다.

**핵심 리스크**: 비관 시나리오에서 44%만 완료되면, Phase 3 7주 동안 잔여 56%(336K건)를 백그라운드 처리해야 한다. 이는 Phase 3의 Batch API 할당과 충돌할 수 있다.

**평가**: 실질적으로 가능하지만, 크롤링 분석까지 포함하면 **품질이 저하**될 수 있다. CMEK 이동(O1)으로 0.5일을 확보하는 것을 권장.

## 9. 결론

v3는 v2의 **9개 리뷰 사항과 v12 extraction logic을 빠짐없이 통합**한 성숙한 계획이다.

**강점:**
- v2 리뷰 N1~N9 전부 반영, HIGH 리스크 1건(AuraDB Auto-Pause) 해소
- v12 적응형 호출, Hybrid 섹션 분리, v19 관계명의 정합적 통합
- PII 보안 체계 대폭 강화 (CMEK + 전용 서비스 계정 + API 필터링)
- 비용 레인지 하한 크게 개선 ($8,215 → $5,523, Gold Label 2단계 효과)

**신규 발견 이슈:**
- 600K 볼륨 + 적응형 호출로 Phase 2 일정 여유 소진 (HIGH)
- PII 마스킹 코드 오프셋 버그 (HIGH — 코드 수정으로 즉시 해결)
- 비용 문서 간 $70~90 불일치 (MEDIUM)
- DETACH DELETE 4단계 과도 세분화 (MEDIUM)

**과도한 설계 7건**: 대부분 LOW~MEDIUM. v2 부족한 설계 보강 과정에서 유입된 것으로, 프로젝트 성패에 영향을 미치는 수준은 아님.

**부족한 설계 5건**: HIGH 2건은 즉시 반영 필요 (PII 코드 수정, Phase 2 일정 전략). MEDIUM 2건은 문서 정비 수준.

**최종 판단**: 전체 27주, $5,523~9,138 계획은 **2명 체제에서 실행 가능**하다. R1(PII 코드 수정)은 반드시 반영하고, R5/R6(Phase 2 일정)은 Phase 0 Go/No-Go 시점에서 실제 Career 수 분포를 확인한 후 결정할 것을 권장한다. **R1~R4 반영 후 실행 단계로 진입 가능**하다.
