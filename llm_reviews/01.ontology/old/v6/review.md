| 변경 감지 증분 전략 | SHA-256 해시 비교로 변경된 페이지만 재추출하여 LLM 비용 절감 |
| facet 병합 규칙 | 차이 <= 0.20이면 가중 평균 + 보강, > 0.20이면 JD 우선 + 충돌 기록 -- 보수적이고 합리적 |
| 데이터 플로우 다이어그램 | 6개 테이블 간 관계가 시각적으로 명확 |
| 카테고리별 ceiling | funding 0.65, performance 0.60 차등 적용이 팩트 기반 카테고리의 신뢰도를 적절히 반영 |

### 3.2 잔여 우려 사항

#### [C6-1] P4~P6 추출 프롬프트 미비 (중요도: 낮음)

P1~P3(필수)에 대해서는 프롬프트가 완비되었으나, P4(기술 블로그), P5(팀/문화), P6(고객 사례)는 여전히 추출 구조만 정의되어 있다. 이들은 "선택" 우선순위이므로 v1 파일럿에서 P1~P3 프롬프트 안정화 후 추가해도 무방하다.

**제안**: 파일럿 2차에서 P4~P6 프롬프트를 추가하되, P1~P3 프롬프트 안정화를 우선한다.

#### [C6-2] N2, N3, N5 추출 프롬프트 미비 (중요도: 낮음)

N1(투자), N4(조직변화)에 대해서는 상세 프롬프트가 제공되었으나, N2(제품), N3(M&A), N5(실적)는 추출 구조만 정의되어 있다. N1과 N4가 가장 중요한 카테고리이므로 현재 수준에서 파일럿은 가능하다.

**제안**: 파일럿에서 N2/N3/N5의 추출 품질을 확인하고, 필요 시 프롬프트를 추가.

#### [C6-3] facet 병합 규칙의 threshold 0.20 근거 (중요도: 낮음)

차이 <= 0.20을 "합의 범위"로 정의했으나, 이 임계값의 근거가 명시되지 않았다. 파일럿 데이터로 검증이 필요하다.

**제안**: 파일럿에서 JD vs 크롤링 facet score 차이 분포를 분석하여 합의/충돌 임계값을 조정.

#### [C6-4] 서브도메인 탐색의 DNS 확인 비용 (중요도: 낮음)

URL 발견 4단계에서 서브도메인 DNS 확인을 추가했는데, 기업당 6개 서브도메인 x 1,000기업 = 6,000 DNS 쿼리가 발생한다. 비용은 무시할 수 있지만, 크롤링 시간이 늘어날 수 있다.

**제안**: DNS 확인을 병렬화하거나, 1~3단계에서 필수 P1~P3가 모두 발견되면 서브도메인 탐색을 스킵.

## 4. v4 Amendments 리뷰 (`02_v4_amendments.md`)

### 4.1 Amendment별 리뷰

#### A1 (v6 수정) -- 양호

FOUNDER의 경력 10년 기준 HEAD 승격 규칙이 추가되어, "경력 연수에 따라 HEAD 가능"이라는 주석이 실제 로직으로 구현되었다. 10년 기준은 합리적이나, 파일럿에서 FOUNDER 케이스를 집중 검토할 필요가 있다.

#### A2 (v6 수정) -- 양호

is_regulated 판정 기준이 NICE 대분류 코드 기반으로 추가되었다. 4개 대분류(K, Q, D, H)는 명확한 규제 산업이므로 적절하다. 다만 리뷰에서 언급한 "세분류 수준 규제(핀테크 등)"에 대한 v2 보완 계획도 명시되어 있어 로드맵이 일관적이다.

#### A4 (v6 수정) -- 우수

3단계 캘리브레이션 계획(초기값 -> 1차 Human eval -> 2차 운영 데이터)이 추가되어 매트릭스 값의 근거와 개선 경로가 명확해졌다. v4 7.1절의 Human evaluation과 연계된 점이 좋다.

#### A6 (v6 수정) -- 우수

배타성 가이드(3개 모호 조합에 대한 판정 기준)가 추가되어 구현 시 모호함이 크게 줄었다. `related_tensions` 구조도 TypeScript 정의와 JSON 예시 모두에 반영되어 정합적이다.

#### A7 (v6 신규) -- 양호 (개선 1건)

v3 평가 이후 미해결이었던 GraphRAG ROI 검증을 드디어 실험 계획으로 구체화했다.

**[A7-1] Vector baseline의 구체성 부족 (중요도: 중간)**

"JD+이력서 임베딩 cosine similarity"라고만 기술되어 있는데, 구체적으로:
- 어떤 임베딩 모델을 사용하는지 (text-embedding-3-large? multilingual-e5?)
- JD와 이력서 전체를 임베딩하는지, 구조화된 필드별로 임베딩하는지
- cosine similarity만 사용하는지, reranking이 있는지

이 부분이 명확하지 않으면 공정한 비교가 어렵다.

**제안**: Vector baseline의 구체적 구성을 다음과 같이 명시:
```
Vector Baseline:
- 임베딩 모델: text-embedding-3-large (OpenAI) 또는 text-multilingual-embedding-002 (Vertex AI)
- 입력: JD 전문 + 이력서 전문 각각 임베딩
- 유사도: cosine similarity
- Top-K: K=10에서 reranking 없이 순위 산출
```

## 6. v6 신규 도입 항목 분석

### 6.1 해시 기반 변경 감지 (01, 2.3절)

**평가**: 양호. SHA-256 해시 비교로 변경 감지는 단순하고 효과적이다. HTML 전체를 해싱하므로 CSS/JS 변경만으로도 재추출이 트리거될 수 있으나, 과도한 재추출보다는 안전한 방향이다.

**개선 가능**: 텍스트 추출 후 텍스트를 해싱하면 레이아웃만 변경된 경우의 불필요한 재추출을 방지할 수 있다. 다만 v1에서는 현재 HTML 해시로 충분하다.

### 6.2 데이터 플로우 다이어그램 (01, 5.3절)

**평가**: 우수. 4단계 흐름(크롤링 -> 추출 테이블 -> 요약 테이블 -> CompanyContext -> MappingFeatures)이 명확하다. 집계 Job의 트리거 시점(크롤링 완료 후 자동?)만 추가하면 완전하다.

### 6.3 Graph Organization 확장 (01, 5.4절)

**평가**: 양호. structural_tensions를 v1에서 JSON 속성으로 유지하고 v2에서 노드 분리를 검토하는 결정은 적절하다. v1 MappingFeatures에서 직접 입력이 아니므로 노드 분리의 ROI가 낮다.

### 6.4 GraphRAG vs Vector 비교 실험 (02, A7절)

**평가**: 양호. 실험 설계가 구체적이고, 성공 기준(+0.5점)이 명확하다. 다만 Vector baseline의 구성이 더 구체화되어야 공정한 비교가 가능하다 (A7-1 참조).

## 8. 요약

v6 문서는 v5 리뷰의 **13건 피드백을 100% 반영**하여, 크롤링 전략과 v4 Amendments 모두 **파일럿 실행에 즉시 투입 가능한 수준**으로 완성했다.

**핵심 개선 사항**:
- P2/P3/N4 추출 프롬프트 완비로 필수 페이지/기사 유형의 추출 품질 기준선 확보
- 한국어 URL 패턴 + 서브도메인 탐색으로 국내 스타트업 크롤링 커버리지 향상
- 카테고리별 confidence ceiling 차등 적용으로 팩트 기반 뉴스의 신뢰도 반영 개선
- facet 병합 규칙 확정으로 JD+크롤링 통합의 모호함 해소
- GraphRAG vs Vector 비교 실험 계획으로 v3 이후 미해결 항목 최종 해소

**잔여 리스크**: 최소 수준. 파일럿 중 검증 5건과 운영 전 조치 3건은 모두 파일럿 데이터가 있어야 최적화할 수 있는 항목이므로, 현 시점에서 blocking이 아니다.

**종합 판정**: v6는 **파일럿 실행 승인 가능** 수준이다.
