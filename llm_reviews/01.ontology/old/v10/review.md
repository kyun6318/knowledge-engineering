### [N-2] 05_evaluation_strategy.md: 가상 데이터 SD와 power analysis 간의 내적 긴장 (심각도: 낮음)

**현황**:
- 10절 가상 데이터: Mean(A) = 3.82, SD = 0.94, Mean(B) = 3.34, SD = 1.12, p = 0.0062 (유의)
- 1.1절 power analysis: SD = 1.00일 때 power = 0.70 → "n=50 부족"

**문제**: 가상 데이터의 SD 범위(0.94~1.12)에서 power는 0.55~0.75로 0.80 미만이라고 1.1절에서 경고한다. 그런데 동일 가상 데이터의 통계 검정 결과는 p = 0.0062로 매우 유의하게 나온다. 이 두 결론이 동일 문서 안에 공존하면 읽는 사람이 혼란을 느낄 수 있다.

물론 "power가 낮아도 운이 좋으면 유의한 결과가 나올 수 있다"는 것은 통계적으로 맞지만, 가상 데이터가 이 미묘한 관계를 설명하지 않고 있어 오해의 여지가 있다.

**권장**: 10절 가상 데이터에 주석을 추가: "이 가상 결과는 SD 범위에서의 power < 0.80에도 불구하고 유의한 결과가 나온 경우를 예시한 것이다. 실제 실험에서는 유의하지 않은 결과가 나올 확률(Type II error)이 25~45%임을 1.1절에서 명시한다." 또는 v9 리뷰 L-2대로 가상 데이터의 수치를 마스킹.

### [N-3] 04_graph_schema.md: 크롤링 확장 속성의 타입 표기 불일치 (심각도: 매우 낮음)

**현황**: v10에서 추가된 Organization 노드의 크롤링 확장 속성은 `STRING | null` 형식으로 표기된다:

```
product_description: STRING | null,   -- 홈페이지 P2에서 추출
market_segment: STRING | null,        -- 홈페이지 P1+P2에서 추출
```

반면 기존 속성들은 타입만 표기하고 nullable 여부를 별도로 명시하지 않는다:

```
name: STRING,
industry_code: STRING,
stage_label: STRING,         -- "EARLY" / "GROWTH" / ...
```

**문제**: 기존 속성 중에도 실제로는 nullable인 것이 있을 수 있으나(예: `revenue_range`가 NICE에 없는 경우), 기존 속성은 nullable 표기를 하지 않는다. 신규 속성만 `| null`을 붙이면 "기존 속성은 모두 NOT NULL"이라는 암시를 준다.

**영향**: 미미. 실제 Neo4j 구현 시 개발자가 컨텍스트를 파악할 수 있다.

**권장**: 크롤링 확장 속성 블록 전체에 "// optional, v1에서 모두 null" 주석을 달고, 개별 속성의 `| null`은 제거하여 기존 표기 스타일과 통일. 또는 기존 속성에도 nullable 여부를 주석으로 추가. **비긴급**.

## 5. 잘된 부분 (4건)

### [P-1] 05_evaluation_strategy.md의 사전 검정력 분석 설계

v9 리뷰의 가장 중요한 후속 조치로 지목한 N-1이 매우 높은 품질로 반영되었다. 특히:

- **검정력 테이블**: SD 0.70~1.20 범위에서 n=50/70/100의 power를 명시하여 의사결정 근거를 선제적으로 제공
- **적응적 프로토콜**: "파일럿 50건 → SD 관측 → power 계산 → 필요 시 표본 확대"의 실무적 프로세스가 명확
- **N* > 100 대응**: 표본 확대가 실무적으로 어려운 경우 Cohen's d를 주 판단 기준으로 전환하는 현실적 대안 제시
- **6절 의사결정 트리와의 연동**: Case 1~4에서 Cohen's d와 p-value를 공동으로 사용하도록 설계되어 소표본 리스크를 흡수

### [P-2] 04_graph_schema.md의 Organization 확장 속성 설계

크롤링 보강 속성을 optional로 추가하면서 "v1 Neo4j 스키마 구현 시 nullable로 선언해 두면 v1.1 마이그레이션 없이 바로 사용 가능"이라는 실용적 가이드를 포함한 것이 좋다. 06_crawling_strategy.md 5.4절의 Cypher 확장 스키마와 정합되며, forward-compatible 설계다.

### [P-3] 효과 크기의 공동 주 판단 기준 격상

05 1절의 성공 기준에서 Cohen's d >= 0.5를 p < 0.05와 동등한 주 판단 기준으로 격상한 것은 소표본 실험 설계의 모범 사례다. "두 기준을 공동 적용하되, 소표본에서는 효과 크기를 우선 판단 기준으로 사용"이라는 명시적 가이드라인은 실험 결과 해석 시 Type II error 리스크를 줄여준다.

### [P-4] 누적 문서 간 정합성 유지

v3→v10까지 11차 반복에서도 문서 간 정합성이 유지되고 있다. 특히 v10에서 04와 06 사이의 Organization 확장 속성 불일치(v9 N-3)가 해결되어, 현재 알려진 문서 간 불일치가 없다(버전 넘버링 제외).

v10은 **v9 리뷰의 핵심 권장사항 2건을 모두 높은 품질로 반영**한 버전이다. 특히 05_evaluation_strategy.md의 사전 검정력 분석은 실험 설계의 통계적 견고성을 크게 강화하였고, 04_graph_schema.md의 Organization 확장 속성은 06과의 정합성 문제를 해결하면서 forward-compatible 설계를 유지했다.

잔여 이슈는 대부분 v9에서 비긴급으로 분류되었던 항목의 재기이며, 새로 발견된 이슈(N-1 버전 넘버링 악화, N-2 가상 데이터-power analysis 긴장, N-3 타입 표기 불일치)도 모두 파일럿 실행에 영향을 주지 않는 수준이다.

**가장 중요한 후속 조치**: [N-1] 버전 넘버링 체계 정리. 01/02/03의 헤더가 v8인 채로 v10 디렉토리에 존재하는 것은 혼란의 소지가 있으며, 문서를 외부에 전달하기 전에 정리하는 것이 바람직하다. 다만 이는 내용의 정확성과는 무관한 형식적 이슈이므로 파일럿 진행을 차단하지는 않는다.

**종합 판정**: v10은 파일럿 투입에 충분하며, v3부터 시작된 11차 반복 개선의 성숙 단계에 있다. 즉시 조치가 필요한 결함이 없다.
