## 3. 과도한 부분 (3건)

### [E-1] 06_crawling_strategy.md의 tension taxonomy 참조 경로 불일치 (심각도: 낮음)

**문제**: 06 문서 3.2절 N4 추출 프롬프트(~:623행)에서 tension_type taxonomy를 설명하며 `02_v4_amendments.md A6에서 확정한 8개 taxonomy enum`이라고 참조한다. 그러나 v8에서 A6는 "통합판 이관 완료"로 정리되었고, 정본은 `01_company_context.md`로 변경되었다.

**영향**: 참조 경로가 더 이상 유효하지 않다. 06에서 A6를 참조하여 taxonomy를 확인하려 하면 "정본은 통합판 참조"라는 안내만 보게 되어 한 단계 추가 점프가 필요하다.

**권장**: 06 문서의 A6 참조를 `01_company_context.md 2.2절`로 변경. 단, 06의 N4 프롬프트 내 taxonomy enum은 독립 실행을 위해 유지(현재 상태가 맞음).

### [E-2] amendments A7과 05_evaluation_strategy 간 내용 중복 (심각도: 낮음)

**문제**: `02_v4_amendments.md` A7(GraphRAG vs Vector 비교 실험 계획)은 여전히 전체 내용을 유지하고 있다(~130줄). 그러나 05_evaluation_strategy.md가 이 내용을 독립 문서로 확장한 것이므로, A1~A6과 동일한 논리로 "통합판 이관 완료" 처리가 가능하다.

**영향**: A7의 Vector Baseline 구성, 통제변수표 등이 05 문서와 중복 유지된다. A1~A6은 정리했으나 A7만 남아 있어 일관성이 깨진다.

**권장**: A7도 A1~A6과 동일하게 "[v8] 05_evaluation_strategy로 이관 완료" 처리. A8(추출 프롬프트 로드맵)은 06 문서에 해당 내용이 없으므로 현재대로 유지.

### [E-3] 05_evaluation_strategy 10절 가상 데이터의 잔여 리스크 (심각도: 매우 낮음)

**문제**: v8에서 경고 배너가 추가되어 E-3의 핵심 문제는 해결되었다. 다만 가상 데이터 내의 구체적 수치(p=0.0062, Cohen's d=0.48 등)가 그대로 유지되어, 빠르게 훑어보는 독자가 배너를 놓칠 수 있다.

**영향**: 매우 낮음. 배너가 존재하므로 주의 깊은 독자는 구분 가능. 다만 JSON 블록 내부에는 가상 데이터 표시가 없다.

**권장**: JSON 블록 내부에 `"_warning": "SIMULATED_DATA"` 필드를 추가하면 코드로 파싱하는 경우에도 안전. 또는 현재 상태 유지 후 실험 완료 시 실제 데이터로 교체.

## 4. 부족한 부분 (4건)

### [M-1] v7에서 이관된 파일럿 검증 항목 진행 계획 부재 (심각도: 낮음)

**문제**: v7 리뷰의 M-4(NEEDS_SIGNAL 자동 추론 검증), M-5(tech_stack 정규화 사전), M-6(operating_model 8 facets 후보), M-7(PastCompanyContext 시점 보정) 4건은 "파일럿 중 검토" 또는 "비긴급"으로 분류되어 v8에서 반영하지 않았다. 이 자체는 문제가 아니나, **어디에서도 이 항목들의 추적 상태가 기록되지 않았다**.

**영향**: 버전이 올라가면서 이 권장사항들이 "잊히는" 리스크. 현재 4건 모두 v7 리뷰 문서에만 존재하며, 작업 문서에는 반영되지 않았다.

**권장**: 다음 중 하나:
1. 해당 항목을 각 문서의 로드맵 섹션에 "파일럿 후 검토 예정" 항목으로 추가
2. 별도 "잔여 권장사항 추적" 문서에 목록 유지
3. 파일럿 실행 시점에 v7 리뷰를 재확인하는 체크리스트에 포함

### [M-2] 05_evaluation_strategy의 평가자 간 일치도 목표 미정의 (심각도: 낮음)

**문제**: 05 문서 7절 평가 지표 상세에서 Krippendorff's alpha의 "양호" 기준을 0.6, "우수" 기준을 0.8로 정의했다. 그러나 10절 가상 데이터에서는 `inter_rater_agreement: 0.72`로 예시를 제시하면서, **alpha가 0.6 미만인 경우의 대응 방안**이 정의되지 않았다.

**영향**: 실험 수행 시 평가자 간 일치도가 낮으면 결과의 신뢰성이 저하된다. 대응 방안이 없으면 실험을 반복해야 할지, 평가 기준을 수정해야 할지 판단이 지연된다.

**권장**: 6절 의사결정 트리에 alpha < 0.6인 경우의 분기를 추가:
- alpha 0.4~0.6: 평가 기준 재교육 후 재평가 (10건 샘플)
- alpha < 0.4: 평가 기준 자체를 재설계

### [M-3] 02_candidate_context TypeScript 정의에 domain_depth evidence 미반영 (심각도: 매우 낮음)

**문제**: v8에서 domain_depth에 evidence 필드를 추가했고, 3절 JSON 스키마 예시에는 올바르게 반영되었다. 그러나 2절의 TypeScript interface 정의 영역(Experience, Outcome 등)에서 `domain_depth`의 TypeScript interface가 **별도로 정의되지 않았다**. JSON 스키마 예시에서만 evidence 필드가 보인다.

**영향**: 매우 낮음. domain_depth는 JSON 스키마에서 구조가 명확하고, TypeScript interface가 누락된 것이 아니라 원래 domain_depth의 TypeScript 정의가 문서에 없었다(v7에서도 마찬가지).

**권장**: 필수는 아니나, 다른 구조체(WorkStyleSignals, RoleEvolution 등)와의 일관성을 위해 DomainDepth interface를 추가하면 좋다:
```typescript
interface DomainDepth {
  primary_domain: string;
  domain_experience_count: number;
  description: string;
  confidence: number;
  evidence: Evidence[];
}
```

### [M-4] 05_evaluation_strategy의 처리량 추정 불일치 (심각도: 매우 낮음)

**문제**: 05 문서 2.1절에서 Vector Baseline의 처리량을 "50건 × 후보풀~500명 = ~25k 쿼리, ~2시간"으로 추정한다. 그러나 4.1절 Step 2에서는 "시간: ~2초 (배치 처리)"라고 표기한다. 25k 쿼리를 2초에 처리한다는 것은 ANN 인덱스의 배치 쿼리를 전제한 것이지만, 임베딩 생성 시간(500명 이력서 × Vertex AI API)은 포함하지 않은 수치이다.

**영향**: 매우 낮음. 실험 일정에 영향을 주지 않을 정도의 차이이나, 2.1절과 4.1절의 수치가 혼동을 줄 수 있다.

**권장**: 4.1절 Step 2의 시간을 "~2시간 (임베딩 생성 포함)" 또는 "임베딩 사전 생성 완료 시 ~2초 (ANN 검색만)"로 조건부 표기.

5. **임베딩 모델 통일** (M-1): 04/05 양쪽 모두 `text-multilingual-embedding-002`로 확정
6. **설계 의도 명문화** (M-2/M-3): confidence 이중 감쇠의 의도적 설계 근거, domain_depth의 Evidence-first 원칙 준수
