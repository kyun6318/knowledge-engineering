### C1. Seniority enum 3계층 불일치 — F5 매칭 실패 가능

**위치**:
- `01.ontology/v23/01_company_context.md` L125: `JUNIOR / MID / SENIOR / LEAD / HEAD / UNKNOWN`
- `01.ontology/v23/03_mapping_features.md` L506-507: `SENIORITY_ORDER = {"JUNIOR": 1, "MID": 2, "SENIOR": 3, "LEAD": 4, "HEAD": 5}`
- `02.knowledge_graph/v16/03_prompt_design.md` L105: `Literal["JUNIOR", "SENIOR", "LEAD", "HEAD", "C_LEVEL"]`

**문제**: 3곳에서 seniority 열거값이 서로 다르다.

| 값 | 01.ontology (정본) | 03_mapping_features F5 | 02.knowledge_graph 추출 |
|-----|-------------------|----------------------|----------------------|
| JUNIOR | ✅ | ✅ | ✅ |
| **MID** | ✅ | ✅ | **❌ 누락** |
| SENIOR | ✅ | ✅ | ✅ |
| LEAD | ✅ | ✅ | ✅ |
| HEAD | ✅ | ✅ | ✅ |
| **C_LEVEL** | **❌** | **❌** | ✅ |
| UNKNOWN | ✅ | ❌ (SENIORITY_ORDER에 없음) | ❌ |

**영향**:
1. LLM이 `C_LEVEL`을 추출하면 F5의 `SENIORITY_ORDER.get("C_LEVEL", 0)`에서 0으로 fallback → JUNIOR보다 낮은 seniority로 잘못 평가
2. LLM이 `MID`를 추출할 수 없으므로(enum에 없음), 경력 4~6년의 개인 기여자 공고에서 정확한 seniority 추출 불가
3. `02_candidate_context.md` L141-163의 `ic_to_seniority()`는 MID를 반환하지만, 이 값이 추출 스키마에서 생성될 수 없음

**권고**:
1. 정본(01.ontology)의 6개 값을 기준으로 통일: `JUNIOR / MID / SENIOR / LEAD / HEAD / UNKNOWN`
2. `02.knowledge_graph/v16/03_prompt_design.md`의 `seniority` 필드에 `MID` 추가, `C_LEVEL` → `HEAD`로 변경 (또는 정본에 C_LEVEL 추가 후 전체 반영)
3. F5 `SENIORITY_ORDER`에 `UNKNOWN` 처리 로직 추가 (현재는 `.get(, 0)`으로 묵시적 처리)

**위치**:

```python
    0.70,  # role matching은 비교적 판단 가능
```

**문제**: Confidence 전파 규칙 테이블은 "양쪽 입력 소스의 최소값"을 명시하지만, 실제 코드는 seniority_source_confidence 대신 **0.70 하드코딩 상한**을 사용한다. 이로 인해:

**권고**: 둘 중 하나로 통일:
- (A) 코드를 테이블에 맞춤: `confidence = min(company_ctx.vacancy.seniority_confidence, role_evo.confidence)`
- (B) 테이블을 코드에 맞춤: `F5 = min(0.70, role_evolution_confidence)` — "role matching 판단 가능도의 이론적 상한 0.70과 role_evolution confidence의 최소값"

### H2. F4 ALIGNMENT_LOGIC 매핑 테이블 불완전 — 대부분 기본값 fallback

**위치**: `01.ontology/v23/03_mapping_features.md` L429-437

**문제**: `ALIGNMENT_LOGIC` 딕셔너리에 정의된 키가 4개뿐이다:
- `("speed_high", "autonomy_HIGH")`: 0.9
- `("speed_high", "autonomy_MID")`: 0.6
- `("speed_high", "autonomy_LOW")`: 0.3
- `("process_high", "process_tolerance_HIGH")`: 0.9 / MID / LOW

누락된 조합:
- `speed_mid`, `speed_low` × 3가지 autonomy 수준 = 6개
- `autonomy_high/mid/low` × 3가지 autonomy_preference 수준 = 9개
- `process_mid`, `process_low` × 3가지 process_tolerance 수준 = 6개

총 27개 조합 중 **6개만 정의**, 나머지 21개는 기본값 0.5로 fallback. F4가 v1에서 대부분 INACTIVE이므로 당장 영향은 적으나, v2에서 활성화 시 변별력이 사실상 없다.

**권고**: `# ...` 주석으로 생략 표기된 부분을 "v2 활성화 전까지 전체 매트릭스 정의 보류, Phase 5에서 완성" 등 명시적 상태 표기로 변경. 또는 최소한 `speed_low`, `autonomy_high` 등 핵심 조합의 값을 정의.

### H5. 04_graphrag_g3_matching F2 간소화 구현의 논리 오류

**위치**: `03.graphrag/separate/v6/graphrag/04_graphrag_g3_matching.md` L72-75

```python
# F2: vacancy_fit (scope_type + signals)
scope_match = 1.0 if candidate.get("scope_type") == vacancy.get("hiring_context_scope") else 0.5
signal_overlap = jaccard_similarity(
    set(candidate.get("signals", [])), set(vacancy.get("needed_signals", [])))
```
