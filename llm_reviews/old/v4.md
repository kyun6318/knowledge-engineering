**위치**: `01.ontology/v22/03_mapping_features.md` L148-151

```python
confidence = min(

    best_match_pcc_confidence   # ← 선언/할당 없는 변수
```

**문제**: 구현 시 `NameError`로 F1 전체가 실패한다. F1은 가중치 0.25로 전체 매칭 점수의 핵심 피처이다.

**권고**: `stage_experiences` 구성 시 PCC confidence를 함께 수집하고, `best_match` 선택 후 해당 confidence를 추출:

### C2. SituationalSignal 열거값 불일치 — Data Contract와 Ontology 간 2건 오류

**위치**:
- `01.ontology/v22/02_candidate_context.md` §2.3: 14개 고정 taxonomy 정의
- `03.graphrag/separate/v5/interface/00_data_contract.md` L61, L86
- `03.graphrag/separate/v5/sf/04_sf_phase3_jd_company.md` L25

Data Contract 내 JSON 예시:

```json
"situational_signals": [{"label": "SCALING_TEAM", ...}]   // L61
"needed_signals": ["SCALING_TEAM", "GREENFIELD"]           // L86
```

(1) `SCALING_TEAM`은 ontology 정본에 존재하지 않는다. 정본은 `TEAM_SCALING`이다.
(2) `GREENFIELD`는 14개 taxonomy 어디에도 없다. 유사 라벨은 `NEW_SYSTEM_BUILD` 또는 `EARLY_STAGE`이다.

이 오류는 v1부터 v5까지 **모든 버전의 data contract에 동일하게 존재**한다.

**문제**: GraphRAG가 이 JSON을 수신하여 SituationalSignal 노드와 매칭할 때, 존재하지 않는 라벨이므로 `NEEDS_SIGNAL`/`HAS_SIGNAL` 관계가 생성 불가하다. F2 vacancy_fit이 의도대로 작동하지 않는다.

**권고**:
1. `SCALING_TEAM` → `TEAM_SCALING`으로 전체 수정 (v5/interface, v5/sf 모든 파일)
2. `GREENFIELD` → `NEW_SYSTEM_BUILD` 또는 적절한 정본 라벨로 교체
3. Data Contract에 유효 열거값 목록을 명시: `"valid_signal_labels": ["EARLY_STAGE", "SCALE_UP", ..., "OTHER"]`

### C3. operating_model 타입 불일치 — float(0.0~1.0) vs string enum

**위치**:
- `02.knowledge_graph/v15/03_prompt_design.md` L70-73: Pydantic 스키마
- `03.graphrag/separate/v5/interface/00_data_contract.md` L88: JSON 예시

v15 Pydantic 스키마:

```python
class OperatingModelFacets(BaseModel):
    speed: Optional[float] = Field(None, ge=0.0, le=1.0)
    autonomy: Optional[float] = Field(None, ge=0.0, le=1.0)
    process: Optional[float] = Field(None, ge=0.0, le=1.0)
```

v5 Data Contract:

```json
"operating_model": {"speed": "FAST", "autonomy": "HIGH", "process": "AGILE"}
```

v15는 **연속형 float**를 정의하고, v5는 **이산형 string enum**을 사용한다. 변환 로직이 문서화되어 있지 않다.

**문제**: S&F가 float로 추출한 operating_model을 GraphRAG에 전달할 때, JSON Schema 검증이 실패하거나 타입 불일치로 F4 culture_fit 계산이 불가하다. (v1에서 F4는 INACTIVE이지만, 데이터는 적재되므로 차후 활성화 시 데이터 마이그레이션 필요)

**권고**: 정본을 확정하고 다른 쪽을 수정:
- **Option A**: 정본 float 유지 → Data Contract JSON도 `"speed": 0.8`로 수정
- **Option B**: 정본 enum 전환 → Pydantic 스키마를 `Literal["HIGH", "MID", "LOW"]`로 수정
- 어느 쪽이든 변환 함수를 문서화

### H6. F2 "no matching signals" 시 ACTIVE 반환 — 논리 불일치

**위치**: `01.ontology/v22/03_mapping_features.md` L251-253

```python
else:
    return FeatureResult(score=0.15, confidence=0.40,
                         status="ACTIVE", reason="no matching signals")
```

후보의 signal이 vacancy 요구 signal과 하나도 매칭되지 않을 때 `ACTIVE(score=0.15)`를 반환한다. 그러나:

- `03.graphrag/separate/v5/interface/00_data_contract.md` L70: `chapters[] period_start 오름차순 정렬`

F5 role_fit은 `experiences[0]`을 **최신(latest)** 경력으로 가정한다. 그러나 Data Contract은 chapters를 `period_start 오름차순`(oldest first)으로 정렬한다.

**문제**: Data Contract 정렬 규약대로면 `experiences[0]`은 **가장 오래된** 경력이다. F5가 이를 latest로 사용하면 잘못된 seniority 비교가 발생한다.

**권고**: 둘 중 하나로 통일:
- **A**: Data Contract을 `period_start 내림차순`(latest first)으로 변경
- **B**: F5 코드를 `experiences[-1]` 또는 `max(experiences, key=lambda e: e.period_end)`로 수정

## 전체 아키텍처 관찰

### 강점

1. **v3 리뷰 반영 완성도가 높다**: 22/23건 반영으로 CRITICAL 항목(규모 듀얼 시나리오, 비용 틸드, SituationalSignal 공유 노드)이 모두 해결됨
2. **3-Layer 분리 원칙이 잘 준수된다**: 각 레이어의 관심사가 명확하게 분리되어 있고, 경계 침범이 거의 없음
3. **Phase 0 파일럿 계획이 구체적이다**: 50건 매핑, precision/recall 목표, 캘리브레이션 계획이 명시되어 있어 data-driven 의사결정이 가능

### 구조적 우려

1. **Data Contract(v5/interface/00_data_contract.md)이 Layer 1/2의 정본과 동기화되지 않고 있다**: CRITICAL 2건(C2, C3)과 HIGH 3건(H2, H3, H5)이 모두 Data Contract ↔ 정본 간 불일치. Data Contract은 구현 팀 간 인터페이스의 핵심이므로, 정본 변경 시 자동/반자동 검증 메커니즘이 필요하다.

2. **F1 `best_match_pcc_confidence` 미정의가 v7~v22까지 15개 버전에 걸쳐 잔존한다**: 이는 코드 블록이 실제 실행 검증 없이 문서로만 유지되고 있음을 의미한다. Phase 0 전에 **모든 코드 블록의 syntax/reference validation**을 수행할 것을 강력히 권고한다.

3. **Layer 2 → Layer 3 전달 과정의 필드 매핑이 암묵적이다**: `stage_estimate` → `stage_label`, `embedding` → `evidence_chunk_embedding`, `experiences[0]` 정렬 규약 등. **Layer 2→3 필드 매핑 테이블**을 Data Contract에 부록으로 추가하면 구현 오류를 크게 줄일 수 있다.
