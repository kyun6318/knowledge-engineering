## HIGH

### H1. candidate_id vs person_id 접두사 불일치

- **위치**: `01.ontology/v24/02_candidate_context.md` L419, L442
- **현상**:
  - L419: `"candidate_id": "cand_99999"`
  - L442: `"experience_id": "P_99999_ch0"`
- **문제**: candidate_id는 `cand_` 접두사, experience_id는 `P_` 접두사. `{person_id}_ch{idx}` 규약(L112)에 따르면 person_id = `P_99999`여야 하나, 상위 JSON의 candidate_id는 `cand_99999`. Graph Schema(04_graph_schema.md L14)는 `person_id` 사용
- **영향**: S&F 파이프라인이 candidate_id를 출력하면 GraphRAG의 Person 노드 person_id와 조인 불가
- **권장**: `"candidate_id": "cand_99999"` → `"person_id": "P_99999"`로 통일. 또는 명시적 매핑 규약 문서화 (`candidate_id = person_id`)

### H2. F5 코드 주석 "SENIOR_IC" 비존재 enum 참조

- **위치**: `01.ontology/v24/03_mapping_features.md` L578
- **현상**: `# scope_type을 seniority로 변환: IC/SENIOR_IC/LEAD/HEAD/FOUNDER 매핑`
- **문제**: ScopeType enum은 `IC/LEAD/HEAD/FOUNDER/UNKNOWN` (02_candidate_context.md L136). `SENIOR_IC`는 어디에도 정의되지 않음
- **권장**: 주석을 `# scope_type을 seniority로 변환: IC/LEAD/HEAD/FOUNDER/UNKNOWN 매핑`으로 수정

### H3. SituationalSignal evidence 단수/복수 불일치

- **위치**: `01.ontology/v24/02_candidate_context.md` L240
- **현상**: `evidence: Evidence;` (단수)
- **문제**: v5 L2에서 Outcome의 evidence를 `Evidence[]` (복수)로 통일했으나, SituationalSignal의 evidence는 여전히 단수. 동일 패턴의 데이터 구조에서 일관성 부재
- **정본 비교**: Outcome(L219): `evidence: Evidence[]` ✓, SituationalSignal(L240): `evidence: Evidence` ✗
- **권장**: `evidence: Evidence` → `evidence: Evidence[]`로 변경

### H4. OutcomeType "OTHER" Graph Schema 누락

- **위치**: `01.ontology/v24/04_graph_schema.md` L129
- **현상**: `outcome_type: STRING, -- "METRIC" / "SCALE" / "DELIVERY" / "ORGANIZATIONAL"`
- **문제**: 02_candidate_context.md L222~227은 `"OTHER"` 포함 5개값 정의. Graph Schema는 4개만 나열
- **권장**: L129에 `/ "OTHER"` 추가

### H5. org_stage vs stage_label 필드명 불일치

- **위치**: `03.graphrag/separate/v7/interface/00_data_contract.md` L80 vs `01.ontology/v24/01_company_context.md`, `04_graph_schema.md` L52
- **현상**: Data Contract는 `"org_stage": "MATURE"`, 정본과 Neo4j 스키마는 `stage_label`
- **문제**: S&F가 `org_stage` 필드로 출력하면 GraphRAG Neo4j 적재 시 `stage_label`로의 필드 매핑이 필요하나 매핑 규칙 미정의
- **권장**: Data Contract의 `org_stage` → `stage_label`로 통일. 또는 명시적 필드 매핑 테이블 추가

### H6. compute_match_score()에 freshness_weight 미적용

- **위치**: `03.graphrag/separate/v7/graphrag/04_graphrag_g3_matching.md` L92~95
- **현상**: `overall` 반환만 있고, `ranking_score = overall × freshness_weight` 계산 없음
- **문제**: `08_serving.md`에서 `ranking_score FLOAT64` 컬럼(H3)을 추가했으나, 실제 스코어링 코드는 overall_match_score만 산출. freshness_weight 곱셈과 ranking_score 산출 로직이 어디에도 구현되어 있지 않음
- **정본 근거**: `01.ontology/v24/03_mapping_features.md` §4에서 `ranking_score = overall_match_score × freshness_weight` 정의
- **권장**: compute_match_score 또는 별도 함수에서 freshness_weight 적용 로직 추가. 또는 "v1 MVP 간소화: freshness_weight=1.0 고정" 명시

### H7. F4 culture_fit INACTIVE인데 가중치 재분배 미반영

- **위치**: `03.graphrag/separate/v7/graphrag/04_graphrag_g3_matching.md` L84~85, L92~94
- **현상**: F4 = 0.5 고정, 가중치 0.10이 항상 overall에 포함
- **문제**: `01.ontology/v24/03_mapping_features.md`의 INACTIVE 정책은 "해당 가중치를 나머지 활성 피처에 재분배"이나, 간소화 구현은 0.5 × 0.10 = 0.05를 항상 가산. 이는 모든 후보에게 동일 보너스로 순위에 영향 없지만, 정본의 INACTIVE→재분배 정책과 불일치
- **권장**: "v1 MVP: F4 INACTIVE이나 기본값 0.5로 포함, Phase 2에서 가중치 재분배 적용" 주석 추가. 또는 실제로 0.10을 나머지 4개 피처에 재분배하는 코드로 변경

### H8. scope_type과 vacancy_seniority 직접 비교 (다른 enum 공간)

- **위치**: `03.graphrag/separate/v7/graphrag/04_graphrag_g3_matching.md` L73
- **현상**: `scope_match = 1.0 if candidate.get("scope_type") == vacancy.get("vacancy_seniority") else 0.5`
- **문제**: v7 주석(L54)에서 이 문제를 명시했지만, 코드 자체는 미수정. scope_type (IC/LEAD/HEAD/FOUNDER/UNKNOWN)과 vacancy_seniority (JUNIOR~HEAD/UNKNOWN)는 LEAD/HEAD에서만 우연히 일치. C3 수정 시 vacancy_seniority → seniority로 변경하면 이 필드명도 함께 갱신 필요
- **권장**: scope_type → seniority 변환 함수를 적용하거나, "v1 MVP 한계: signal_overlap이 F2의 실질적 변별력" 주석을 코드 바로 위에 추가
