## 2. 실현 가능성 평가

### 2.1 [양호] LLM 출력 파싱 실패 전략 (02 §8.3)

**긍정적**:
- 3-tier retry 전략이 실무적으로 합리적 (json-repair → temperature 조정 → skip)
- Pydantic 부분 추출 허용 규칙이 스키마별로 구체적 (필수 vs optional 필드 구분)
- 부분 추출 시 confidence 0.3 하향은 MappingFeatures 품질 보호에 효과적
- Grafana 알림 임계값(5% WARNING, 10% CRITICAL)이 합리적
- `LLMOutputParser` 클래스가 바로 개발 착수 가능한 수준

**경미한 이슈**:
- Tier 2에서 temperature 0으로 재시도하는데, 기본 호출이 이미 temperature 0이면 효과 없음 → 기본 temperature 설정을 02 문서 설계 원칙에 명시하면 좋음 (후술 R-1)

### 2.2 [양호] 오케스트레이션 전략 (04 §0-3.1)

**긍정적**:
- Pipeline DAG가 명확 (A∥B → C → D → E)
- A/B 병렬 실행 시 3주 단축 가능이라는 정량적 이점 제시
- Prefect vs Cloud Workflows 비교표가 실용적
- Chunk 관리 전략의 상태 추적(PENDING→PROCESSING→COMPLETED/FAILED)이 구체적

**경미한 이슈**:
- 부분 적재 전략("A 완료 후 Organization/Vacancy 먼저 적재")은 실질적 이점이 제한적 — Pipeline D에서 양쪽 노드가 모두 필요하므로 최종 결과 시점은 동일
- chunk 크기 1,000건의 근거가 "Batch API 요청 크기보다 작게"인데, Phase 0에서 처리 시간과 메모리 기반으로 최적값을 결정하는 것이 나음 (후술 R-2)

### 2.3 [양호] 타임라인 (04)

**긍정적**:
- 18~22주로 현실화, Phase 1-3(+1주), Phase 1-5(+1주) 변경 근거 구체적
- Pre-Phase 0 미확보 시 +2~4주 지연 가능성 언급

**경미한 이슈**:
- DE+MLE 2명 풀타임 전제이나, Phase 0의 도메인 전문가 파트타임 투입이 Gold Label 검수 외에 taxonomy 검증도 포함되어 실제 필요 시간이 명시보다 클 수 있음. 다만 Phase 0-2 PoC(50건)에서의 taxonomy 검증은 2-3일이면 충분하므로 일정 영향은 미미

### 2.4 [양호] 비용 모델 (03)

**긍정적**:
- 시나리오 A/A'/B/C/D로 다양한 상황을 커버
- Haiku→Sonnet 전환의 구체적 기준(scope_type < 60%, outcomes F1 < 50%) 제시
- A'와 B의 차이(의사결정 경로)를 명확히 구분

**경미한 이슈**:
- 03 문서의 시나리오 A LLM 소계 $1,211에 파싱 실패 retry 비용이 미포함. 02 §8.3의 Tier 2 retry는 LLM 재호출이므로 추가 비용 발생. 다만 최악 $35~175 수준으로 전체 비용의 2% 미만 (후술 R-3)

### 2.5 [양호] Pre-Phase 0 + NICE blocking dependency (04, 05)

**긍정적**:
- NICE DB 접근을 blocking dependency로 명시하여 일정 리스크를 가시화
- fallback(DART, 사업자등록 조회) 제시
- 04 문서와 05 문서에서 일관되게 참조

## 3. 과설계 평가

### O-1. [Low] Phase 3 크롤링 파이프라인 상세도

04 문서의 Phase 3 크롤링 파이프라인(C1~C4, 7주)은 v6에서 v10 정합을 위해 추가된 것이나, MVP(Phase 0~2)와 직접 관련이 없는 향후 계획이다. 4개 하위 Phase, 페이지 우선순위, 뉴스 카테고리 코드 등의 상세도가 높다.

**판단**: v10 06_crawling_strategy.md의 참조 반영이므로 수용 가능. 실행 시 세부사항은 변경될 수 있음을 인지하면 됨. **수정 불필요**.

### O-2. [Low] structural_tensions 조기 구현 상세 (02 §1.1, §2.5)

Pydantic 스키마, 프롬프트, 추출 함수까지 상세히 정의되어 있으나, v1에서는 `null`로 처리되고 Phase 3 크롤링 데이터 확보 후에야 활성화된다.

**판단**: 문서에 존재해도 해가 없고, Phase 3 진입 시 설계 시간을 절약할 수 있음. **수정 불필요**.

### O-3. [Low] facet merge 로직 (02 §2.4)

`merge_facets_with_crawling()` 함수가 구현 수준으로 정의되어 있으나, Phase 3 크롤링 데이터가 있을 때만 활성화되는 코드이다.

**판단**: O-2와 동일 — Phase 3 설계 예비 작업으로 수용 가능. **수정 불필요**.

## 4. 부족 설계 평가

### R-1. [Low] LLM 호출 시 기본 temperature 미명시

v7 §8.3의 Tier 2 retry에서 "temperature를 0으로 낮춰"라고 했지만, 기본 LLM 호출의 temperature가 어디에도 명시되어 있지 않다. 구조화 추출에는 temperature 0~0.2가 권장된다.

**영향도**: Low — 구현 시 자연스럽게 결정되는 사항. 다만 기본값이 0이면 Tier 2의 "temperature 0 재시도"가 무의미해지므로, 기본값을 0.1~0.2로 명시하면 Tier 2 전략이 더 효과적이 됨.

### R-2. [Low] chunk 크기 최적화 시점 미명시

§0-3.1에서 chunk 크기 1,000건을 고정했으나, Phase 0-4 의사결정 테이블에 "chunk 크기 최적화" 항목이 없다.

**영향도**: Low — 1,000건은 합리적 기본값이고, 운영 중 조정 가능.

### R-3. [Low] 파싱 실패율과 비용 모델의 연결 부재

§8.3에서 Tier 2 retry는 LLM 재호출이므로 추가 비용이 발생하지만, 03 문서의 비용 모델에 이 추가 비용이 반영되지 않았다.

**영향도**: Low — 최악의 경우 ~$35~175 증가 (150만 호출 × 2~10% × Haiku Batch $0.00115/건). 전체 비용의 2% 미만.

### R-4. [Low] overall_match_score INACTIVE 피처 가중치 재분배 로직 미완성

02 §5.3의 `compute_overall_score()`에서 INACTIVE 피처의 가중치 재분배 부분이 `pass`로 비어있다. culture_fit이 대부분 INACTIVE(10-30%)이므로 이 로직이 없으면 overall_score가 체계적으로 낮게 산출된다.

**영향도**: Low — 구현 시 자연스럽게 완성되는 코드이고, v10 mapping_features.md에 재분배 로직이 정의되어 있을 것으로 예상. 다만 계획 문서 수준에서는 재분배 방식(균등 분배 vs 비례 분배)을 한 줄이라도 명시하면 좋음.
