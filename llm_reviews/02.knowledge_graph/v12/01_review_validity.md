## 1. 적응형 LLM 호출 전략 (M1)

### 1.1 설계 평가: 합리적

Career 수 기반 1-pass(1~3)/N+1 pass(4+) 분기는 **기술적으로 타당**하다.

**근거**:
- Career 1~3이 전체의 80%를 차지한다는 가정이 맞다면, 1-pass로 대부분을 효율적으로 처리
- Career 4+에서 context window 내 정보 밀도가 떨어지는 것은 LLM 추출의 알려진 한계
- N+1 구조(개별 추출 N회 + 전체 요약 1회)는 정보 분리와 통합의 균형이 적절

**우려**:
| 항목 | 내용 | 심각도 |
|------|------|--------|
| Career 3개 분기점의 근거 | "80%+ 해당"이라는 추정만 존재, DB 프로파일링 전 | Medium |
| N+1 pass의 컨텍스트 단절 | Career별 개별 호출 시 "이전 회사에서 이 경험을 바탕으로..."와 같은 교차 참조 정보 상실 | High |
| max_tokens 2,048의 적정성 | Career 3개 × chapters 출력 시 실제 토큰 소모량 미검증 | Low |
| 비용 +25% 정당성 | 정확도 향상 "기대"만 있고 정량적 근거 없음 | Medium |

### 1.2 N+1 pass 컨텍스트 단절 상세

v12의 N+1 pass에서 Career별 호출(Pass 1~N)은 해당 Career의 workDetails + CareerDescription만 입력한다. 이때 다음 패턴이 누락된다:

```
누락 패턴 1: 교차 참조
"전 직장(A사)에서의 경험을 바탕으로 B사에서 시스템을 설계"
→ A사 Career 호출 시 B사 성과를 모르고, B사 호출 시 A사 배경을 모름

누락 패턴 2: 암묵적 성장
"처음에는 주니어로 시작했지만" (이전 Career에서의 역할이 힌트)
→ 개별 Career만으로는 scope_type 판단이 부정확해질 수 있음

누락 패턴 3: 중복 시그널
A사와 B사 모두 SCALE_UP 시그널 → 이것이 동일 사건인지 별개인지 판단 불가
```

**완화 제안**: Pass 1~N에 전체 Career 요약(company + period + role)을 컨텍스트로 추가 주입 (~200 토큰 추가). 현재 Pass N+1에서 이 정보를 사용하고 있으므로, Pass 1~N에도 동일 형태로 제공 가능.

### 1.3 결론

적응형 호출 전략 자체는 **타당(Valid)**하나, N+1 pass의 컨텍스트 단절은 **Phase 0 PoC에서 반드시 검증**해야 할 품질 리스크. Career 3/4 분기점도 실측 데이터 기반으로 조정 필요.

## 2. 파일 섹션 분리 Hybrid 전략 (S1)

### 2.1 설계 평가: 현실적이며 적절

패턴 기반(Step 1) → LLM 폴백(Step 2)의 2단계 접근은 한국 이력서의 극심한 형식 다양성을 고려할 때 **최선의 전략**이다.

**강점**:
- 비용 $0인 패턴 기반을 우선 적용하여 비용 효율적
- LLM 폴백으로 비정형 이력서도 처리 가능
- 최종 폴백(전체→단일 Career)으로 완전 실패 방지

**우려**:
| 항목 | 내용 | 심각도 |
|------|------|--------|
| 패턴 성공률 70% 근거 부재 | 실측 없이 추정한 수치 | High |
| SECTION_PATTERNS 한국어 커버리지 | "경력사항", "경력 사항", "경 력" 등 띄어쓰기 변형 미고려 | Medium |
| 잡코리아/사람인 양식 탐지 | CAREER_SEPARATOR_PATTERNS에 "테이블 형식" 패턴이 있으나, HTML 테이블 구조와 plain text 변환 후의 차이 미고려 | Medium |
| LLM 폴백 비용 $0.002/건 적정성 | LLM에 전체 이력서 텍스트를 전달하면 토큰이 $0.002를 초과할 수 있음 (긴 이력서) | Low |

### 2.2 패턴 누락 가능성

v12의 SECTION_PATTERNS는 일반적인 헤더 패턴을 잘 다루지만:

```
미탐지 가능 패턴:
- "[ 경력 ]" (대괄호 감싼 형식)
- "■ 경력사항" (특수문자 prefix)
- "1. 경력" (번호 매기기 형식)
- "CAREER HISTORY" (덜 흔한 영문 변형)
- 아예 섹션 헤더 없이 시간순 나열 (프리폼)
```

패턴 기반 70% 성공률은 **낙관적**일 수 있으나, Phase 2-0 PoC에서 20건으로 검증하겠다고 명시했으므로 조기 발견 가능.

### 2.3 confidence 상한 차등의 타당성

| 소스 | v12 상한 | 평가 |
|------|---------|------|
| DB 소스 | 0.85 | 적절 — 구조화 데이터 신뢰도 |
| 파일 (패턴 분리) | 0.75 | 적절 — 패턴 매칭 불확실성 반영 |
| 파일 (LLM 분리) | 0.70 | 적절 — LLM 분리 오류 가능성 추가 반영 |
| 파일 (분리 실패→단일 Career) | 0.30 | **약간 높음** — 전체 텍스트를 단일 Career로 처리하면 scope_type 등이 사실상 무의미, 0.20이 더 적절할 수 있음 |

### 2.4 결론

Hybrid 전략은 **타당(Valid)**. 패턴 성공률 70%는 PoC 검증 필요하나, 최악의 경우 LLM 폴백 비율이 50%가 되어도 비용 추가는 ~$40 수준으로 관리 가능.

## 3. operating_model 진정성 체크 단순화 (C3)

### 3.1 설계 평가: 올바른 방향

v11의 "LLM이 광고성 vs 실제를 판단"이라는 비현실적 요구를 **구체적 맥락 유무**라는 명확한 규칙으로 대체한 것은 **매우 적절**하다.

```
v11: "애자일 팀" → LLM이 진정성 판단 (불가능)
v12: "애자일 팀" (구체적 맥락 없음) → speed = null
     "2주 스프린트로 빠르게 배포" (구체적 맥락 있음) → speed = 0.8
```

**유일한 우려**: `min(count / 5, 1.0)` 공식에서 count 기준이 키워드 수인데, 같은 키워드가 JD에서 반복 사용되는 경우(예: "빠른" 3회) 중복 카운팅 여부가 불명확. **고유 키워드 수 vs 총 출현 수** 명시 필요.

### 3.2 결론

**타당(Valid)**. JD 단일 소스의 한계를 솔직히 인정(상한 0.60)하고 단순 규칙으로 전환한 것은 실용적.

## 4. PII 매핑 저장소 (S2)

### 4.1 설계 평가: 합리적 선택

GCS CMEK를 추천하면서 BigQuery/CloudSQL 대안을 제시한 것은 적절한 결정 프로세스.

| 평가 기준 | GCS CMEK | 비고 |
|----------|---------|------|
| 배치 처리 특성 적합 | **적합** | 개별 CRUD 드묾 |
| 비용 | **최저** | $0에 가까움 |
| 감사 추적 | **자동** | Cloud Audit Logs |
| 삭제 요청 대응 | **주의 필요** | JSONL 라인 단위 삭제는 파일 재작성 필요 |
| 접근 통제 | **적절** | 별도 서비스 계정 kg-pii-reader |

**우려**: 개인정보 삭제 요청(GDPR/개인정보보호법) 시 JSONL 파일 내 특정 라인 삭제가 비효율적. 파일을 매번 재작성해야 하므로, 삭제 요청이 빈번하면 CloudSQL이 나을 수 있음. 다만 v12에서 "법률 검토 결과에 따라 조정"이라고 명시했으므로 **현시점에서는 합리적**.

### 4.2 결론

**타당(Valid)**. Phase 0 법률 검토 결과에 따라 유연하게 전환 가능한 설계.

## 5. 전화번호 정규식 (S4)

### 5.1 설계 평가: 실질적 개선

v11 단일 패턴(010-\d{4}-\d{4})에서 8종 변형 커버로 탐지율 **70~80% → 90%+** 향상 추정은 합리적.

**추가 고려 필요**:
| 항목 | 내용 | 심각도 |
|------|------|--------|
| False positive | `0[2-6][0-9]?` 패턴이 일반 숫자열(예: "2024년 3월")과 매칭될 가능성 | Low |
| 국제번호 변형 | `+82-10-...`는 커버하나 `82-10-...`(+없이)는 미커버 | Low |
| 자연어 전화번호 | "공일공-..." 한글 숫자 표기 | Low (NER 영역) |

### 5.2 결론

**타당(Valid)**. v1 MVP 수준에서 충분한 커버리지. 잔여 미탐지는 Phase 2+ NER로 보완 계획 적절.

## 7. 구현 순서 안내 (M3)

### 7.1 설계 평가: 필수적 추가

```
논리적 순서: A → B → B' → C (문서 기술 순서)
실제 순서:   B → C → B' → C → A → C (04.graphrag Phase별)
```

이 차이를 명시적으로 안내한 것은 **구현자의 혼란을 방지**하는 핵심 정보.

**우려**: 01_extraction_pipeline.md §1.3에서 "04.graphrag의 Phase별 일정과 상세 Task를 참조할 것"이라고만 안내하는데, Core v2의 Phase 번호와 추출 로직의 파이프라인 문자(A/B/B'/C)의 **교차 참조 테이블**이 있으면 더 명확.

### 7.2 결론

**타당(Valid)**. 다만 교차 참조 테이블 추가를 권고.

## 8. 관계명 canonical 소스 (M2)

### 8.1 설계 평가: 부분적 해소

v12는 "v19 온톨로지가 유일한 canonical 소스"라고 명시하고, 04.graphrag Core v2 측에서 업데이트할 의무를 선언했다. 그러나:

| 항목 | v12 상태 | 평가 |
|------|---------|------|
| v12 자체 관계명 | v19 기준 (PERFORMED_ROLE, OCCURRED_AT 등) | **정합** |
| 04.graphrag Core v2 관계명 | HAD_ROLE, AT_COMPANY 등 v19 이전 | **불일치 잔존** |
| 구현 시 혼란 가능성 | 추출 로직과 graphrag 코드에서 다른 관계명 사용 | **Medium 리스크** |

v12 문서 내에서 "불일치 발견 시 04.graphrag 문서를 수정"이라고 안내했으나, **누가, 언제** 수정할지는 불명확.

### 8.2 결론

**부분적 타당(Partially Valid)**. 선언만으로는 불충분. 04.graphrag Core v2의 관계명 업데이트를 Phase 0 또는 Phase 1 초반의 **명시적 Task**로 등록 필요.
