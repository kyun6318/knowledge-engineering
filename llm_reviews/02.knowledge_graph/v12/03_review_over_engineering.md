# v12 과도한 설계 리뷰

> 리뷰일: 2026-03-11 | 리뷰어: Claude Opus 4.6
> 대상: 02.knowledge_graph/results/extraction_logic/v12/
> 기준: v1 MVP 범위에서 불필요하게 복잡하거나, 구현 시점에서 결정해도 되는 설계

---

## 전체 평가

v12는 v11 대비 **과도한 설계가 거의 없다**. v12의 변경은 모두 v11 리뷰의 지적 사항을 해소하기 위한 것이고, 불필요한 추가 기능이나 과잉 추상화가 보이지 않는다.

다만, v11에서 이미 존재했던 일부 과도한 설계가 v12에서도 유지되고 있으며, v12가 새로 추가한 설계 중 **"문서 수준에서 과도하게 상세한 부분"**이 일부 존재한다.

심각도: Low 2건, Medium 1건. **v12 단독으로 과도한 설계는 없으며**, v11 잔존 이슈만 남아 있다.

---

## 1. [Low] PII 매핑 저장소 대안 3개 나열

### 위치
04_pii_and_validation.md §1.3.1

### 내용
GCS CMEK를 추천하면서 BigQuery 별도 데이터셋, CloudSQL 2개의 대안을 상세히 기술.

### 과도한 이유
- GCS CMEK 추천이 이미 명확하고 합리적인데, 대안의 **장단점까지 상세히 기술**할 필요는 v1 MVP 단계에서 낮음
- Phase 0 법률 검토 결과에 따라 전환한다고 했으므로, 대안은 "BigQuery 또는 CloudSQL" 한 줄이면 충분
- 문서 길이 증가 대비 실질적 가치 낮음

### 영향
- 문서 복잡도 약간 증가
- 구현자가 "어떤 걸 선택해야 하지?"라고 불필요한 고민을 할 수 있음

### 권고
유지해도 무방하나, **"추천: GCS CMEK, 대안은 법률 검토 결과에 따라 BigQuery 또는 CloudSQL 검토"** 수준으로 축약 가능.

## 2. [Low] 파일 섹션 분리 정규식의 과도한 상세

### 위치
01_extraction_pipeline.md §4.1.1

### 내용
SECTION_PATTERNS와 CAREER_SEPARATOR_PATTERNS의 구체적 정규식을 문서에 기술.

### 과도한 이유가 **아닌** 이유
이 정규식은 v11 리뷰에서 S1("파일 이력서 섹션 분리 로직이 전혀 기술되지 않음")을 해소하기 위해 추가된 것이므로, **과도하지 않고 적절**하다.

다만, 이 정규식은 **구현 코드에 들어갈 내용**이지 설계 문서에 최종 형태로 있을 필요는 없다. Phase 2-0 PoC에서 검증 후 수정될 가능성이 높으므로, "초기 버전"임을 명시하면 충분.

### 결론
**과도하지 않음**. 구현 방향을 명확히 보여주는 적절한 수준.

## 3. [Medium] 비용 추정의 과도한 정밀도

### 위치
01_extraction_pipeline.md §8.2, §8.3

### 내용
```
CandidateContext DB: 가변 / 500K / $992 / Batch $496
파일 섹션 분리 LLM 폴백: $0.002 × 30K건 = ~$60 / Batch $30
```

### 과도한 이유
- **PoC 미실시 상태**에서 $992, $60 같은 구체적 금액은 **거짓 정밀도(false precision)**
- 핵심 가정(Career 분포, 패턴 성공률, workDetails null 비율)이 모두 미검증
- 실제 비용은 ±30% 이상 변동 가능
- 구현자가 이 금액을 예산 기준으로 삼으면 오해 소지

### 영향
- 실제 비용과의 괴리 시 신뢰도 하락
- 예산 승인 과정에서 과도한 기대치 형성

### 권고
금액을 **범위(range)**로 표현하는 것이 더 정직:
```
CandidateContext DB: $400~$600 (Batch, Career 분포 의존)
파일 섹션 분리: $20~$60 (패턴 성공률 의존)
```

## 4. v11 잔존 — 과도한 설계 목록

v11 리뷰에서 지적했으나 v12에서 해소되지 않은 항목:

| 항목 | v11 심각도 | v12 상태 | 비고 |
|------|----------|---------|------|
| 6단계 검증 체크포인트 (CP1~CP6) | Low | 유지 | v1 MVP에서 6단계 모두 구현은 과도할 수 있으나, 각 체크포인트가 가벼우므로 수용 가능 |
| Confidence 4단계 캘리브레이션 | Low | 유지 | `min(llm_confidence, source_ceiling)` 한 줄 구현이므로 과도하지 않음 |

### 결론

v12는 **과도한 설계가 거의 없는 성숙한 문서**. 비용 정밀도 문제만 주의 필요.
