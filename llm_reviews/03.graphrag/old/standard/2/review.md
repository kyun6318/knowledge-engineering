## 총평

standard.2은 standard.1 리뷰의 15건 권장사항(R-1~R-15)을 **전수 반영**한 개선판이다. CRITICAL 1건(Phase 1-1 일정 확장)과 HIGH 4건 모두 해결책이 기술되었고, 과설계 영역(Workflows YAML, Looker Studio, 품질 평가 통계, 크롤링 테이블)이 축소되어 **인력 부담이 더 균등하게 분배**되었다.

총 타임라인은 27~33주로 standard.1(26~33주)와 실질적으로 동일하나, Phase 1의 여유 확보(+1주 전처리, +1주 버퍼)와 Phase 2의 비핵심 태스크 축소(품질 평가/서빙 인터페이스 각 -4일)로 **핵심 태스크에 집중하는 구조**가 되었다.

## 1. 실현 가능성 검증

### 1.1 [RISK-MEDIUM] Phase 1-0 + 1-1 합산 일정 — 적절하나 타이트

- **현황**: Phase 1-0(2~3일) + Phase 1-1(3주) = 약 3.5주
- **분석**:
  - standard.1의 2주에서 3.5주로 확장되어 태스크당 평균 2.2인일→3.3인일로 개선
  - 3주 일정 배분표가 DE/MLE 병렬 작업을 명확히 보여줌
  - 다만 Week 5에 1-0(2~3일) + 1-1 시작이 동시에 진행되어 여유가 적음
- **잔존 리스크**: HWP 파서(1-1-2)가 Phase 0 결과에 따라 LibreOffice Docker 통합이 필요하면 예상보다 1~2일 추가 소요 가능
- **평가**: standard.1 대비 크게 개선. **RISK-HIGH→RISK-MEDIUM으로 하향**.

### 1.2 [RISK-MEDIUM] Pre-Phase 0 복합 지연 — 최악 시나리오 명시됨

- **현황**: standard.2에서 NICE DB + 법무 PII + Batch API의 동시 지연 시 4주 밀림 시나리오 명시
- **분석**:
  - Batch API quota를 "지금 즉시 확인"으로 강화하여 지연 원인 1개 제거
  - NICE DB + 법무 PII 병렬 요청 + contingency가 명시됨
  - 최악 시나리오 31~37주가 명시되어 이해관계자 기대치 관리 가능
- **평가**: 적절한 수준의 위험 관리. **잔존 리스크 수용 가능**.

### 1.3 [RISK-MEDIUM] 인력 2명 체제 — 역할 분담으로 완화

- **현황**: DE/MLE 역할 분담표 + 1주 버퍼 + on-call SLA 추가
- **분석**:
  - 역할 분담표로 병목 지점(예: MLE가 프롬프트와 PII를 동시에 해야 하는 상황) 사전 식별 가능
  - 1주 버퍼로 번아웃 방지
  - Phase 2-1의 업무 시간 외 SLA 정의("Slack만 전송, 다음 업무일 아침 대응")로 야간/주말 부담 경감
  - 인력 추가 판단 시점(Phase 1 완료, Week 19~20)이 명시됨
- **평가**: standard.1 대비 크게 개선. **실행 가능한 수준**.

### 1.4 [RISK-LOW] Batch API 수치 — 즉시 확인으로 불확실성 해소 가능

- **현황**: "계획 확정 전 즉시 확인"으로 강화, 결과별 타임라인 보정 표 추가
- **분석**: Anthropic 콘솔에서 대부분의 정보를 즉시 확인 가능하므로, 이 리스크는 **계획 확정 전에 해소 가능**
- **평가**: **RISK-HIGH→RISK-LOW로 하향**. 다만 실제로 확인을 수행해야 함.

### 1.5 [RISK-LOW] Neo4j Connection Pool — 확인 + 코드 대응 추가

- **현황**: Phase 0-5-2b에 connection pool 확인 + exponential backoff 코드
- **분석**: tasks 수를 connection pool에 맞춰 조정하는 방식이 실용적. Professional 전환 후 상향도 계획됨.
- **평가**: **적절한 대응**.

### 1.6 [RISK-MEDIUM] standard.1에서 언급된 잔존 리스크 — 변경 없음

다음 리스크는 standard.2에서도 변경 없이 잔존:
- Neo4j AuraDB APOC Extended 지원 여부 (Phase 0-5-2a에서 확인 예정)
- Anthropic Claude Haiku 4.5 Batch API 가용성 (즉시 확인으로 해소 가능)
- Neo4j 도쿄↔서울 RTT (Phase 1-4-9 벤치마크에서 측정 예정)
- pyhwp 라이브러리 성숙도 (Phase 0-4-7에서 평가)
- Docker 이미지 크기 cold start (Phase 0-4-7 결과에 따라 결정)

> 이들은 Phase 0에서 해소 예정이므로 standard.2에서 추가 대응 불필요.

## 2. 과도한 일정/기능 (Over-Engineering / Premature)

### 2.1 과설계 영역 — standard.1 대비 효과적 축소

standard.1 리뷰에서 지적된 과설계 4건이 모두 적절히 축소됨:

| standard.1 과설계 | standard.2 대응 | 평가 |
|-----------|-----------|------|
| Workflows YAML 30줄+ | DAG 텍스트 구조만 | **적절** — Phase 2 시작 시 YAML 작성 |
| Looker Studio 11패널 | Phase 3 이동, Saved Queries | **적절** — 2명 체제에 맞는 방식 |
| Phase 2-2 Cohen's d | Phase 3-5로 이동 | **적절** — 모델 비교 시 적용 |
| 서빙 인터페이스 1주 | 3일로 축소 | **적절** |

### 2.2 [LOW] standard.2 자체의 과설계 여부

- **Phase 1-0 코드 리팩토링**: 2~3일은 적절. 프로젝트 구조(`src/` 디렉토리)까지 명시한 것은 구현 시 바로 참조 가능하여 유용.
- **DE/MLE 역할 분담표**: Phase별 매트릭스가 상세하나, 실제 운영에서는 유동적일 수 있음. 다만 **가이드라인**으로서의 가치가 높으므로 과설계로 보기 어려움.
- **운영 매뉴얼 10개 목차**: Phase 2-6에서 작성 시 범위가 넓을 수 있으나, 인수인계에 필수적이므로 적절.

> **결론**: standard.2에서 새로 추가된 과설계 요소는 **없음**.

## 3. 부족한 일정/기능 (Missing / Under-specified)

### 3.1 [MEDIUM] 증분 처리 일일 1,000건 근거 — 보완되었으나 추정 기반

- **현황**: "가정: 월 20,000~30,000건 유입 → 일 ~1,000건"으로 명시, Phase 0-3에서 실제 유입량 확인 후 보정
- **분석**: 근거가 "가정"에 기반하며, 실제 데이터가 없는 상태. 다만 보정 전략(< 500/일 시 주 1회, > 2,000/일 시 Batch API)이 있어 유연하게 대응 가능.
- **평가**: 현 단계에서 가능한 최선의 추정. Phase 0-3에서 검증 예정이므로 **수용 가능**.

### 3.2 [MEDIUM] Phase 2-2 전문가 확보 리드타임

- **현황**: "Phase 1 완료 시점(Week 18~19)에서 전문가 확보 시작, Phase 2-2 시작(Week 22) 전까지 최소 3주 리드타임"
- **분석**: 전문가 2인 확보에 3주는 적절하나, 전문가 풀이 부족한 경우(도메인 전문가, HR/채용 경험자) 확보 실패 리스크 있음
- **권장**: 전문가 후보 리스트를 Phase 1 초반(Week 5~6)에 사전 작성하여 리드타임 확보

### 3.3 [MEDIUM] crawl_raw_data 통합 테이블 — 쿼리 복잡성 증가 가능

- **현황**: homepage_pages + news_articles를 crawl_raw_data로 통합, source_type으로 구분
- **분석**:
  - 통합으로 테이블 수 감소는 긍정적
  - 다만 homepage와 news의 스키마 차이(page_type vs category, source_media, publish_date 등)로 인해 많은 nullable 컬럼 발생
  - 쿼리 시 `WHERE source_type = 'homepage'` 필터가 항상 필요
- **잔존 리스크**: 성능 문제는 아님 (BigQuery는 컬럼 스토리지), 쿼리 가독성 약간 저하
- **평가**: 초기 1,000개 기업 규모에서는 충분. 쿼리 가독성은 뷰로 보완 가능.

### 3.4 [LOW] Phase 1-4-4a ER 전수 검수 시간

- **현황**: 500개 Organization 전수 검수에 0.5일 배정
- **분석**: 500개를 2명이 0.5일(4시간)로 검수하면 1건당 ~30초. 단순 목록 확인이면 가능하나, 애매한 케이스(예: "삼성전자" vs "삼성전자 DS부문")에 대한 판단에 시간 소요
- **권장**: 1일로 여유 확보하거나, 자동 검수 도구(유사 쌍 하이라이트)를 Phase 1-4-4에서 함께 구현

### 3.5 [LOW] standard.1에서 지적되었으나 standard.2에서 미반영된 사항

없음. standard.1 리뷰 15건 전수 반영됨.

## 5. 잘된 부분 (Strengths)

### 5.1 standard.1 리뷰 전수 반영 — 2차 리뷰 사이클 완성

- 15건 전수 반영으로 v2→standard.1→standard.2의 **3단계 리뷰 사이클**이 완성됨
- 각 변경에 `[standard.1.1-N]` 태그 부여로 추적 가능

### 5.2 DE/MLE 역할 분담표 — 병목 사전 식별

- Phase별 역할 매트릭스가 "누가 언제 무엇을 하는지" 명확하게 보여줌
- 인력 추가 시 온보딩 가이드 역할도 겸함

### 5.3 증분 처리 Graph 업데이트 전략 — 실용적 설계

- DETACH DELETE + 재생성 전략이 Chapter 노드의 부분 업데이트보다 안전하고 구현 단순
- Embedding, MAPPED_TO 재계산 연쇄 효과까지 기술

### 5.4 업무 시간 외 SLA — 현실적 대응

- "CRITICAL은 Slack만, 다음 업무일 아침 대응"으로 2명 체제의 한계를 인정하면서도 자동화로 보완
- kg-batch-poll 30분 주기 자동 실행으로 야간 수동 대응 최소화

### 5.5 Phase 간 Go/No-Go 기준 완성 — 의사결정 투명성

- Phase 0→1, Phase 1→2, Phase 2→운영 3단계 Go/No-Go 기준이 모두 명시됨
- 각 기준에 "미달 시 조치"가 있어 판단 모호성 최소화

### 5.6 비용 수치 통일 — 문서 신뢰성 향상

- 04_cost와 05_models 간 $28~30 불일치 해소
- LLM/인프라 경계 명확화로 비용 구조 투명성 향상

### 5.7 과설계 축소 — 적절한 복잡성 수준

- Workflows YAML, Looker Studio, Cohen's d, 크롤링 테이블 등 4개 영역에서 복잡성 축소
- 축소된 인력을 증분 처리 보완 + 인수인계에 재할당하여 **총 타임라인 거의 변경 없이** 내실화
