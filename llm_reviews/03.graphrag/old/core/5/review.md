> **리뷰 일자**: 2026-03-11

## 1. 총평

v5는 v4 리뷰의 결론대로 **"최소한의 보완만 반영한 최종 실행 버전"**이다. S1(비용 동기화) + A1~A4(인지 사항 문서화) + U2(mask_phones 주석) — 총 6건만 반영했으며, 아키텍처나 일정 변경은 없다. v4 리뷰에서 "추가 리비전 불필요, 실행 단계 즉시 진입 가능"으로 판정했고, v5는 이를 존중하여 **문서 정합성 보완에만 집중**했다.

**v5의 핵심 가치**:
- S1: Overview와 06_cost 간 비용 수치 $36~71 불일치 해소 → 단일 출처 완성
- A1~A4: v4 리뷰의 "인지 사항"을 문서에 명시화하여 암묵지 제거
- U2: mask_phones() 토큰 형식 변경 시 주의 주석으로 유지보수성 확보

**결론**: v5는 **더 이상의 리비전이 불필요한 최종 실행 버전**이다. 문서 완성도는 높고, 27주 $5,527~9,137 계획의 내부 일관성이 확보되었다. 아래 리뷰에서 발견된 이슈는 모두 LOW 이하이며, 실행 단계에서 자연스럽게 해소할 수 있는 수준이다.

## 3. 과도한 설계 (Over-Engineering)

### O1. AuraDB 마이그레이션 7단계 절차 — 1,000건에 비해 과잉 (VERY LOW)

**현상**: 03_phase2 §2-2에 AuraDB Free→Professional 마이그레이션 절차를 7단계로 명시했다.

```
현재:
  1. Professional 인스턴스 생성
  2. Cypher 쿼리로 기존 Free에서 노드/관계 읽기
  3. UNWIND 배치 적재
  4. Vector Index 재생성 + Embedding 재적재
  5. 연결 정보 업데이트 (URI, 인증)
  6. API + Scheduler 연결 대상 변경
  7. Free 인스턴스 삭제

평가:
  - 1,000건 + Vector Index → 실제 작업 시간 30분~1시간
  - 7단계 중 5, 6, 7은 표준 인프라 운영이지 마이그레이션 고유 작업이 아님
  - v4 리뷰에서 "한 줄 추가" 수준을 권장했으나 v5에서 7단계로 확대
  - 해롭지는 않으나, 문서 볼륨 대비 실제 가치가 낮음
```

**권장**: 수정 불필요. 실행 시 참조 가이드로 활용 가능하므로 있는 것이 없는 것보다 나음. 다만 이것이 "계획 문서"에 포함될 필요가 있는지는 의문 — 실행 단계에서 Runbook에 옮기는 것이 적절.

### O2. v5 변경점 추적의 누적 복잡도 (LOW)

**현상**: 06_cost.md §5에 리뷰 반영 추적 테이블이 **4세대** 쌓여 있다.

```
현재 추적 테이블:
  1. v5 반영 (v4 리뷰): S1, A1~A4, U2 — 6행
  2. v4 반영 (v3 리뷰): R1~R8, O3, O7 — 10행
  3. v3 반영 (v2 리뷰): N1~N9 — 9행
  4. v3 반영 (v12 Extraction Logic): M1~M3, S1~S5, C3 — 10행

총 35행의 추적 테이블이 비용 문서에 존재.

v4 리뷰(O1)에서 이미 지적:
  "추적 테이블이 06_cost에 있는 이유는 '마지막 문서'이기 때문일 뿐"
  → v5에서도 동일 구조 유지

평가:
  - v5가 최종 실행 버전이므로 추가 리비전이 없다면 문제 없음
  - 하지만 실행 중 소규모 수정이 발생하면 이 추적 체계가 짐이 될 수 있음
  - 비용 문서의 핵심 역할(Single Source of Truth)이 추적 테이블에 의해 희석됨
```

**권장**: 현재 상태로 실행 진입. 만약 실행 중 문서 수정이 필요해지면 추적 테이블을 별도 `CHANGELOG.md`로 분리하는 것을 검토. v4 리뷰(O1)과 동일한 판단.

### O3. Overview §3의 변경 이력 과잉 (LOW)

**현상**: 00_overview.md §3에 v5 변경(§3.1) + v4 변경(§3.2) + 데이터 볼륨(§3.3) + 비용 변동 요약(§3.4) — 총 4개의 변경 이력 섹션이 존재한다. v3→v4→v5의 **전체 변경 이력이 보존**되어 있어 Overview가 "현재 상태 요약"이 아닌 "변경 이력 문서" 성격을 가짐.

```
문서 길이 분석:
  00_overview.md: 450행
    - §1 로드맵: ~84행 (핵심)
    - §2 타임라인: ~85행 (핵심)
    - §3 변경 이력: ~70행 (참조용)
    - §4~§15 스키마/환경/비용/후속: ~211행

변경 이력이 Overview의 ~15%를 차지.
실행 단계에서 v3→v4 변경 이력의 참조 가치는 낮음.
```

**권장**: 수정 불필요. 실행 단계에서 Overview를 읽는 독자(Phase 시작 시 전체 그림 파악)에게 변경 이력은 불필요하지만, 삭제하면 의사결정 근거가 사라지므로 보존이 합리적. **다만 실행 중 이 섹션을 참조할 일은 거의 없을 것**임을 인지.

### O4. 각 Phase 문서의 "v3와 동일" 반복 표기 (VERY LOW)

**현상**: 모든 Phase 문서에서 변경 없는 섹션에 "> v3와 동일." 또는 "> v3와 동일 (코드 예시는 XX 참조)."를 반복한다. v5 전체에서 이 표현이 약 **20회** 이상 등장.

```
예시:
  01_phase0: "v3와 동일" × 6회
  02_phase1: "v3와 동일" × 3회
  03_phase2: "v3와 동일" × 2회
  04_phase3: "v3와 동일" × 3회
  05_phase4: "v3와 동일" × 4회
  06_cost: "v3와 동일" × 3회

v3 문서가 별도로 존재하지 않는 한(v5가 최종본이므로 v3은 이력 속에만 존재),
"v3와 동일"이라는 참조는 실행 단계에서 의미 없음.
```

**평가**: 리뷰 과정에서 변경점 추적을 위해 필요했던 표기이며, **v5가 최종본이므로 해가 되지 않는다**. 실행 단계에서 이 표기를 의식적으로 무시하면 됨.

**권장**: 수정 불필요. 실행 단계에서 문서를 "클린업"하고 싶다면 일괄 제거 가능하나, 우선순위는 매우 낮음.

## 4. 부족한 설계 (Under-Engineering)

### U1. Phase 0 PoC 비용 외삽 기준 ±50%의 관대함 (LOW)

**현상**: v5 A2에서 Go/No-Go 기준에 "PoC 20건 평균 토큰 → 600K 외삽, 예산 $1,690 대비 ±50%"를 추가했다. **±50%는 매우 넓은 범위**이다.

```
±50% 의미:
  - 하한: $845 (50% 절감 → 매우 낙관적, 문제 없음)
  - 상한: $2,535 (50% 초과 → Phase 2 비용 $2,049~2,364의 LLM 부분이 ~$2,535까지 상승)
  - $2,535는 전체 예산 $5,527~9,137의 ~28~46%

  - 상한 시나리오에서도 Go?
    → $1,690 × 1.5 = $2,535이면 Phase 2 LLM 비용이 $845 추가
    → 전체 예산 대비 +9~15% 수준 → 관리 가능한 범위

  - 그러나 ±30%로 좁히면?
    → 상한 $2,197: 추가 비용 $507, 전체 대비 +5~9%
    → 더 보수적이지만 Go/No-Go 판정이 엄격해짐

비교:
  - PoC 20건 샘플의 통계적 신뢰도 자체가 낮음 (표본 크기 제한)
  - 20건으로 600K를 외삽하는 것 자체가 ±30% 정도의 불확실성을 내포
  - 따라서 ±50%가 비현실적이지는 않으나, "통과 기준"으로서는 관대함
```

**평가**: Phase 0 Go/No-Go의 핵심은 **LLM 추출 품질**이지 비용이 아니다. 비용이 ±50%를 초과하면 "모델 변경" 또는 "프롬프트 최적화"로 대응 가능하므로, ±50%는 **"중단할 정도는 아닌"을 판별하는 안전장치**로서 적절하다.

**권장**: 수정 불필요. 실행 시 PoC 20건의 토큰 사용량이 예상(평균 ~1,200 input + ~800 output)과 크게 다르면 ±50% 이전에 원인 분석이 먼저 이루어질 것이다.

### U2. A1의 JD 파서 대응 절차 — "3-3에서 0.5주 흡수"의 실현 가능성 (LOW)

**현상**: v5 A1에서 job-hub API 미확정 시 "즉시 0.5주→1주 확장, 3-3 Organization ER에서 0.5주 흡수"를 명시했다. **3-3에서 0.5주를 흡수할 수 있는가?**

```
Phase 3 일정 (v5):
  3-0: 2일 (매칭 설계 + MAPPED_TO 추정)
  3-1: 0.5주 (JD 파서) → 미확정 시 1주
  3-2: 2주 (CompanyContext)
  3-3: 1.5주 (Organization ER) → 흡수 시 1주
  3-4: 2주 (MappingFeatures + MAPPED_TO)
  3-5: 병행 (통합 테스트)
  합계: 6.4주 → 흡수 시 여전히 6.4주 (3-1에 +0.5, 3-3에 -0.5)

v4 리뷰(O4)에서 이미 지적:
  "3-3(ER 1.5주): '주요 그룹 초기 버전'이 어디까지인지 불명확"

3-3을 1.5주→1주로 축소하면:
  - 주요 그룹 5개(삼성/현대/SK/LG/롯데)의 계열사 사전 구축만으로도 1주 소요 가능
  - 한국어 특화(회사명 정규화, 약칭 처리 등) 포함 시 1주는 타이트
  - 하지만 "주요 그룹 초기 버전"을 더 축소(상위 3개 그룹만)하면 실현 가능

실제 리스크:
  - 3-1이 1주로 확장되면 3-3은 1주 → 총 6.4주 (변동 없음)
  - 문제는 3-3의 1주가 ER 품질에 미치는 영향
  - ER 품질이 낮으면 매칭 스코어의 stage_match(25%) 정확도 하락
  - 하지만 stage_match가 NICE Rule 기반이지 ER에 직접 의존하지 않음
  - ER은 "동일 기업 인식"(삼성전자 = 삼성 = SAMSUNG)에 영향
  - 결론: 매칭 정확도보다 검색 리콜에 영향 → Phase 4 이후 보강 가능
```

**평가**: A1의 대응 절차는 **논리적으로 성립**한다. 3-3에서 0.5주 흡수 시 ER 범위 축소가 필요하지만, ER은 Phase 4 이후 점진 보강 가능한 영역이다.

**권장**: 수정 불필요. 다만 Phase 3 시작 시 "3-3 축소 시 ER 범위를 상위 3개 그룹으로 한정"이라는 구체적 기준을 사전에 합의하면 실행 시 혼란 방지.

### U3. 증분 처리 주기 결정 기준의 모호함 (LOW)

**현상**: Phase 0에서 일일 유입량을 확인(N4)하고 증분 주기를 결정하는데, Phase 4의 Cloud Scheduler 설정에서 **실제 주기 결정 논리**가 명시되지 않았다.

```
현재 설계:
  01_phase0 (N4):
    - daily_count < 500 → 주 1회
    - daily_count 500~2,000 → 일 1회
    - daily_count > 2,000 → Batch API 필요

  05_phase4 §4-4:
    - Cloud Scheduler 일일 증분 (구체적 주기 미명시)
    - Cloud Workflows DAG (처리 흐름은 있으나 스케줄 미명시)

  갭:
    - Phase 0에서 측정한 유입량이 Phase 4 시점(~6개월 후)에도 유효한가?
    - 유입량이 변동하면 증분 주기를 자동으로 조정하는 메커니즘이 없음
    - "일일 증분"으로 고정 표기 → 실제 유입량이 일 100건이면 비효율
```

**평가**: 이것은 "설계 부족"이라기보다 **운영 단계에서 자연스럽게 튜닝할 영역**이다. Phase 4 시점에서 실측 데이터 기반으로 스케줄을 결정하는 것이 합리적이며, 지금 시점에서 자동 조정 메커니즘을 설계하는 것은 과도하다.

**권장**: 수정 불필요. Phase 4 Runbook에 "증분 주기는 실측 유입량 기반으로 결정, 분기별 재검토" 정도를 포함하면 충분.

### U4. Phase 2 기본 시나리오 여유 0일 — 여전히 타이트 (MEDIUM, v4에서 이관)

**현상**: v4 리뷰 §5.2에서 이미 지적한 사항이며, v5에서 변경되지 않았다. Phase 2 기본 시나리오(12h/라운드)에서 45일/45일 = **여유 0일**.

```
v5 상태:
  - 기본 시나리오: 100% 완료, 여유 0일 (v4와 동일)
  - 비관 시나리오: 50%, 명시적 Batch 할당 (v4와 동일)

핵심 불확실성:
  1. Batch API 라운드 시간이 12h를 초과할 가능성
     - Anthropic 서버 부하, 네트워크, rate limiting 등
     - 1~2시간 초과만으로도 전체 일정에 파급

  2. 적응형 호출(v12 M1) 비율이 가정(80/20)과 다를 가능성
     - 1-pass 70% / N+1 30%이면 chunk 수 증가 → 라운드 수 증가
     - Phase 0 PoC에서 Career 수 분포를 확인하므로 조기 감지 가능

  3. 파일 100K의 Hybrid 섹션 분리 실패율이 30% 초과 시
     - LLM 폴백 건수 증가 → Batch API 부하 증가 → 라운드 시간 증가

대응 수단 (v5에 이미 존재):
  - R6: DB 500K 먼저 처리 → Week 12에 실측 데이터 확보
  - 비관 대응: Phase 3에 명시적 Batch 할당
  - 버퍼 1주 (Week 16)
  - Phase 0 Go/No-Go에서 Career 수 분포 + 비용 외삽(A2)
```

**평가**: v5에서 이 리스크를 추가로 완화하는 것은 가능하지 않다(일정 연장 외에). v4의 대응(R5+R6+비관 계획)이 현실적인 최선이며, **관리 가능한 리스크**라는 v4 리뷰의 판단을 유지한다.

**권장**: 수정 불필요. 실행 시 **Week 10 시점에서 라운드 시간 실측**을 중간 체크포인트로 활용. 라운드 시간이 18h를 초과하면 즉시 비관 대응 계획 발동.

### U5. Gold Label 인건비의 비용 비중 과대 — 대안 부재 (LOW)

**현상**: Phase 4 비용 $3,128~6,198 중 Gold Label 인건비가 $2,920~5,840으로 **전체의 93~94%**를 차지한다. 이것은 Phase 4가 실질적으로 "품질 평가 인건비" Phase임을 의미한다.

```

  Phase 4 Gold Label: $2,920~5,840 (93%)

전체 예산 중 Gold Label 비중:
  - $2,920~5,840 / $5,527~9,137 = 53~64%
  - 전체 프로젝트 비용의 절반 이상이 "사람이 100~200건 검수"에 사용

현재 설계의 대안:
  - "(국내 전문가 대안) ~100~200만원" → 06_cost §1.5에 언급만
  - 구체적 절차, 전문가 확보 방안, 품질 비교 없음

관점 정리:
  - Gold Label이 매칭 품질의 유일한 검증 수단이므로 비용 자체는 정당
  - 하지만 국내 전문가($1,370 원화 환산) 대안이 실현 가능하면 비용 60%+ 절감
```

**평가**: 품질 검증은 투자 대비 가치가 높은 항목이므로 비용 자체를 문제 삼기 어렵다. 다만 **국내 전문가 대안의 구체화**가 Phase 4 진입 전에 이루어져야 비용 효율성을 확보할 수 있다.

**권장**: 수정 불필요. Phase 3 후반(Week 22~23)에 Gold Label 전문가 확보 절차를 시작하고, 국내 채용 도메인 전문가 활용 가능성을 사전 검토.

## 9. 결론

v5는 v3→v4→v5의 **3회 리뷰-반영 사이클을 거친 최종 실행 계획**이다.

**강점:**
- 비용 Single Source of Truth 완성 (S1 해소, 전 문서 $5,527~9,137 통일)
- v4 리뷰의 "인지 사항" 4건이 문서에 명시적 대응 절차로 기록됨
- HIGH/MEDIUM 리스크는 v3→v4에서 이미 해소/완화되었고, v5는 이를 유지
- 27주, 2인 체제, $5,527~9,137 계획의 내부 일관성 완전 확보

**과도한 설계**: 4건, 전부 VERY LOW~LOW. 리뷰-반영 과정에서 자연스럽게 발생한 문서 볼륨 증가이며, 프로젝트 실행에 부정적 영향 없음.

**부족한 설계**: 5건, 전부 LOW~MEDIUM. 실행 단계에서 자연스럽게 해소 가능. Phase 2 기본 시나리오 여유 0일(U4)이 가장 높은 리스크이지만, v4에서 이미 관리 방안(R5+R6+비관 대응)이 마련되어 있음.

**최종 판단**: v5 계획은 **추가 리비전 없이 실행 단계로 즉시 진입**해야 한다. 추가 리뷰-반영 사이클은 한계효용이 거의 없으며, 실행을 통해서만 해소할 수 있는 불확실성(Batch API 라운드 시간, JD 파서 API 스펙, 적응형 호출 실측 비율)이 남아 있다. **지금은 계획의 시간이 아니라 실행의 시간이다.**
