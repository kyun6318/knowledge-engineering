## 1. 종합 평가 (Executive Summary)

**전체적인 흐름과 설계 정합성은 매우 우수합니다.** 
방향성을 잃지 않고, 스키마 레벨에서의 촘촘한 기획(v10)이 추출 레벨(v7)의 파이프라인 로직으로 정확히 번역되었으며, 최종적으로 GCP 인프라 레벨(v2)의 Cloud Run / Workflows / BigQuery 아키텍처로 물 흐르듯 연결되고 있습니다. 특히, '어떻게 추출할 것인가'를 넘어서 '어떻게 대용량을 비용 효율적으로 클라우드에 얹을 것인가'에 대한 고민이 깊게 반영되어 있습니다.

다만, 세 단계의 문서를 교차 검증한 결과 **데이터 동기화(삭제 처리) 및 스킬 노드 과부하(Supernode) 대응 측면에서 미세한 끊어짐(Gap)** 이 발견되었습니다. 아래에 상세한 강점과 개선 필요 사항을 정리했습니다.

---

## 2. 단계별 정합성 강점 (Strengths of Alignment)

### 2.1 개념 및 데이터 모델의 완벽한 1:1 매핑
* **흐름 유지**: `CompanyContext`, `CandidateContext`, `MappingFeatures`라는 3대 축이 스키마(v10)에 정의된 형태 그대로 추출 파이프라인(v7)의 입력/출력 구조로 쓰이고, 플랫폼(v2)의 GCS 디렉토리 구조 및 BigQuery 스키마로 1:1 매핑되어 있습니다.
* **신뢰도(Confidence) 보존**: 스키마(v10)에서 정의한 Source Tier 기반 Ceiling(`min(extraction, ceiling)`) 원칙이 추출 파이프라인(v7) 코드에 정확히 구현되어 있으며, 플랫폼(v2) 빅쿼리 테이블에도 `confidence` 및 `adjusted_confidence`로 적재되도록 설계되어 추적 가능성이 보장됩니다.

### 2.2 대용량 처리와 비용 모델의 현실적 매핑
* **비용 현실화**: 스키마 레벨에서 요구하는 LLM 추론 영역이 많음(50-65% 비중)을 추출 파이프라인(v7)에서 인정하고, 이를 해결하기 위해 비용이 50% 저렴한 Anthropic Batch API를 적극 도입했습니다.
* **오케스트레이션 정합성**: 플랫폼(v2)은 이 Batch API의 비동기적 특성(최대 24시간)을 완벽히 흡수하기 위해 워크플로우를 `kg-batch-prepare` → `kg-batch-submit` → `kg-batch-collect` 3단계 Cloud Run Job으로 세분화하고 Cloud Workflows로 묶어 결함 없는 아키텍처를 제시했습니다.

### 2.3 크롤링 파이프라인의 유기적 결합
* **흐름 유지**: 스키마(v10)의 `06_crawling_strategy.md`에서 기획한 "홈페이지+뉴스" T3/T4 확장 전략이 추출 로직(v7)의 `extract_structural_tensions()` 등으로 구체화되었고, 플랫폼(v2)의 Phase 2에 이식되어 Playwright와 네이버 API를 활용한 서버리스 크롤링 아키텍처로 빈틈없이 연결되었습니다.

## 3. 식별된 틈새 및 개선 권고사항 (Gaps & Recommendations)

설계의 방향성은 매우 훌륭하나, 각 페이즈 간 세부 구현을 연결할 때 누락되거나 어긋나는 지점이 있습니다.

### GAP 1: "노드 삭제(Soft-delete)" 로직의 구현 누락
* **증상**: 추출 계획(v7)의 `04_execution_plan.md` 증분 처리 플로우에서는 "이력서 삭제/갱신 건 발생 시 Graph에서 관련 노드/엣지 soft-delete"한다고 명시되어 있습니다. 그러나 추출 로직(v7)의 `load_company_to_graph()` 코드나 플랫폼(v2)의 `kg-graph-load` 파이프라인 로직은 모두 **생성과 덮어쓰기(`MERGE`) 로직만 존재하고 삭제를 처리하는 로직(Cypher 쿼리 등)이 없습니다**.
* **영향**: 후보자가 이력서에서 특정 기술이나 경력을 지웠을 때, Context JSON은 갱신되지만 Neo4j에는 이전 엣지(Edge)가 고스트 데이터로 영구히 남게 되어 매칭 오염이 발생합니다.
* **조치 권고**: `CandidateContext`를 파싱하여 나온 결과를 Graph에 `MERGE`하기 전에, 해당 노드(Person, Chapter 등)에 연결된 **기존 엣지들을 한 번 끊어주거나(Detach) 만료(Soft-delete) 시키는 Graph Sync 로직**을 플랫폼(v2) 로우레벨 설계에 명시적으로 추가해야 합니다.

### GAP 2: 범용 스킬 "초거대 노드 (Supernode)"에 대한 플랫폼 방어 부재
* **증상**: 이전 V10 스키마 리뷰에서도 지적했듯 `Python`, `Java`, `Communication` 같은 노드는 수십만 개의 엣지와 연결되는 지뢰(Supernode)가 됩니다. 스키마와 추출(v7)에서는 "스킬 정규화"를 언급하지만, 플랫폼(v2)에서는 이 Supernode가 야기하는 Neo4j OOM(Out of Memory)이나 트랜잭션 타임아웃 방지책이 없습니다.
* **조치 권고**: 플랫폼(v2)의 `kg-graph-load` Job에 배치 트랜잭션 크기 제한(예: 1,000건 단위 Commit)을 설정하거나, 지나치게 빈도가 높은 범용 스킬(Stop-words)은 Neo4j 노드로 잇지 않고 임베딩 텍스트(Vector)로만 처리하도록 파이프라인 설계를 최적화할 필요가 있습니다.

### GAP 3: 데드 레터 큐(Dead-letter Queue) 재처리의 수동성
* **증상**: 파싱에 실패하거나 LLM JSON이 깨질 경우 플랫폼(v2)은 `kg-dead-letter`를 통해 일 1회 자동 재시도하도록 셋업되었습니다. 하지만 LLM 파싱 단계가 3-tier 구조(json-repair 등)까지 갔는데도 실패했다면 단순 재시도로는 풀리지 않습니다 (원문 이력서 레이아웃의 완전한 비표준 등).
* **조치 권고**: 에러 유형(일시적 API 통신 실패 vs 파싱/추론 구조적 실패)을 분리하여 `BigQuery parse_failure_log`에 명시하고, 단순 재시도 루프에 빠지지 않도록 룰을 조정해야 합니다. 일정 횟수 실패 시 "Human in the Loop(수동 검수)" 태스크로 격리하는 운영 매뉴얼이 추가되어야 완벽합니다.

## 4. 결론
지금까지의 설계는 "현실 세계의 더러운 데이터(이력서/크롤링)"를 "깨끗한 온톨로지와 그래프"로 바꾸기 위한 **매우 훌륭한 엔지니어링 청사진**입니다. 위에서 언급한 [삭제 처리 동기화]와 [그래프 성능(슈퍼노드) 병목] 두 가지만 V2/V3 플랫폼 운영 단계 이전에 방어해 둔다면 성공적인 론칭이 가능할 것입니다.
