# V1 분리 계획 비교 리뷰 및 타당성 분석 (vg2)

> **대상**: v1 `c` 계열 계획서(4종) vs v1 `g` 계열 계획서(1종)
> **목적**: 두 가지 접근 방식의 장단점을 구조적으로 비교 리뷰하고, 타당한 강점을 융합하여 새로운 **vg2** 계획 수립의 기반을 마련한다. 
> *(참고: 기존 vc2 계획은 본 분석에서 철저히 배제함)*

---

## 1. 계획서 접근 방식 비교

| 차원 | `g` 계열 (g_01_...) | `c` 계열 (c_00_... ~ c_03_...) |
|------|--------------------|------------------------|
| **핵심 철학** | **관심사 분리 및 아키텍처 비전** (Microservice/Polyglot) | **실행 가능성 및 태스크 중심** (상세한 분할과 추적) |
| **작업 기간** | 매우 공격적 (Phase 1~3 총 **9주**) | 보수적, 병렬화 고려 (GraphRAG 기준 **12.5주**) |
| **팀 간 통신** | **비동기 이벤트(Kafka/PubSub) 중심**의 파이프라인 | GCS 기반 비동기 전달 (이벤트 트리거 기술은 약함) |
| **API 흐름** | **2-Tier 동기 통신** (S&F Recall ➔ GraphRAG Precision) | API 흐름은 유사하나 SLA 및 역할 명시가 상대적으로 약함 |
| **태스크 매핑** | 거시적 요약 (Phase 단위의 블록) | **미시적 전수 매핑** (v5 태스크 73개 개별 추적) |
| **Data Contract** | 필수 데이터 항목 위주의 거시적 사양 | **JSON 스키마 샘플** 및 전달 조건, 경로의 구체화 |
| **비용 및 리스크**| 필터링 역전 등 핵심 아키텍처 리스크에 집중 | 상세한 **Phase별 비용 분배표** 및 4가지 실행 리스크 정의 |

## 2. 리뷰 및 타당성 분석

각 계열이 가진 고유한 강점을 도출하고, 이를 바탕으로 vg2에 반영할 타당한 요소들을 분석한다.

### 2.1. `g` 계열의 강점 및 타당성

1. **마이크로서비스 및 Polyglot 아키텍처 비전 제시**
   * **타당성 (High)**: 분리의 근본 목적이 단순히 "일 나누기"가 아니라 "시스템 결합도 저하 및 스케일링 유연성 확보"임을 명확히 한 것은 장기적 비전에 부합한다. Vector DB와 Graph DB의 관심사 분리는 매우 타당하다.
2. **이벤트 기반 (Kafka/PubSub) 아키텍처**
   * **타당성 (High)**: GCS에 단순히 JSON을 적재하고 기다리는 방식보다, PubSub/Kafka 등을 통한 **비동기 이벤트 트리거 방식**이 마이크로서비스 간의 파이프라인 결합도를 낮추고 처리 지연을 막는 데 유리하다.
3. **명확한 2-Tier API 흐름 (Recall ➔ Precision)**
   * **타당성 (High)**: S&F가 Top 500~1000명을 1차 필터링(Recall)하고 GraphRAG가 이를 20명으로 좁히는(Precision) 구조는 레이턴시 최적화에 필수적이며 현실적이다.

### 2.2. `c` 계열의 강점 및 타당성

1. **73개 태스크의 전수 추적 매핑**
   * **타당성 (Critical)**: v5 원본 계획과의 연속성을 유지하기 위해서는, 어떤 태스크가 어떤 팀으로 이동했는지 1:1로 매핑되는 과정이 필수적이다. 실행 시 혼동을 없애는 최적의 장치이다.
2. **비용, 예산 및 Go/No-Go 기준 분리**
   * **타당성 (High)**: 경영진의 승인을 구하고 팀 단위 평가를 진행하려면, 비용의 분할($2,115 vs $6,785 수준 등)과 팀별 고유의 Go/No-Go 기준 설정이 반드시 필요하다.
3. **세밀한 Data Contract (JSON 스키마 명세)**
   * **타당성 (High)**: API나 페이로드를 스키마 레벨에서 합의하지 않으면 병렬 개발이 불가능하다. JSON 포맷 구체화는 실행 단계에서 매우 타당하다.
4. **대기 구간을 고려한 캘린더 타임라인**
   * **타당성 (High)**: g계열의 9주 계획은 이상적 순수 근무 시간이지만, 실제는 S&F의 산출물을 "대기"하는 구간이 존재한다. 현실적인 프로젝트 관리를 위해서는 대기 구간을 포함한 약 12~13주의 타임라인(네트워크 병목 고려)이 더 타당하다.

## 3. vg2 계획 합성 전략 (Synthesis Strategy)

타당성 분석 결과를 바탕으로, **`g` 계열의 강력한 아키텍처 사상과 `c` 계열의 치밀한 실행 계획을 융합**하여 vg2 문서를 작성한다.

* **문서 구조 (vg2)**:
  * `01_architecture_overview.md`: `g` 계열의 2-Tier API 흐름, Polyglot 비전, 이벤트 기반 통신을 중심으로 한 거시적 개요 작성. Data Contract 스키마(`c` 계열 강점) 포함.
  * `02_execution_phases.md`: 타임라인(대기 구간 포함 12.5주) 및 Go/No-Go 마일스톤, 73개 태스크 요약을 통합하여 작성.
  * `03_team_responsibilities.md`: 각 팀(S&F, GraphRAG)의 R&R, 예산 분배, 리스크 관리 위주의 상세 운영 계획.
