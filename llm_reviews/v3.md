### C3. SituationalSignal 노드 설계 모순 — 공유 노드 vs 비공유 노드

**위치**:
- `01.ontology/v21/04_graph_schema.md` §1.7: SituationalSignal은 14개 고정 taxonomy **공유 노드**
- `02.knowledge_graph/v14/05_extraction_operations.md` §1.4: SituationalSignal은 **비공유 노드** (Chapter 삭제 시 함께 삭제)

04_graph_schema.md에서:
> "같은 상황을 경험한 후보를 그래프 탐색으로 연결하기 위한 **공유 노드**"
> `signal_id: STRING -- signal_label과 동일` (즉 14개 고정 노드)

05_extraction_operations.md §1.4에서:
```
// Step 2: Chapter -> 비공유 노드 관계 + 노드 삭제
WHERE owned:Outcome OR owned:SituationalSignal
DELETE r, owned
```

**모순**: 14개 고정 taxonomy 공유 노드를 Chapter 삭제 시 함께 삭제하면, 다른 Chapter에서 참조하는 SituationalSignal 노드가 사라진다. 이는 그래프 무결성을 파괴한다.

**분석**: SituationalSignal의 설계 의도를 보면 두 가지 해석이 가능:
1. **공유 노드 (04_graph_schema 의도)**: 14개 라벨 노드를 사전 생성, 여러 Chapter가 HAS_SIGNAL 엣지로 연결 → 삭제 시 **엣지만 제거, 노드 유지**
2. **비공유 노드**: Chapter별 독립 SituationalSignal 인스턴스 생성 → 삭제 시 노드도 삭제

**권고**: 04_graph_schema의 의도(공유 노드)가 올바르다. 05_extraction_operations.md의 공유 노드 분류표에서 SituationalSignal을 **Outcome과 분리하여 공유 노드로 재분류**해야 한다.

**문제**: 3-Layer 경계 원칙에 따르면 "매칭 알고리즘의 정의"는 01(what)이고, 03(how to serve)은 구현이어야 한다. 그런데 03의 구현이 01의 정의와 다른 로직을 사용하고 있어, 어느 것이 실제 적용될지 불명확.

**권고**: 03의 `compute_match_score()`에 "01.ontology/v21/03_mapping_features.md의 정의를 구현한다"는 명시적 참조를 추가하고, 간략화된 로직에 대해 "v1 MVP 간소화 구현"인지 "최종 구현"인지 명시. 이상적으로는 03에서 01의 pseudo-code를 그대로 참조하고 구현 차이점만 기술.

### H5. culture_fit INACTIVE 시 가중치 재분배의 비대칭 영향 미분석

**위치**: `01.ontology/v21/00_data_source_mapping.md` §3.5

`0.693`은 `ln(2)`의 근사값이지만, Python에서는 `math.log(2)` = 0.6931471805599453. 실측 반감기 958일에서 0.693 사용 시 1년 차이에서 ~0.01 오차 발생.

**권고**: `math.log(2)` 사용으로 변경하거나, pseudo-code이므로 그대로 두되 주석에 `# ln(2) ≈ 0.693` 명시.

## 전체 아키텍처 관찰

### 강점
1. **Evidence-first 원칙 일관 적용**: 모든 추출 필드에 confidence + evidence 부여. Graceful degradation이 체계적
2. **3-Tier 비교 전략**: 정규화 가능/불가를 데이터 특성별로 분류한 것이 실용적
3. **Phase 0 검증 문화**: 거의 모든 설계 가정에 Phase 0 PoC 검증이 계획되어 있음
4. **비용 실용주의**: LLM 토큰 절감(구조화 필드 사전 주입 44%), Batch API 활용 일관
5. **의도적 제외 명문화**: v1에서 빠지는 기능의 이유와 복귀 로드맵이 명확

### 구조적 우려
1. **Phase 0 의사결정 과부하**: W0~W1 시점에 PII 책임, Neo4j 티어, 크롤링 법률, LLM 모델 선정, Embedding 검증, REPLACE 비율 실측 등 최소 8개 이상의 결정이 몰려 있음. 우선순위 정렬 필요
2. **v1 MVP에서 culture_fit이 사실상 dead feature**: 가중치 0.10 × ACTIVE <10% = 실질 기여 <1%. 이 피처를 v1 스키마에 포함하는 것 자체가 복잡도만 증가시킬 수 있음. v1에서 완전히 제거(4→4 피처)하고 v2에서 도입하는 것이 더 깔끔할 수 있음
3. **Gold Set 50건의 다중 용도 충돌**: 모델 선정, 임계값 캘리브레이션, 가중치 튜닝, 회귀 테스트, 실험 평가를 모두 50건으로 수행. 동일 데이터에 다중 최적화를 수행하면 overfitting 위험. 최소한 calibration/tuning set과 evaluation set을 분리해야 함
