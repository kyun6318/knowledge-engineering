## 1. 01.ontology/v20 — 스키마 설계 리뷰

### 1.1 강점

**Evidence-first 원칙의 일관된 적용**
- CompanyContext, CandidateContext, MappingFeatures 모두 Evidence 인터페이스를 강제한다. source_id, source_type, span, confidence, extracted_at이 모든 claim에 부착되어 추적성이 확보됨.

**Graceful Degradation 설계**
- 5개 MappingFeatures 모두 INACTIVE 상태를 허용하고, 활성 피처만으로 가중 평균을 재분배하는 설계가 현실적이다. culture_fit이 v1에서 대부분 INACTIVE가 될 것이라는 사실을 숨기지 않고 명시한 점이 좋다.

**SituationalSignal 14개 Taxonomy + 경계 가이드**
- 모호한 조합(SCALE_UP vs TEAM_SCALING 등)에 대한 판정 기준이 구체적이다. OTHER 비율 모니터링 기준(30%/50% 임계값)도 실용적.

**ScopeType → Seniority 변환 규칙**
- CandidateContext의 scope_type(IC/LEAD/HEAD/FOUNDER)과 CompanyContext의 seniority(JUNIOR~HEAD) 간 변환 매트릭스가 명확하게 정의됨. IC를 경력 연수로 세분화하는 것은 합리적.

### 1.2 개선 필요 사항

#### [HIGH] F1 STAGE_SIMILARITY 매트릭스의 비대칭성 근거 부족

```
("EARLY", "GROWTH"): 0.30   vs   ("GROWTH", "EARLY"): 0.50
```

GROWTH 기업에서 EARLY 경험자(0.50)가 EARLY 기업에서 GROWTH 경험자(0.30)보다 높은 점수를 받는 이유가 명시되지 않았다. **"큰 조직에서 작은 조직으로의 적응이 그 반대보다 쉽다"는 가정이 암묵적**인데, 이를 문서에 명문화하고 Phase 0 파일럿에서 검증해야 한다.

**제안**: STAGE_SIMILARITY 매트릭스 바로 아래에 비대칭성 설계 근거를 주석으로 추가.

#### [HIGH] freshness_weight 단계 함수의 급격한 불연속

```python
# 364일: 0.9  →  366일: 0.7  (22% 급감)
```

현재 step function(90일/1년/3년/5년)은 경계에서 20~40%의 급격한 점수 변동이 발생한다. 364일에 갱신한 이력서와 366일에 갱신한 이력서의 freshness_weight가 0.2나 차이나는 것은 부자연스럽다.

**제안**: 연속 감쇠 함수(예: 지수 감쇠)로 전환하거나, 최소한 boundary smoothing을 적용. v1 파일럿에서 step vs smooth의 랭킹 차이를 비교 검토.

#### [MEDIUM] REPLACE 공고의 vacancy_fit INACTIVE 처리

REPLACE 공고에서 vacancy_fit(가중치 0.30)이 INACTIVE가 되면 role_fit(0.15)이 실질적으로 가중치의 큰 비중을 흡수한다. 문서에서 "도메인 직관과 일치"라고 설명하지만, **REPLACE 공고의 비율이 전체의 몇 %인지 실측 데이터가 없다**. 만약 REPLACE가 30%+라면 전체 매칭 품질에 상당한 영향을 미칠 수 있다.

**제안**: Phase 0에서 JD의 hiring_context 분포를 먼저 확인하고, REPLACE 비율이 높으면 REPLACE 전용 피처(경력 연수 정확 매칭, 기술 스택 중첩도 등) 도입을 검토.

#### [MEDIUM] operating_model facets의 v1 실효성

speed, autonomy, process 3개 facet 모두 confidence 기대치가 0.30~0.60이고, culture_fit(가중치 0.10) 매칭에만 사용되며, work_style_signals가 v1에서 대부분 null이므로 culture_fit 자체가 INACTIVE다. **v1에서 operating_model 추출에 LLM 토큰을 투입하는 ROI가 낮다**.

**제안**: v1에서 operating_model 추출을 생략하고, 해당 토큰을 Outcome/SituationalSignal 품질 향상에 투자하는 것이 비용 대비 효과적. v1.1(크롤링 보강) 이후 활성화를 권장.

#### [LOW] designation_codes → seniority 매핑의 "거짓 직명" 문제

문서에서 "거짓 직명이 많아 단순 매핑에 대한 고려 필요"라고 언급했으나 구체적 대응 전략이 없다. 예: 스타트업에서 "이사"가 실제로는 IC인 경우.

**제안**: designation 기반 seniority는 confidence에 0.65 상한을 이미 적용하고 있으므로 v1에서는 현행 유지하되, Phase 0 PoC 20건에서 직명-실제 역할 불일치율을 측정.

## 2. 02.knowledge_graph/v13 — 추출 로직 리뷰

### 2.1 강점

**DB-first 원칙의 철저한 적용**
- 구조화 필드(company, period, tech_stack)를 DB에서 직접 추출하고, LLM은 scope_type, outcomes, situational_signals 등 추론 필요 필드에만 사용한다. 이로써 LLM 비용을 $654(Batch)로 억제한 것은 매우 현실적.

**적응형 LLM 호출 전략 (M1)**
- Career 1~3은 1-pass, 4+는 N+1 pass로 분기하는 전략이 비용과 품질을 균형 있게 다룬다. 80%가 1-pass에 해당하므로 대부분 효율적으로 처리됨.

**6단계 파이프라인 검증 체크포인트**
- CP1(입력) → CP2(마스킹) → CP3(LLM 출력) → CP4(정규화) → CP5(적재) → CP6(임베딩)까지 각 단계별 검증 코드가 구체적으로 정의됨.

**DETACH DELETE 시 공유 노드 보호**
- Skill, Organization, Role, Industry는 관계만 제거하고 노드는 유지, Outcome/SituationalSignal은 노드까지 삭제하는 2단계 전략이 명확.

### 2.2 개선 필요 사항

#### [CRITICAL] CareerDescription FK 부재(D4)의 영향 범위 과소평가

CareerDescription에 career_id FK가 없어 LLM으로 귀속을 판단하는 전략은 합리적이나, **fill rate 16.9%이고 중앙값 527자**에 불과하다. 복수 Career가 있는 이력서에서 LLM 귀속 정확도를 Phase 0에서 반드시 측정해야 한다.

더 심각한 것은 SelfIntroduction(64.1%) fallback인데, 자기소개서는 특정 Career에 귀속시키기가 CareerDescription보다 훨씬 어렵다. **Outcome/SituationalSignal의 Career별 정확도가 전체 매칭 품질의 병목**이 될 수 있다.

**제안**:
1. Phase 0 PoC 20건 중 Career 2건+ 이력서를 최소 10건 포함하여 귀속 정확도 측정
2. 귀속 실패 시 resume 전체 귀속(confidence 하향) 전략의 confidence 감쇠율을 구체적으로 정의 (현재 "하향"만 명시)

#### [HIGH] 06_normalization.md의 임베딩 모델 불일치

```python
# 06_normalization.md §1
def compute_embedding_similarity(text_a, text_b):
    emb_a = embed_text(text_a)  # 1536d vector
```

06_normalization.md에서 참조하는 임베딩 차원이 **1536d**(text-multilingual-embedding-002)인데, 02_model_and_infrastructure.md와 01_extraction_pipeline.md에서는 **text-embedding-005 (768d)**를 사용한다고 명시되어 있다. 07_neo4j_schema.md에서도 1536d를 언급한다.

**두 문서 간 임베딩 모델/차원이 불일치**한다. 이는 v20 이전 문서에서 이동할 때 업데이트가 누락된 것으로 보인다.

**제안**: 임베딩 모델을 text-embedding-005 (768d)로 통일할지 text-multilingual-embedding-002 (1536d)로 할지 확정하고, 모든 문서에서 일관되게 반영. Phase 0에서 두 모델의 한국어 분별력 비교가 선행되어야 함.

#### [HIGH] PII 마스킹의 데이터 플랫폼 책임 경계 미확정

04_pii_and_validation.md 상단에 "개인 정보 처리는 데이터 플랫폼에서 책임지고 진행할 수 있는지 확인 필요"라는 메모가 있다. **PII 마스킹의 주체가 S&F 팀인지 데이터 플랫폼인지 미확정** 상태이며, 이것이 확정되지 않으면 Phase 1 PII 마스킹 모듈 구현 일정에 직접적 영향을 미친다.

**제안**: Phase 0 시작 전(W0)에 데이터 플랫폼 팀과 PII 처리 책임 경계를 확정하고, Go/No-Go 01_go_nogo_decisions.md에 의사결정 항목으로 추가.

#### [MEDIUM] HWP 파서 품질 리스크(R5.1)의 대안 비용 미산정

HWP 파서 실패율 10~20%를 예상하고 Gemini Multimodal 폴백을 계획하나, **Gemini Multimodal의 HWP 처리 비용과 정확도가 미검증**이다. HWP는 한국 고유 포맷으로 Gemini의 지원 수준이 불확실하다.

**제안**: Phase 2-0 PoC에서 HWP 10건 → Gemini Multimodal 처리 결과를 반드시 포함. 실패 시 hwp5 → LibreOffice CLI 변환 → PDF 파서 경유 전략을 대안으로 추가.

#### [LOW] Gold Set 관리 규모의 통계적 근거

Phase 0: 50건, Phase 4: 200건으로 확장하는 계획이나, **200건으로 95% CI ±5% 정확도를 달성하려면 전체 모집단 대비 충분한지**에 대한 통계적 근거가 없다. 09_evaluation.md에서는 384건 통계 샘플링을 별도로 진행하는데, Gold Set과 통계 샘플링의 관계가 불명확.

**제안**: Gold Set(회귀 테스트용)과 통계 샘플링(품질 검증용)의 역할을 명확히 구분하는 문단 추가.

## 3. 03.graphrag/separate/v3 — 구현 계획 리뷰

### 3.1 강점

**2팀 분리의 Data Contract 설계**
- S&F ↔ GraphRAG 간 GCS JSONL + PubSub 기반 비동기 인터페이스가 깔끔하다. 5개 Artifact(A~E)의 전달 시점, 포맷, 트리거가 명확히 정의됨. "S&F는 그래프 구조를 알 필요 없고, GraphRAG는 파싱 방법을 알 필요 없다"는 원칙이 잘 지켜짐.

**2-Tier API SLA 설계**
- S&F API(하드필터+벡터, p95 < 500ms) → GraphRAG API(IN-list 매칭, p95 < 2s) → 전체 체인 p95 < 3s 구조가 명확. 각 팀의 SLA 책임 범위가 분리됨.

**Go/No-Go 14건의 구체성**
- 각 Phase 전환 시 정량적 기준(scope_type > 60%, throughput > 80%, Top-10 fit > 70% 등)이 정의되어 있어 "감으로 판단"하는 리스크가 낮다.

**평가 실험 설계(09_evaluation.md)의 엄밀성**
- Cohen's d ≥ 0.5 성공 기준, power analysis(n=50~100 적응형), 5명 평가자 blind evaluation, 4가지 의사결정 시나리오(GraphRAG 우위 / LLM reranking 충분 / 불명확 / Vector 우위)가 체계적.

### 3.2 개선 필요 사항

#### [HIGH] Neo4j AuraDB Professional의 8M 노드 수용 검증 부재

07_neo4j_schema.md에서 27M 노드, 133M 엣지를 추정하면서도, AuraDB Professional의 상한이 "800K+"로만 기술되어 있다. **27M 노드가 Professional 티어로 가능한지 확인되지 않았다**. 02_model_and_infrastructure.md에서도 "Phase 0 검증 필수"라고만 언급.

이것은 Phase 2 시작 시점(W10)에야 확인되므로, 불가 판정 시 AuraDB Enterprise(월 $500+) 또는 자체 호스팅으로 전환해야 하며 비용이 2~5배 증가한다.

**제안**:
1. Phase 0(W1)에서 Neo4j 영업팀에 사이징 문의를 선행
2. Go/No-Go에 "Neo4j 티어 확정"을 Phase 0 필수 의사결정으로 격상
3. 비용 문서에 Enterprise 시나리오 추가($300~500/월 → 27주 $2,100~3,500)

#### [HIGH] MAPPED_TO 엣지 규모 추정의 불확실성

04_graphrag_g3_matching.md에서 MAPPED_TO 규모를 3가지 시나리오(500K/2M/5M)로 추정하나, **실제 매칭 임계값(0.4)과 JD 수(10K)가 확정되지 않았다**. 5M 관계 시 32GB 메모리가 필요하며 이는 Professional 티어를 초과할 가능성이 높다.

**제안**: Phase 1(1K 적재 후) 시점에서 소규모 매칭 시뮬레이션(JD 10개 × Person 1K)을 수행하여 평균 매칭 수를 실측하고, 이를 기반으로 전체 규모를 외삽.

#### [HIGH] 07_neo4j_schema.md vs 01_extraction_pipeline.md 규모 추정 불일치

| 항목 | 01_extraction_pipeline.md | 07_neo4j_schema.md |
|---|---|---|
| Person | 600K | 3.2M |
| Chapter | 1.8M | 18M |
| 총 노드 | ~8M | ~27M |
| 총 엣지 | ~25M | ~133M |

**두 문서 간 규모 추정이 3~5배 차이**난다. 01_extraction_pipeline.md의 600K은 서비스 가용 이력서(PUBLIC+COMPLETED+main_flag=1)를 기준으로 한 것이고, 07_neo4j_schema.md의 3.2M은 전체 이력서를 기준으로 한 것으로 추정되나, 어느 것이 실제 적재 범위인지 불명확.

**제안**: 적재 범위를 확정하고 두 문서의 규모를 통일. 00_data_source_mapping.md의 서비스 풀(5.5M) → EXPERIENCED(3.7M) → HIGH+PREMIUM(3.2M) 필터 기준에 따라 적재 범위를 명시적으로 결정.

#### [MEDIUM] Phase 간 idle time 활용 전략 부재

GraphRAG 팀의 G-0(W1) → G-1(W5~6) 사이 3주, G-1(W6) → G-2(W10~11) 사이 3주의 유휴 시간이 있다. 00_graphrag_overview.md에서 "API 골격, 선행 작업"으로 언급되나 구체적 태스크가 불명확.

**제안**: idle time에 Organization ER 사전 구축, Gold Set 50건 작성, Neo4j 쿼리 최적화 등을 할당하여 G-3/G-4 일정 압박을 완화.

#### [MEDIUM] 크롤링(Phase 4) 법률 리스크의 Go/No-Go 미반영

S&F Phase 0에서 "법률 검토 결과에 따라 크롤링 여부 결정"이라고 하지만, 법률 검토가 Phase 4(W24)까지 지연되면 크롤링 계획 자체가 무산될 수 있다. **법률 검토 완료 시점이 명시되지 않았다**.

**제안**: Go/No-Go에 "W4까지 크롤링 법률 검토 완료" 항목 추가. 미완료 시 Phase 4를 DB 보강(NICE 추가 조회, 투자DB API)으로 대체하는 대안 계획 수립.

#### [MEDIUM] S&F 비용의 Anthropic Batch API 가격 의존성

S&F 비용의 88%($1,730/$1,955)가 Anthropic Batch API이다. **Batch API의 50% 할인이 변경되거나, 처리 지연이 24시간을 초과하면** 비용이 2배가 되거나 일정이 지연된다.

**제안**: Gemini Flash Batch를 Phase 0에서 테스트하는 것은 이미 계획되어 있으나, 비용 문서에 "Anthropic 가격 변경 시" 시나리오를 추가. Provider 추상화(LLM Provider abstraction)를 Phase 1이 아닌 Phase 0에서 설계하는 것을 권장.

#### [LOW] 월간 운영 비용의 인건비 미포함

GraphRAG 운영 비용이 월 $146~246로 산정되나, 이는 인프라 비용만이다. **증분 처리 모니터링, Gold Set 관리, 모델 재학습 등 인건비가 미포함**되어 실제 운영 비용을 과소 추정할 수 있다.

**제안**: 운영 단계 인력 투입(0.5 FTE? 1 FTE?) 추정을 별도 섹션으로 추가.

## 6. 전체 아키텍처에 대한 의견

### 6.1 이 설계의 가장 큰 장점

**"부분 완성을 허용하면서도 품질을 추적한다"**는 철학이 전체 문서에 일관되게 적용되어 있다. null/Unknown/INACTIVE를 정상 상태로 인정하고, confidence와 evidence를 통해 불확실성을 명시적으로 관리하는 접근은 실세계 데이터의 불완전성을 다루는 올바른 방식이다.

### 6.2 가장 큰 리스크

**v1에서 실제로 의미 있는 매칭을 생산할 수 있는 이력서 비율이 예상보다 낮을 수 있다**. F1~F5 중 3개 이상이 ACTIVE인 이력서가 전체의 30~40%에 그칠 가능성이 있다(F4 거의 INACTIVE, F1 50~60%, F2 50~65%, F5 50~60%). 이 경우 전체 서비스 풀 대비 매칭 가능 풀이 너무 작아 비즈니스 임팩트가 제한될 수 있다.

**제안**: Phase 1 완료 시점(1K 적재 후)에 "피처 활성화 분포" 리포트를 필수 산출물로 추가하여, 활성 피처 수 별 이력서 비율을 실측. 활성 2개 이하가 50%+면 피처 활성화 기준을 완화하거나, Vector baseline과의 앙상블을 조기 검토.
